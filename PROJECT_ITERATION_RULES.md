## 二、代码编写约束

### 2.1 命名规范

**强制命名规则**：

| 类型 | 命名方式 | 示例 |
|:----:|:---------:|:-----|
| 变量名 | snake_case | order_no, process_name |
| 函数名 | snake_case | get_order_by_id, update_status |
| 类名 | PascalCase | OrderService, ProcessController |
| 常量名 | SCREAMING_SNAKE_CASE | MAX_RETRY_COUNT, DB_TIMEOUT |
| 文件名 | snake_case | order_service.py, process_view.py |
| 数据库表名 | snake_case | orders, process_steps |
| 数据库字段名 | snake_case | order_no, create_time |

**约束规则**：

- R-020：禁止使用拼音命名，必须使用英文单词或常用缩写
- R-021：订单号字段必须命名为 order_no，禁止使用 orderId/order_number
- R-022：时间戳字段必须使用 create_time/update_time 命名

### 2.2 Flask 开发约束

**Flask 应用约束**：

```python
# 强制：before_request 中使用字典代替 global 变量
def create_app():
    app = Flask(__name__)
    _warmup_state = {'done': False}  # ✅ 使用字典

    @app.before_request
    def _warmup_on_first_request():
        if not _warmup_state['done']:  # ✅ 闭包内直接访问
            ...
```

**约束规则**：

- R-030：Flask 应用初始化顺序：创建实例 → 配置 → 扩展 → 钩子 → 蓝图 → 路由
- R-031：禁止在 before_request 中使用 global 关键字，必须使用字典/列表包装
- R-032：数据库连接预热必须延迟到首次请求时执行，禁止在 create_app() 中预热
- R-033：缓存操作必须使用字典实现，写操作后必须清除缓存

### 2.3 异常处理约束

**约束规则**：

- R-040：禁止使用裸 except（必须指定具体异常类型）
- R-041：捕获异常后必须记录日志，包含错误信息和上下文
- R-042：异常处理后必须给出明确的错误提示，禁止静默失败
- R-043：关键业务操作必须使用事务控制，回滚失败必须告警

**正确示例**：

```python
# ✅ 正确
try:
    result = perform_operation()
except ValueError as e:
    logger.error(f"参数错误: {e}, order_no={order_no}")
    return error_response("参数错误")
except DatabaseError as e:
    logger.critical(f"数据库错误: {e}, order_no={order_no}")
    notify_oncall()
    return error_response("系统错误")
```

## 三、业务流程约束

### 3.1 订单生命周期约束

**强制状态流**：

```
创建(created) → 确认(confirmed) → 排产(scheduled) → 执行(in_progress) → 完工(completed) → 发货(shipped) → 归档(archived)
```

**约束规则**：

- R-050：订单状态必须按上述顺序流转，禁止跨状态跳转（除归档回退外）
- R-051：订单状态变更必须记录操作日志（操作人、时间、变更前后状态）
- R-052：归档后的订单禁止修改，必须新建订单处理

### 3.2 工序任务约束

**强制状态流**：

```
待发布(pending) → 已发布(distributed) → 执行中(in_progress) → 已完成(completed)
```

**约束规则**：

- R-060：工序任务必须关联到具体订单，禁止创建游离任务
- R-065：工序任务必须指定执行人员，禁止无人认领
- R-066：工序完成后必须触发报工确认，更新订单进度
- R-067：工序撤回必须在执行前完成，执行后禁止撤回

### 3.3 四大发布流程约束

**工序发布**：桌面端 → 调度中心 → 移动端 → 工序完成

**物料发布**：申请(material_requested) → 确认(material_confirmed) → 入库(material_arrived) → 出库(material_delivered)

**外协发布**：创建(outsource_created) → 确认(outsource_confirmed) → 生产(outsource_production) → 质检(outsource_qc) → 回厂(outsource_returned)

**质检发布**：待检(pending) → 已分发(distribution) → 已上报(quality_reported) → 合格(quality_passed)/不合格(quality_failed)

**约束规则**：

- R-070：所有发布任务必须通过 5003 调度中心，禁止绕过调度中心直接操作
- R-071：任务状态变更必须记录完整的状态流转历史
- R-072：任务完成后必须触发相关方通知（企业微信/邮件）

## 七、代码风格补充约束（v1.1 2026-06-21 追加）

> **背景**：v2.1 缓存失效机制重构时发现以下恶劣写法，固化为强制规则。

### 7.1 禁止"dir() 检查变量"恶劣写法

**R-240**：禁止使用 `X if 'X' in dir() else Y` 检测局部变量是否定义。

**恶劣示例**（[models/order.py:48](file:///d:/yuan/%E4%B8%8D%E9%94%90%E9%92%A2%E7%BD%91%E5%B8%A6%E8%B7%9F%E5%8D%953.0/models/order.py#L48) 修复前）：

```python
# ❌ 恶劣
def _sync():
    affected = 0
    try:
        affected = cc_cur.rowcount
    except Exception:
        return 0
    return affected if 'affected' in dir() else 0
```

**正确写法**：

```python
# ✅ 在 try 块开始前初始化
def _sync():
    affected = 0
    try:
        affected = cc_cur.rowcount
    except Exception:
        return 0
    return affected
```

**理由**：
- `dir()` 检查不可靠，可能误判模块级变量
- 逻辑混乱，依赖 Python 内部行为
- 可读性差，新人无法理解

### 7.2 禁止"循环中 list.remove()"模式

**R-241**：禁止 `for x in lst: lst.remove(x)` 模式，必须用 list comprehension 重建新列表。

**恶劣示例**（[dispatch_center/_core.py:1441](file:///d:/yuan/%E4%B8%8D%E9%94%90%E9%92%A2%E7%BD%91%E5%B8%A6%E8%B7%9F%E5%8D%953.0/mobile_api_ai/dispatch_center/_core.py#L1441) 修复前）：

```python
# ❌ 恶劣 - 循环中修改列表
update_data(lambda d: [
    d['process_tasks'].remove(t) for t in d.get('process_tasks', [])
    if t.get('task_id') == task_id
])
```

**正确写法**：

```python
# ✅ list comprehension 重建
update_data(lambda d: d.update({
    'process_tasks': [
        t for t in d.get('process_tasks', [])
        if t.get('task_id') != task_id
    ]
}))
```

**理由**：
- for 循环中 remove 会跳过相邻元素（IndexError / 漏删）
- list comprehension 语义清晰、原子操作
- 性能更优（单次遍历 O(n)）

### 7.3 单一职责原则 - 拆分混合函数

**R-242**：当函数混合两种职责（"同步" + "通知"），必须拆分为独立函数。

**案例**：[models/order.py](file:///d:/yuan/%E4%B8%8D%E9%94%90%E9%92%A2%E7%BD%91%E5%B8%A6%E8%B7%9F%E5%8D%953.0/models/order.py) 修复时将：

```python
# ❌ 之前: 1 个函数干 2 件事
def _sync_is_archived_to_container_center(order_ids, is_archived):
    # 1) 同步到 container_center
    # 2) 触发调度中心缓存失效 (同时遍历)
    pass

# ✅ 现在: 拆分为 2 个函数
def _sync_is_archived_to_container_center(order_ids, is_archived):
    # 只做同步
    pass

def _emit_invalidation_for_orders(order_nos, is_archived):
    # 只做失效事件
    pass
```

**理由**：
- 单一职责更易测试
- 业务逻辑（归档 vs 取消归档）分支清晰
- 后续维护成本降低

### 7.4 业务语义"反向逻辑"必须明确

**R-243**：当操作有正/反两方向时（如 archive vs unarchive），代码必须明确区分，不允许"两边都调清理"。

**案例**：取消归档时**不应该**触发缓存清理（订单应重新可见）。修复前错误地两边都调 emit_invalidate，修复后只归档时清理。

```python
# ✅ 正确: 分支明确
if is_archived:
    for order_no in order_nos:
        emit_invalidate(order_no, source='order_archive')  # 清理
else:
    logger.info(f'取消归档, 缓存由对账 worker 兜底')  # 不清理
```

## 八、监控告警约束

### 8.1 监控指标约束

**强制监控项**：

| 监控类别 | 监控指标 | 告警阈值 |
|:--------:|:--------:|:--------:|
| 服务可用性 | HTTP 成功率 | < 99.5% 告警 |
| 响应性能 | API 平均响应时间 | > 2s 告警 |
| 资源使用 | CPU 使用率 | > 80% 告警 |
| 资源使用 | 内存使用率 | > 85% 告警 |
| 资源使用 | 磁盘使用率 | > 90% 告警 |
| 业务指标 | 订单处理失败数 | > 10/分钟 告警 |

**约束规则**：

- R-200：所有服务必须接入监控告警系统
- R-201：告警必须分级处理：严重（P0）、警告（P1）、提示（P2）
- R-202：告警响应时间：P0 ≤ 5 分钟，P1 ≤ 30 分钟，P2 ≤ 4 小时
- R-203：告警必须记录处理过程，形成故障案例库

### 8.2 日志规范约束

**约束规则**：

- R-210：日志必须分级：DEBUG、INFO、WARNING、ERROR、CRITICAL
- R-211：生产环境默认 INFO 级别，禁止输出 DEBUG 日志
- R-212：日志必须包含：时间戳、级别、模块名、请求ID、日志内容
- R-213：错误日志必须记录异常堆栈信息和上下文数据
- R-214：敏感信息（密码、密钥、身份证号）禁止写入日志

## 十、违规处理

### 10.1 违规级别定义

| 级别 | 标识 | 说明 |
|:----:|:----:|:-----|
| 严重违规 | 🔴 | 影响系统稳定性、安全性、核心功能的规则 |
| 一般违规 | 🟡 | 影响代码质量、可维护性的规则 |
| 轻微违规 | 🔵 | 代码规范、文档格式等非功能性规则 |

### 10.2 违规处理方式

**严重违规（🔴）**：

- 必须立即修复
- 需要提交代码审查特别说明
- 纳入技术债务，需在 1 周内偿还

**一般违规（🟡）**：

- 需要在本次迭代内修复
- 纳入代码审查检查清单
- 累计 3 次同类违规升级为严重违规

**轻微违规（🔵）**：

- 建议修复，非强制要求
- 纳入代码格式化工具自动修复
- 持续出现纳入绩效考核参考

### 10.3 规则豁免

**豁免条件**：

- 技术方案评审通过的临时方案
- 紧急故障修复的临时补丁（需在 3 天内补全）
- 第三方系统限制导致的特殊处理

**豁免流程**：

1. 技术负责人评估并批准
2. 在代码注释中说明豁免原因和计划解决时间
3. 记录到技术债务清单
