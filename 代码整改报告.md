## 整改项一：全局 `print` 替换为 `logger`（P1）

### 问题描述

项目在 `main.py` 中已配置了完整的 `logging.basicConfig`（含文件 Handler），但多个文件仍直接使用 `print` 输出，导致：

- 生产环境日志无法集中收集与持久化
- 无法按级别（INFO / WARNING / ERROR）过滤
- 打包为 EXE 后 `print` 输出完全丢失，线上问题难以排查

### 涉及文件与行号

| 文件 | 行号 | 违规内容摘要 |
|------|------|-------------|
| `core/event_bus.py` | 71 | `print(f"事件处理错误 [{event} -> {handler.__name__}]: {e}")` |
| `backup_system.py` | 74 | `print(f"[备份] 设置文件权限失败: {e}")` |
| `backup_system.py` | 76 | `print(f"[备份] 数据库备份成功: {backup_file}")` |
| `backup_system.py` | 82 | `print(f"[备份] 备份失败: {e}")` |
| `backup_system.py` | 93 | `print(f"[备份] 删除旧备份失败 {filename}: {e}")` |
| `backup_system.py` | 114 | `print("[备份系统] 自动备份已启动")` |
| `container_event_listener.py` | 28、35、53、55、57、78、88、90、93、111、126、131、134、153、157、168、172 | 多处 `print(...)` |
| `analyze_order.py` | 全文 | 整文件使用 `print` 输出分析结果 |
| `main.py` | 114 | `print(banner)` — 与 `logger.info(banner)` 重复输出 |

### 整改方案

**步骤一**：每个待改文件顶部补充 logger 声明（若已有则跳过）：

```python
import logging
logger = logging.getLogger(__name__)
```

**步骤二**：按以下对照表替换：

```python
# ❌ 整改前 — core/event_bus.py:71
print(f"事件处理错误 [{event} -> {handler.__name__}]: {e}")

# ✅ 整改后
logger.warning(f"事件处理错误 [{event} -> {handler.__name__}]: {e}", exc_info=True)
```

```python
# ❌ 整改前 — backup_system.py
print(f"[备份] 数据库备份成功: {backup_file}")
print(f"[备份] 备份失败: {e}")
print("[备份系统] 自动备份已启动")

# ✅ 整改后
logger.info(f"[备份] 数据库备份成功: {backup_file}")
logger.error(f"[备份] 备份失败: {e}", exc_info=True)   # exc_info=True 自动附加堆栈
logger.info("[备份系统] 自动备份已启动")
```

```python
# ❌ 整改前 — container_event_listener.py（模块级 import 失败）
print(f'[容器监听器] 依赖模块不可用: {e}')

# ✅ 整改后
logger.warning(f'[容器监听器] 依赖模块不可用: {e}')
```

```python

logger.info(banner.strip())

# ✅ 整改后（只保留 logger，避免重复）

```

## 整改项二：`backup_system.py` 连接/游标/SQL 管理（P1）

### 问题描述

`backup_database()` 函数存在三个独立缺陷：

#### 缺陷 A：游标未在 `finally` 中关闭

```python
# ❌ 整改前（第 23-68 行）
conn = pymysql.connect(**MYSQL_CONFIG)
cursor = conn.cursor()
# ... 大段操作 ...
conn.close()   # cursor 从未在 finally 中关闭
               # 若中途抛出异常，游标永久泄漏
```

#### 缺陷 B：表名使用了参数化占位符（运行时报错）

```python
# ❌ 整改前（第 41、44 行）
cursor.execute("SELECT * FROM %s", (table,))   # pymysql 参数化仅适用于值，不适用于表名
cursor.execute("SHOW COLUMNS FROM %s", (table,))  # 实际执行会抛出 ProgrammingError
```

#### 缺陷 C：`config.py` 的 `MYSQL_CONFIG` 引用路径不一致

```python
# ❌ 整改前（第 11 行）
from config import MYSQL_CONFIG   # config.py 中该变量实际来自 db_config.py
```

### 整改方案

```python
# ✅ 整改后 — backup_system.py 核心结构

import logging
from contextlib import contextmanager
from db_config import MYSQL_CONFIG   # 直接从权威来源引入

logger = logging.getLogger(__name__)

@contextmanager
def _get_backup_cursor():
    """备份专用数据库连接（独立连接，不走应用连接池）"""
    import pymysql
    config = MYSQL_CONFIG.copy()
    config["cursorclass"] = pymysql.cursors.DictCursor
    conn = pymysql.connect(**config)
    cursor = None
    try:
        cursor = conn.cursor()
        yield cursor
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:
        if cursor:
            cursor.close()   # ✅ 确保游标关闭
        conn.close()

def backup_database():
    try:
        backup_dir = ...
        with _get_backup_cursor() as cursor:
            cursor.execute("SHOW TABLES")
            db_name = MYSQL_CONFIG['database']
            key = f"Tables_in_{db_name}"
            tables = [row[key] for row in cursor.fetchall()]

            with open(backup_file, 'w', encoding='utf-8') as f:
                for table in tables:
                    # ✅ 表名用反引号直接内嵌（来源可信：SHOW TABLES 返回）
                    cursor.execute(f"SELECT * FROM `{table}`")
                    rows = cursor.fetchall()
                    cursor.execute(f"SHOW COLUMNS FROM `{table}`")
                    columns = [col['Field'] for col in cursor.fetchall()]
                    # ... 写入逻辑不变 ...

        logger.info(f"[备份] 数据库备份成功: {backup_file}")

    except Exception as e:
        logger.error(f"[备份] 备份失败: {e}", exc_info=True)
```

### 变更对照表

| 缺陷 | 整改前 | 整改后 |
|------|--------|--------|
| 游标泄漏 | `cursor` 无 `finally` 保护 | `contextmanager` 确保必然关闭 |
| 表名参数化错误 | `"SELECT * FROM %s", (table,)` | `` f"SELECT * FROM `{table}`" `` |
| 引入路径 | `from config import MYSQL_CONFIG` | `from db_config import MYSQL_CONFIG` |
| `print` 残留 | 多处 `print(...)` | 全部替换为 `logger.*` |

## 整改项三：`core/app.py` 游标管理规范（P1）

### 问题描述

`_ensure_audit_table()` 和 `_warmup_connection_pool()` 两个函数中，游标在 `try` 块内手动关闭，异常时跳过，导致资源泄漏。

```python
# ❌ 整改前 — _ensure_audit_table()（第 70-94 行）
conn = get_connection()
try:
    cursor = conn.cursor()
    cursor.execute('CREATE TABLE IF NOT EXISTS ...')
    conn.commit()
    cursor.close()    # ← 若上方抛出异常，此行不会执行
finally:
    conn.close()

# ❌ 整改前 — _warmup_connection_pool()（第 97-110 行）
conn = get_connection()
cursor = conn.cursor()    # ← 无任何异常保护
cursor.execute("SELECT 1")
cursor.fetchone()
cursor.close()
conn.close()              # ← 若上方抛出异常，两者都不会关闭
```

### 整改方案

```python
# ✅ 整改后 — _ensure_audit_table()
def _ensure_audit_table():
    """确保审计日志表存在"""
    from models.database import get_connection
    conn = get_connection()
    cursor = None
    try:
        cursor = conn.cursor()
        cursor.execute('''
            CREATE TABLE IF NOT EXISTS audit_logs (
                id INTEGER PRIMARY KEY AUTO_INCREMENT,
                timestamp TEXT NOT NULL,
                operator TEXT,
                action TEXT NOT NULL,
                entity_type TEXT NOT NULL,
                entity_id TEXT,
                before_data TEXT,
                after_data TEXT,
                remark TEXT,
                ip_address TEXT,
                extra_info TEXT
            )
        ''')
        conn.commit()
    except Exception as e:
        logger.error(f"[DB] 创建审计日志表失败: {e}")
        raise
    finally:
        if cursor:
            cursor.close()    # ✅ 无论是否异常都执行
        conn.close()


# ✅ 整改后 — _warmup_connection_pool()
def _warmup_connection_pool():
    """预热数据库连接池"""
    from models.database import get_connection
    import time

    start = time.time()
    conn = get_connection()
    cursor = None
    try:
        cursor = conn.cursor()
        cursor.execute("SELECT 1")
        cursor.fetchone()
        elapsed = (time.time() - start) * 1000
        logger.info(f"[DB] 连接池预热: {elapsed:.0f}ms")
    except Exception as e:
        logger.error(f"[DB] 连接池预热失败: {e}")
        raise
    finally:
        if cursor:
            cursor.close()
        conn.close()
```

### 变更要点

- `cursor = None` 在 `try` 外声明，`finally` 中通过 `if cursor:` 安全关闭
- `_warmup_connection_pool` 补充 `try/except/finally` 结构
- 新增异常日志记录，便于排查初始化失败原因

## 整改项四：`EventBus` 单例与类方法混用歧义（P2）

### 问题描述

`EventBus` 同时使用**类变量 + 单例实例**两套机制，行为存在歧义：

```python
# ❌ 问题所在 — event_bus.py
class EventBus:
    _instance = None
    _handlers: Dict[str, List[Callable]] = defaultdict(list)  # 类变量（所有实例共享）
    _lock = threading.Lock()

    def __new__(cls):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._handlers = defaultdict(list)  # 赋给实例（覆盖类变量）
                    cls._instance._sync_lock = threading.Lock()  # 从未使用的冗余锁
        return cls._instance

    @classmethod
    def subscribe(cls, event, handler):
        with cls._lock:
            if handler not in cls._handlers[event]:   # 访问类变量 _handlers
                cls._handlers[event].append(handler)   # 与实例 _handlers 不是同一对象
```

**歧义说明**：`__new__` 给实例赋了 `_handlers`，但 `@classmethod` 方法访问的是 `cls._handlers`（类变量）——两者是不同对象。`_sync_lock` 声明后从未使用，属于死代码。

### 整改方案（选择"纯类方法"模式）

```python
# ✅ 整改后 — 纯静态 EventBus，移除单例 __new__

import logging
from typing import Callable, Dict, List, Any
from collections import defaultdict
import threading

logger = logging.getLogger(__name__)


class EventBus:
    """
    全局事件总线（纯类方法实现，无需实例化）
    线程安全：所有操作均在 _lock 保护下执行。
    """
    _handlers: Dict[str, List[Callable]] = defaultdict(list)
    _lock = threading.Lock()

    def __init_subclass__(cls, **kwargs):
        raise TypeError("EventBus 不允许继承")

    @classmethod
    def subscribe(cls, event: str, handler: Callable) -> None:
        with cls._lock:
            if handler not in cls._handlers[event]:
                cls._handlers[event].append(handler)

    @classmethod
    def unsubscribe(cls, event: str, handler: Callable) -> None:
        with cls._lock:
            if handler in cls._handlers[event]:
                cls._handlers[event].remove(handler)

    @classmethod
    def publish(cls, event: str, data: Any = None) -> None:
        with cls._lock:
            handlers = list(cls._handlers.get(event, []))
        for handler in handlers:
            try:
                handler(event, data)
            except Exception as e:
                # ✅ 替换 print 为 logger
                logger.warning(
                    f"事件处理错误 [{event} -> {handler.__name__}]: {e}",
                    exc_info=True
                )

    @classmethod
    def clear(cls, event: str = None) -> None:
        with cls._lock:
            if event:
                cls._handlers[event] = []
            else:
                cls._handlers.clear()

    @classmethod
    def get_handlers(cls, event: str) -> List[str]:
        return [h.__name__ for h in cls._handlers.get(event, [])]
```

### 变更对照表

| 问题 | 整改前 | 整改后 |
|------|--------|--------|
| 单例/类方法双轨 | `__new__` + `@classmethod` 混用 | 纯 `@classmethod`，无实例化 |
| 冗余死代码 | `_sync_lock` 声明但从未使用 | 移除 |
| `print` 残留 | `print(f"事件处理错误...")` | `logger.warning(..., exc_info=True)` |
| 继承保护 | 无 | `__init_subclass__` 阻止意外继承 |

> **向后兼容说明**：现有代码 `EventBus.subscribe(...)` / `EventBus.publish(...)` 的调用方式不变，无需修改调用侧。

## 整改项五：`config.py` 模块级 I/O 与异常日志（P2）

### 问题描述

#### 问题 A：模块顶层执行 I/O（每次 `import config` 触发文件读取）

```python
# ❌ 整改前 — config.py:277-280
COLORS = _load_colors()   # 模块导入时立即执行
FONTS  = _load_fonts()    # 同上
```

在 Tkinter 多窗口环境中，`config` 可能被多个模块导入，每次 `import` 都会触发一次 `settings_manager` 的文件读取，造成重复 I/O 开销。

#### 问题 B：异常被完全吞没，无任何日志

```python
# ❌ 整改前 — _load_colors()
def _load_colors():
    try:
        ...
    except Exception:          # 捕获范围太宽，且无日志
        return _DEFAULT_COLORS.copy()
```

颜色加载失败时静默回退，开发者无从得知是哪里出了问题。

### 整改方案

```python
# ✅ 整改后 — config.py

import logging
logger = logging.getLogger(__name__)

_COLORS_CACHE = None
_FONTS_CACHE  = None


def _load_colors():
    try:
        from utils.settings_manager import settings_manager
        colors = _DEFAULT_COLORS.copy()
        saved_colors = settings_manager.get_all_colors()
        colors.update(saved_colors)
        return colors
    except ImportError:
        # settings_manager 不可用属预期场景（如独立脚本运行），静默回退
        return _DEFAULT_COLORS.copy()
    except Exception as e:
        # ✅ 非预期异常：记录日志，仍回退到默认值
        logger.warning(f"[CONFIG] 颜色配置加载失败，使用默认值: {e}")
        return _DEFAULT_COLORS.copy()


def get_colors() -> dict:
    """懒加载颜色配置（首次调用时初始化，后续复用缓存）"""
    global _COLORS_CACHE
    if _COLORS_CACHE is None:
        _COLORS_CACHE = _load_colors()
    return _COLORS_CACHE


def get_fonts() -> dict:
    """懒加载字体配置"""
    global _FONTS_CACHE
    if _FONTS_CACHE is None:
        _FONTS_CACHE = _load_fonts()
    return _FONTS_CACHE


def reload_colors() -> dict:
    """主题切换后调用，强制重新加载颜色配置"""
    global _COLORS_CACHE
    _COLORS_CACHE = None
    return get_colors()


def reload_fonts() -> dict:
    """字体变更后调用，强制重新加载字体配置"""
    global _FONTS_CACHE
    _FONTS_CACHE = None
    return get_fonts()


# 向后兼容：保留模块级变量（首次导入时只触发一次 I/O）
COLORS = get_colors()
FONTS  = get_fonts()
```

### 变更要点

| 问题 | 整改前 | 整改后 |
|------|--------|--------|
| I/O 时机 | 每次 `import config` 触发 | 首次调用 `get_colors()` 时触发，后续复用缓存 |
| 异常处理 | `except Exception: pass`（吞没） | 区分 `ImportError`（预期）和其他异常（记录日志） |
| 主题刷新 | 无接口 | 新增 `reload_colors()` / `reload_fonts()` |
| 向后兼容 | - | `COLORS` / `FONTS` 变量仍可直接使用 |

## 整改项九：订单状态定义重复消除（P3）

### 问题描述

订单状态在 `constants.py` 和 `config.py` 中均有定义，二者并行存在、职责重叠：

```python
# constants.py — 枚举定义（业务语义）
class OrderStatus(Enum):
    SHIPPED = "已发货"
    FINISHED = "已完成"
    # ...

# config.py — 字典定义（含颜色，但又重复了状态字符串）
ORDER_STATUS = {
    "待确认": "#9E9E9E",
    "已确认": "#2196F3",
    "已发货": "#9C27B0",   # ← 与 OrderStatus.SHIPPED.value 重复
    "已完成": "#4CAF50",   # ← 与 OrderStatus.FINISHED.value 重复
    # ...
}
```

两套定义分离导致：状态字符串（如 `"已发货"`）在两处分别维护，新增状态时容易漏改一处。

### 整改方案（明确职责边界）

在 `config.py` 中显式引用 `constants.py` 的枚举值，消除硬编码字符串重复：

```python
# config.py — 整改后
from constants import OrderStatus

# 状态颜色映射：键来自枚举，不再硬编码字符串
ORDER_STATUS_COLORS = {
    OrderStatus.PENDING.value:      "#9E9E9E",   # 待确认
    OrderStatus.CONFIRMED.value:    "#2196F3",   # 已确认
    OrderStatus.SCHEDULED.value:    "#03A9F4",   # 已排产
    OrderStatus.PRODUCTION.value:   "#FF9800",   # 生产中
    OrderStatus.QC.value:           "#FF5722",   # 质检中
    OrderStatus.FINISHED.value:     "#4CAF50",   # 已完成
    OrderStatus.SHIPPED.value:      "#9C27B0",   # 已发货
    OrderStatus.CANCELLED.value:    "#F44336",   # 已取消
}

# 向后兼容：保留 ORDER_STATUS 变量名
ORDER_STATUS = ORDER_STATUS_COLORS
```

**职责边界说明**：

| 文件 | 职责 |
|------|------|
| `constants.py` | 状态语义（枚举值、状态机转换规则） |
| `config.py` | 状态的 UI 表现（颜色、图标等） |

新增状态时：在 `constants.py` 中加枚举值，在 `config.py` 中加对应颜色，两处职责清晰，不再混用。
