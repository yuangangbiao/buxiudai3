## 一、微服务架构约束

### 1.1 服务端口定义

**强制要求**：系统必须遵循微服务架构设计，各服务独立部署，通过 API 接口交互。

| 端口 | 服务名称 | 入口文件 | 核心功能 |
|:----:|:--------:|:---------|:---------|

| 5002 | 容器中心 | container_center_api.py | 流程数据处理 |
| 5003 | 调度中心 | standalone_dispatch_server.py | 任务调度核心、云端通信转发、预警与告警、**消息队列中继（Poll/ACK）**、**统计表推送**（v3.6.5 合并自原 5005） |
| 5005 | ~~cloud_relay~~ | ~~cloud_relay.py~~ | ⚠️ **已废弃**（v3.6.5 合并入 5003） |

### 1.2 服务间通信约束

**强制规则**：

- **R-001**：禁止在服务 A 中直接连接服务 B 的数据库，必须通过 API 接口交互
- **R-002**：所有云端通信必须通过 5003 调度中心转发到云端 5006，禁止直连云端
- **R-003**：新增服务必须向 5003 容器中心注册，遵循统一路由规范

### 3.2 核心概念区分

#### 3.2.1 流程 vs 工序

**流程 (Flow)** 是业务管理层面的状态流转序列，关注"这件事走到哪一步了"。

**工序 (Process Step)** 是生产执行层面的具体操作单元，关注"具体做什么操作"。

| 维度 | 流程 (Flow) | 工序 (Process Step) |
|------|-------------|---------------------|
| 视角 | 管理层/业务层 | 生产层/执行层 |
| 关注点 | 状态、审批、流转 | 具体操作、设备、人员 |
| 粒度 | 粗 | 细 |
| 示例 | 工单发布、排产确认、完工入库 | 原材料准备、编织、热处理、包装 |
| 状态字段 | `status_key` | `process_code` |
| 存储表 | `process_records` | `process_sub_steps` |

#### 3.2.2 关系映射

```
┌─────────────────────────────────────────────────────────┐
│  process_records (流程记录)                              │
│  - status_key: 流程节点状态                              │
│  - steps: 流程节点列表                                    │
│  - flow_type: 流程类型                                   │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  process_sub_steps (工序明细)                            │
│  - process_code: 工序编码 (P01, P02...)                 │
│  - step_name: 工序名称                                    │
│  - operator: 操作人员                                     │
│  - quantity: 完成数量                                    │
└─────────────────────────────────────────────────────────┘
```

#### 3.2.3 举例说明

订单 ORD-001 从发布到发货的全流程：

| 流程节点 (process_records) | 对应工序 (process_sub_steps) |
|---------------------------|------------------------------|
| 工单发布 | - |
| 排产制定 | - |
| 排产确认 | - |
| 生产执行 | P01 原材料准备 → P02 编织 → P03 热处理 |
| 质检审核 | Q01 外观检验 → Q02 尺寸检验 |
| 报工完成 | - |
| 完工入库 | W01 包装 → W02 入库 |
| 发货 | D01 发货 |

#### 3.2.4 命名约束

- **R-015**：禁止混用"流程"和"工序"概念
- **R-016**：代码注释和文档中必须明确区分
  - 流程相关用词：`流程节点`、`status_key`、`flow_type`
  - 工序相关用词：`工序明细`、`process_code`、`step_name`

### 3.3 任务表结构规范（v3.6）

**强制要求**：所有任务类型必须使用独立表，禁止混用 data_packages 表。

#### 3.3.1 任务表清单

| 表名 | 用途 | flow_type | 流程节点前缀 |
|------|------|-----------|----------|
| process_sub_steps | 生产工序明细 | production | P06-P16 |
| material_records | 物料任务 | material_purchase | M01-M06 |
| quality_records | 质检记录 | quality | Q01-Q06 |
| repair_records | 维修任务 | repair | R01-R07 |
| outsource_records | 外协任务 | outsource | O01-O08 |
| schedule_records | 排产任务 | production | - |
| quality_packages | 质检任务包 | quality | - |
| process_packages | 生产任务包 | production | - |

#### 3.3.2 统一基础字段

所有任务表必须包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) | 主键 |
| order_no | VARCHAR(64) | 关联订单号 |

| updated_at | DATETIME | 更新时间 |
| is_deleted | TINYINT(1) | 软删除 |

#### 3.3.3 流程节点代码规范

| 任务类型 | flow_type | 前缀 | 示例 |
|---------|-----------|------|------|
| 生产流程 | production | P | P06, P07, P09... |
| 物料流程 | material_purchase | M | M01, M02...M06 |
| 质检流程 | quality | Q | Q01, Q02...Q06 |
| 维修流程 | repair | R | R01, R02...R07 |
| 外协流程 | outsource | O | O01, O02...O08 |

#### 3.3.4 流程节点定义

**强制要求**：每个 flow_type 必须定义完整的流程节点列表。

| flow_type | 流程节点 | 说明 |
|-----------|----------|------|
| production | 工单发布→排产制定→排产确认→生产执行→质检审核→报工完成→完工入库→发货 | 生产流程 |
| material_purchase | 物料申请→供应商确认→物料采购→物料到货→质检入库→物料出库 | 物料采购流程 |
| quality | 接收质检→外观检验→尺寸检验→性能检验→判定结果→质检放行 | 质检流程 |
| repair | 故障报修→维修接单→故障诊断→维修执行→功能测试→验收确认→维修完成 | 维修流程 |
| outsource | 外协发单→外协确认→外协生产→外协质检→外协回厂→质检审核→入库登记→发货 | 外协流程 |

#### 3.3.5 表结构约束

- **R-060**：所有数据库表必须包含 create_time 和 update_time 字段
- **R-061**：所有表必须有主键，禁止使用联合主键
- **R-062**：软删除标记字段命名为 is_deleted，禁止物理删除核心业务数据
- **R-063**：外键关联必须建立索引，禁止使用逻辑外键
- **R-064**：数据库操作必须通过存储层，禁止在业务逻辑中直接写 SQL
- **R-065**：任务数据必须写入对应的新表，禁止写入 data_packages

## 四、API 接口规范

### 4.1 任务 API 端点

| 任务类型 | 列表接口 | 更新接口 | 数据表 |
|---------|---------|---------|--------|
| 生产工序 | `/api/report_record/list` | - | process_sub_steps |

| 质检任务 | `/api/quality_record/list` | `/api/quality_record/update` | quality_records |
| 维修任务 | `/api/repair_record/list` | `/api/repair_record/update` | repair_records |
| 外协任务 | `/api/outsource_record/list` | `/api/outsource_record/update` | outsource_records |
| 排产任务 | `/api/schedule_record/list` | `/api/schedule_record/update` | schedule_records |

### 4.2 RESTful 规范

- **R-070**：URL 必须使用名词复数形式，禁止使用动词短语
- **R-071**：禁止在 URL 中使用动词短语（如 /getOrder），必须用 HTTP 方法
- **R-072**：分页参数必须使用 page/page_size/query
- **R-073**：时间范围查询必须使用 start_time/end_time

### 4.3 响应格式规范

- **R-080**：所有 API 必须返回统一格式的 JSON 响应
- **R-081**：HTTP 状态码规范：200 成功，400 参数错误，401 未认证，403 无权限，404 不存在，500 系统错误
- **R-082**：禁止在响应中返回数据库原始错误信息，必须进行脱敏处理

**强制响应格式**：

```json
// 成功响应
{
    "code": 0,
    "message": "success",
    "data": { ... }
}

// 错误响应
{
    "code": 1001,
    "message": "订单不存在",
    "data": null
}
```

- **R-086**：`max_tasks_per_operator` 操作员过载阈值，默认 20 个
- **R-087**：`alert_cooldown_minutes` 重复告警冷却时间，默认 30 分钟

## 六、异常处理约束

### 6.1 异常捕获规范

- **R-040**：禁止使用裸 except（必须指定具体异常类型）
- **R-041**：捕获异常后必须记录日志，包含错误信息和上下文
- **R-042**：异常处理后必须给出明确的错误提示，禁止静默失败

### 6.2 日志记录规范

- **R-044**：日志必须包含：时间戳、级别、模块名、请求ID（如果有）、日志内容
- **R-045**：敏感信息（密码、密钥、身份证号、订单金额）禁止写入日志
- **R-046**：生产环境默认 INFO 级别，禁止输出 DEBUG 日志
