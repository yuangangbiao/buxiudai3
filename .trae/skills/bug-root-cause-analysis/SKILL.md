# Bug 根因分析报告生成器

## 触发条件

**每次修复任何 Bug 后必须执行**，包括但不限于：
- 修复了任何接口/页面/数据库/同步问题
- 解决了任何报错（500/404/Bug）
- 排查后发现不是代码问题而是数据问题

## 输出要求

为每个 Bug 输出一份**大白话分析报告**，放在任务末尾回复给用户。

### 报告结构（必须包含 5 个部分）

1. **一句话总结** — 这个 Bug 是什么，用最通俗的话说
2. **数据怎么跑的** — 数据从哪来，经过哪些地方，最后去哪。画流程箭头。
3. **哪里拐弯了** — Bug 在哪个环节、哪个文件、哪行代码出错
4. **为什么之前没发现** — 为什么会漏进线上的原因分析
5. **以后怎么避免** — 对症下药，给具体可落地的建议

### 写作风格

- **全是大白话**，禁止技术术语堆砌
- 用"比如"、"就像"、"打个比方"等口语化表达
- 类比生活场景帮助理解
- 禁止代码片段（除非必要，放在最后附录）

### 示例结构

```
## 🔍 Bug 分析报告：「工序报工列表同一订单重复显示」

### 1. 一句话总结
系统里有两种记录都叫"工序记录"，它们混在同一张表里，查订单的时候把它们都查出来了，
就像你订外卖时，同一个订单被外卖小哥和商家各记了一遍，你一看——怎么有两笔？

### 2. 数据怎么跑的
桌面端发布工单 → 写「工序记录」→ 工序名是"切割"、"焊接"
手机端排产发布 → 写「工作流记录」→ 步骤名是"工单发布"、"排产制定"
         ↓
都写进了同一张表 process_records，都用 order_no 关联
         ↓
前端查工序任务 → 按 order_no 查 → 把两种记录都查出来了

### 3. 哪里拐弯了
app.py 第 212 行：没有去重，直接返回所有记录
数据库 process_records 表：产品工序和工作流节点混存在一起
前端 mobile_unified.html：没有按记录类型做区分

### 4. 为什么之前没发现
两个系统各自开发，各自测试，没有联调过这个场景
桌面端只管工序，手机端只管排产，没人想过它们会互相影响

### 5. 以后怎么避免
方案A（治本）：把 process_records 表拆成两张——产品工序表和工作流节点表
方案B（治标）：在查列表的 API 层加过滤条件，只返回产品类型记录
方案C（预防）：每次改表结构时，评审有没有多系统写入同一张表的情况

附录：修改的代码位置
- app.py:212-230
- mysql_storage.py:1203-1213
```

## 执行步骤

1. **确认 Bug 修复已完成**（代码已改、已测试通过）
2. **追溯数据流**：从用户操作开始，到数据写入，到查询返回，画出完整链路
3. **定位拐点**：找到哪个环节、哪个文件、哪行代码出了问题
4. **分析根因**：从代码逻辑、表结构设计、多系统交互三个维度分析
5. **输出报告**：按上面的 5 部分结构，用大白话写
6. **给出建议**：给出 1-3 个具体可执行的"以后怎么避免"建议

## 质量标准

- 报告必须让**不懂代码的人**也能看懂 80% 以上
- 每个 Bug 至少包含 1 个可执行的"以后怎么避免"建议
- 如果根因涉及数据库表结构问题，必须指出具体表名和字段
- 如果是多系统交互问题，必须画出数据流图（用文字箭头表示）
