> **代 号**: 调度中心
> **角 色**: 元 skill / 路由器
> **下属专家**: 小圣(架构)/ 小贺(品控)/ 小曦(PM)/ 小钰(安全)

## ⚠️ 重要:触发规则

**本 skill 采用显式触发机制,避免隐性触发干扰正常工作流**

| 触发类型 | 示例 | 是否触发 |
|---------|------|:--------:|
| 显式触发 | "用4专家审一下"、"调调度中心"、"小圣+小贺一起看" | ✅ 触发 |
| 显式触发 | "专家团队"、"4人团队" | ✅ 触发 |
| 隐式触发 | "帮我看看这个bug"、"这个接口有问题" | ❌ 不触发(由具体专家 skill 处理) |

## 一、核心职责

本 skill 是 4 专家团队的**统一入口**。用户**必须显式调用**后,本 skill 会:

```
用户输入(任意描述)
    ↓
[步骤 1] 语义解析 — 提取意图/对象/约束
    ↓
[步骤 2] 专家路由 — 单专家 vs 多专家联动
    ↓
[步骤 3] 顺序编排 — 按 PM→架构→安全→品控
    ↓
[步骤 4] 调用对应 skill 输出
    ↓
[步骤 5] 汇总 + 决策点(如多专家有分歧)
```

## 三、调度算法(7 步决策)

### 第 1 步:语义解析

从用户输入中提取 3 个关键信息:
- **意图**(审查/修复/调研/开发/上线/审计)
- **对象**(模块名/接口名/服务名/功能名)
- **约束**(时间/工作量/优先级)

### 第 2 步:意图分类

```python
INTENT_MAP = {
    '审查': ['xiaoyu-security', 'xiaosheng-architect'],
    '评审': ['xiaosheng-architect', 'xiaoyu-security'],
    '修复': [],  # 按 Bug 类型路由
    '调研': ['xiaoxi-pm'],
    '开发': ['xiaoxi-pm', 'xiaosheng-architect', 'xiaoyu-security', 'xiaohe-qa'],
    '上线': ['xiaoxi-pm', 'xiaohe-qa'],
    '测试': ['xiaohe-qa'],
    '梳理': ['xiaoxi-pm', 'xiaohe-qa'],
}
```

### 第 3 步:Bug 类型路由(修复场景)

```python
BUG_TYPE_MAP = {
    '并发': 'xiaosheng-architect',     # 抢锁/超量/竞争
    '锁': 'xiaosheng-architect',        # FOR UPDATE/行锁
    '鉴权': 'xiaoyu-security',          # JWT/Token/零凭证
    '越权': 'xiaoyu-security',          # 8项越权
    '注入': 'xiaoyu-security',          # SQL注入
    '状态机': 'xiaohe-qa',              # 任意跳级
    '边界': 'xiaohe-qa',                # 数量=0/负数
    '脱敏': 'xiaoyu-security',          # str(e)泄露
    '跨库': 'xiaosheng-architect',      # outbox
    '场景': 'xiaoxi-pm',                # 工厂实际
    '验收': 'xiaoxi-pm',                # 3层验证
}
```

### 第 4 步:多专家联动判断

满足以下任一条件,触发多专家联动:
- 意图是"开发"(新功能必须 4 专家全栈)
- 意图是"审查/审计/评审" + 对象是"模块/服务"(需多维度)
- 涉及多个 Bug 类型(如"修复并发+鉴权 Bug")
- 涉及状态变更 + 并发(架构+品控联动)

### 第 5 步:顺序编排

```
新功能开发: xiaoxi-pm → xiaosheng-architect → xiaoyu-security → xiaohe-qa
Bug 修复:   [主责] → [关联]
代码审查:   xiaosheng-architect → xiaohe-qa → xiaoyu-security → xiaoxi-pm
上线发布:   xiaoxi-pm → xiaohe-qa
安全审计:   xiaoyu-security → xiaosheng-architect
```

### 第 6 步:调用对应 skill

按顺序读取并执行 4 个 SKILL.md 中的"标准工作流"和"修复方案模板"。

### 第 7 步:汇总输出

按用户偏好格式输出(默认:对话 + 报告双模式):
- **对话回答**: 1-3 段简短结论
- **结构化报告**: 标题/表格/清单/代码(便于复制到文档)

## 四、4 专家 Skill 调用顺序(标准)

### 4.1 新功能开发(完整 4 步)

```
Step 1: xiaoxi-pm         → 工厂痛点 + 隐性需求(场景输入)
Step 2: xiaosheng-architect → 技术选型 + 架构方案(技术设计)
Step 3: xiaoyu-security   → 鉴权/越权/注入预防(安全审计)
Step 4: xiaohe-qa         → 3层验证 + 回归测试(品控验收)
```

### 4.2 Bug 修复(2-3 步)

```
Step 1: [主责专家]        → 定位问题 + 给出修复方案
Step 2: [关联专家]        → 验证影响 + 补充方案
Step 3: xiaoxi-pm         → 灰度策略(如需上线)
```

### 4.3 上线审计(4 步)

```
Step 1: xiaoyu-security   → 22 项安全检查清单
Step 2: xiaosheng-architect → 架构风险评估
Step 3: xiaohe-qa         → 3 层验证
Step 4: xiaoxi-pm         → 4 周灰度 + 工厂验收
```

### 4.4 工厂场景验证(1-2 步)

```
Step 1: xiaoxi-pm         → 5 类角色痛点 + 隐性需求
Step 2: xiaosheng-architect → 场景落地的技术可行性
```

## 五、典型调度示例(模拟你的对话)

### 示例 1:你说"工单模块加个撤回功能"

```
🎯 调度判断: 意图=开发 + 对象=工单模块 + 新功能
📋 调度顺序: 4 专家全栈

Step 1 [xiaoxi-pm]:
  跟单员痛点: FM-004 订单变更影响难追踪
  隐性需求: 误操作撤销(5分钟内)
  业务验收: 工人手机 3 秒内完成,老板浏览器看状态

Step 2 [xiaosheng-architect]:
  技术方案: SELECT ... FOR UPDATE 行锁 + 状态机白名单
  架构评估: 必须加事务,避免部分提交
  并发安全: 原子 UPDATE + 乐观锁 rowcount 校验

Step 3 [xiaoyu-security]:
  鉴权: 必须 require_role('admin', 'manager')
  越权: 水平越权校验(只能撤回自己部门的工单)
  审计: 必须写 history 记录

Step 4 [xiaohe-qa]:
  状态机: created→confirmed→archived(可撤回状态)
  边界: 5分钟内/30分钟内/24小时内
  3层验证: 调API看DB + 老板浏览器看 + 全中文错误
```

### 示例 2:你说"报工那里有 bug,工人同时报会超量"

```
🎯 调度判断: 意图=修复 + Bug类型=并发
📋 调度顺序: xiaosheng-architect(主) + xiaohe-qa(辅)

Step 1 [xiaosheng-architect]:
  问题定位: TOCTOU 无原子操作
  修复方案: 原子 UPDATE(乐观锁)
  代码模板: server.py:L2172 修改为...
  验证: 并发测试 100 工人同时报工不超量

Step 2 [xiaohe-qa]:
  状态机: completed_qty 不能超 plan_qty
  边界: 超 5% 警告,超 20% 拒绝
  3层验收: 跑并发 + 工人在手机看到绿色对勾 + 错误全中文
```

### 示例 3:你说"上线前帮我审一下"

```
🎯 调度判断: 意图=上线审计
📋 调度顺序: 4 专家全栈

Step 1 [xiaoyu-security]:
  22 项安全检查清单 → 输出 [PASS/FAIL/WARN]
Step 2 [xiaosheng-architect]:
  架构风险评估 → 连接池/跨库/性能
Step 3 [xiaohe-qa]:
  3 层验收清单 → 技术/业务/异常
Step 4 [xiaoxi-pm]:
  4 周灰度策略 → 工厂真实环境验证
```

### 示例 4:你说"这个接口有问题"

```
🎯 调度判断: 意图=审查 + 对象=接口
📋 调度顺序: xiaoyu-security(主) + xiaosheng-architect(辅) + xiaohe-qa(辅)

Step 1 [xiaoyu-security]:
  越权检查 → 是否有角色校验
  注入检查 → 是否有 SQL 拼接
  鉴权检查 → 是否有 JWT 验证
Step 2 [xiaosheng-architect]:
  并发检查 → 是否有行锁
  错误处理 → 是否脱敏
Step 3 [xiaohe-qa]:
  边界检查 → 是否有 0/负数/超大数校验
  状态机 → 是否白名单
```

## 七、调度日志与可追溯性

每次调度会记录:
- 用户输入原文
- 语义解析结果(意图/对象/约束)
- 调用的专家列表
- 调度顺序
- 各专家输出
- 最终汇总(如有分歧,记录决策点)

格式:
```markdown
## 调度日志 [2026-06-25 14:30:00]

- 用户输入: "工单模块加个撤回功能"
- 语义解析: 意图=开发, 对象=工单模块, 约束=新功能
- 调度方案: 4 专家全栈
- 调用顺序: PM → 架构 → 安全 → 品控
- 各专家输出: [PM 痛点+需求] / [架构方案] / [安全审计] / [品控验收]
- 决策点: 无
- 输出格式: 对话 + 报告
```
