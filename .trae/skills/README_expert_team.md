## 🎯 场景 → Skill 速查表

### 场景 1:开发新功能

```
需求分析阶段 → 调用 xiaoxi-pm      (工厂痛点 + 隐性需求)
  ↓
设计阶段     → 调用 xiaosheng-architect (技术选型 + 架构评审)
  ↓
编码阶段     → 调用 xiaoyu-security   (鉴权 + 越权 + 注入预防)
  ↓
测试阶段     → 调用 xiaohe-qa        (功能正确性 + 边界 + 回归)
```

### 场景 2:修复生产 Bug

```
定位问题     → 4 专家联动分析
  ↓
P0 安全     → xiaoyu-security      (鉴权/越权/注入)
P0 并发     → xiaosheng-architect   (原子 UPDATE/行锁)
P0 业务     → xiaohe-qa            (状态机/边界)
P0 场景     → xiaoxi-pm            (工厂实际/验收)
  ↓
修复方案     → 各专家提供模板代码
  ↓
灰度上线     → xiaoxi-pm 4 周放量策略
  ↓
验收测试     → xiaohe-qa 3 层验证
```

### 场景 3:代码审查(Code Review)

```
架构合理性   → xiaosheng-architect
  ↓
业务正确性   → xiaohe-qa
  ↓
安全合规性   → xiaoyu-security
  ↓
场景适配性   → xiaoxi-pm
```

### 场景 4:上线前安全审计

```
全量安全扫描 → xiaoyu-security (22 项检查清单)
  ↓
架构风险评估 → xiaosheng-architect (连接池/跨库)
  ↓
业务功能验证 → xiaohe-qa (3 层验证)
  ↓
灰度方案设计 → xiaoxi-pm (4 周放量 + 回滚)
```

## 🔄 4 专家协作模式

### 模式 1:全栈修复(4 专家联动)

```
┌──────────────────────────────────────────────────────────┐
│                   4 专家全栈修复流程                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  小钰(安全)  →  发现漏洞     →  越权/注入/鉴权         │
│      ↓                                                   │
│  小圣(架构)  →  设计方案     →  原子 UPDATE/行锁/outbox  │
│      ↓                                                   │
│  小贺(品控)  →  验证修复     →  状态机/边界/3层验证     │
│      ↓                                                   │
│  小曦(PM)    →  灰度上线     →  4周放量/工厂验收        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 模式 2:单一职责调用

| 任务类型 | 单 Skill 调用 | 工作量 |
|---------|--------------|:------:|
| 架构评审 | xiaosheng-architect | 2-4h |
| 状态机梳理 | xiaohe-qa | 1-2h |
| 痛点调研 | xiaoxi-pm | 4-8h |
| 安全扫描 | xiaoyu-security | 2-4h |

### 模式 3:跨专家争议解决

```
分歧出现  →  4 专家分别提交立场
  ↓
TRAE AI 汇总对比  →  共识方法:逐项对比,取分歧最小方案
  ↓
争议项列出 4 专家立场 + 决策
  ↓
用户拍板最终方案
```

## 🛠️ 实战调用示例

### 示例 1:新增一个"工单撤回"功能

```python
# 步骤 1: 工厂痛点分析
skill: xiaoxi-pm
input: "工单撤回功能的工厂痛点"
output: 跟单员 FM-004 痛点 + 隐性需求 #1 误操作撤销

# 步骤 2: 架构设计
skill: xiaosheng-architect
input: "工单撤回的并发安全方案"
output: SELECT ... FOR UPDATE 行锁 + 原子 UPDATE

# 步骤 3: 安全审计
skill: xiaoyu-security
input: "工单撤回的越权风险"
output: 必须 require_role + require_owner_or_admin

# 步骤 4: 品控测试
skill: xiaohe-qa
input: "工单撤回的功能正确性 + 3 层验证"
output: 状态机白名单 + 边界场景 + 3 层验收清单

# 步骤 5: 灰度上线
skill: xiaoxi-pm
input: "工单撤回的灰度策略"
output: 4 周放量 + git revert + 工厂真实环境验证
```

### 示例 2:修复一个 P0 并发 Bug

```python

# 步骤 4: PM 灰度
```

## 💡 最佳实践

### ✅ DO(推荐)

1. **新功能开发**:4 专家按"PM→架构→安全→品控"顺序全栈调用
2. **Bug 修复**:按 Bug 类型精准调用对应专家
3. **代码审查**:4 专家分工审查不同维度
4. **上线审计**:xiaoyu-security 全量 + 其他专家补充

### ❌ DON'T(避免)

1. 不要单凭一个专家的判断就做架构决策
2. 不要跳过 xiaoxi-pm 的工厂实际场景验证
3. 不要让 xiaoyu-security 只做"形式审查"而不提供代码示例
4. 不要让 xiaosheng-architect 只给"理论方案"而不考虑落地成本
