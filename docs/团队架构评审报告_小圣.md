## 一、现有架构分析

### 1.1 技术栈评估

| 技术选型 | 评估 | 理由 | 评级 |
|---------|------|------|------|
| **Flask** | ✅ 合适 | 工厂软件不需要 FastAPI 的异步特性，Flask 够用且稳定 | ⭐⭐⭐⭐ |
| **PyMySQL** | ⚠️ 需注意 | 连接池管理是自己实现的，不是 DBUtils，高并发场景有风险 | ⭐⭐⭐ |
| **单体架构** | ✅ 正确 | 工厂软件初期单体是对的，微服务过度设计 | ⭐⭐⭐⭐ |
| **Tkinter 桌面端** | ⚠️ 可接受 | 工厂内网使用，tkinter够用，但长远看应该 Web 化 | ⭐⭐⭐ |
| **蓝图 Blueprint** | ✅ 良好 | 多服务分离（5003/5008/桌面端）架构清晰 | ⭐⭐⭐⭐ |
| **企业微信集成** | ✅ 实用 | 工厂标配，减少培训成本 | ⭐⭐⭐⭐ |

**结论**: 技术栈选择总体正确，符合工厂软件的务实风格。Flask + 单体 + 蓝图是我在富士康、比亚迪项目常用的组合。

### 1.2 数据层评估

#### 1.2.1 连接池设计（core/db.py）

```
优点：
- 单例模式正确
- 支持 SQLite 离线模式（开发/断网场景）
- 事务上下文管理器（with db.cursor()）
- ping() 健康检查自动重连

问题：
- ⚠️ pool_size 硬编码上限 50，高并发报工（50工人同时提交）可能不够
- ⚠️ 手动实现连接池而非使用 DBUtils（成熟库），存在边界 bug 风险
- ⚠️ 没有连接超时重试机制
```

**大厂经验**: 比亚迪工厂MES系统曾因连接池耗尽导致整条产线停工 2 小时。

#### 1.2.2 DAO 层问题（models/production.py, models/order.py）

| 问题 | 影响 | 严重程度 |
|------|------|---------|
| **每次操作单独 get_connection()** | 连接频繁创建/销毁，性能损耗 | 🔴 高 |
| **create() 中 8 次独立 cursor 查询** | N+1 查询问题 | 🔴 高 |
| **同步 requests.post() 事件发布** | 阻塞主事务，timeout=2 可能丢消息 | 🟡 中 |
| **跨库同步（container_center）** | MySQL 直连，不走连接池 | 🟡 中 |

## 二、大厂经验教训（踩坑清单）

### 2.1 高发问题清单

| 坑 | 原因 | 预防措施 | 当前状态 |
|----|------|---------|---------|
| **连接池耗尽** | 慢查询占用连接 + pool_size 不足 | 1. 扩大 pool_size 到 100+<br>2. 添加查询超时<br>3. 添加连接池监控 | 🔴 未处理 |
| **跨库事务不一致** | MySQL 直连 container_center，不走本地连接池 | 1. 统一连接管理<br>2. 添加重试机制<br>3. 事件驱动替代同步调用 | 🟡 部分处理 |
| **内存泄漏** | tkinter + PIL 图片缓存 | 定期清理图片对象 | 🟢 未发生 |
| **定时任务重复执行** | 多实例部署 + 调度器无锁 | 分布式锁或单实例调度 | 🟡 未见问题 |
| **数据孤岛** | 5003/5008/桌面端各自维护数据 | 统一数据源 + 事件同步 | 🟡 已有同步机制 |
| **大表查询慢** | orders/process_records 无分页/无索引 | 添加 LIMIT + 索引 | 🟡 已有分页 |
| **Excel 导入超时** | 批量导入单事务太长 | 分批提交（500条/批） | 🟡 未见处理 |
| **消息队列丢消息** | 同步 requests.post() 无 ACK | 消息持久化 + 消费确认 | 🟡 部分处理 |

### 2.2 真实案例教训

**案例1: 华为工厂 - 批量报工风暴**
```
现象: 早班开工后 8:00-8:30，50 个工人同时扫码报工
结果: MySQL 连接数瞬间打满，新请求全部超时
根因: 每条报工都开一个新连接，pool_size=10 完全不够
解决: 连接池扩容 + 报工请求合并批量处理
```

**案例2: 富士康 - Excel 导入死锁**
```
现象: 导入 2000 行订单时，MySQL 锁等待超时
根因: 单个事务 UPDATE 太多行，锁链太长
解决: 分批导入（500条/事务）+ 添加进度条
```

## 三、架构改进建议（按优先级）

### P0 - 立即处理（影响生产稳定性）

| 改进项 | 当前方案 | 建议方案 | 理由 |
|--------|---------|---------|------|
| **连接池容量** | max=50 | 扩大到 100-200 | 50工人同时报工场景 |
| **慢查询超时** | 无 | 添加 `query_timeout=30s` | 防止连接被慢查询阻塞 |
| **跨库连接管理** | 直连 MySQL | 统一走连接池 + 添加重试 | 避免连接泄漏 |

### P1 - 近期处理（影响开发效率）

| 改进项 | 当前方案 | 建议方案 | 理由 |
|--------|---------|---------|------|
| **N+1 查询优化** | create() 中 8 次独立查询 | 合并为 2-3 次 JOIN 查询 | 降低数据库往返 |
| **事件发布异步化** | requests.post() 同步 | 消息队列（Redis/Broker） | 避免阻塞主事务 |
| **分页 LIMIT 默认值** | 无限制 | 默认 LIMIT 100 | 防止全表扫描 |

### P2 - 中期规划（技术债务）

| 改进项 | 当前方案 | 建议方案 | 理由 |
|--------|---------|---------|------|
| **连接池库** | 手动实现 | 使用 DBUtils.PooledDB | 成熟库，更稳定 |
| **事务边界** | 每次操作独立事务 | 长事务拆分 + 最终一致性 | 降低锁等待 |
| **缓存策略** | 无 | Redis 热点数据缓存 | 降低数据库压力 |
| **监控告警** | 无 | 添加 Prometheus 指标 | 提前发现问题 |

## 四、高并发场景分析

### 4.1 场景1: 早班集中报工（8:00-8:30）

```
当前瓶颈分析:
┌─────────────────────────────────────────────────────────────┐
│ 50 工人同时扫码 → 50 个并发请求                            │
│     ↓                                                       │
│ 每请求 get_connection() → 连接池耗尽                       │
│     ↓                                                       │
│ 新请求等待/超时 → 工人无法报工 → 产线停摆                   │
└─────────────────────────────────────────────────────────────┘
```

**优化建议**:
| 措施 | 优先级 | 工作量 |
|------|--------|--------|
| 1. 连接池扩容到 150+ | P0 | 1小时 |
| 2. 添加连接超时 + 重试 | P0 | 2小时 |
| 3. 报工请求合并批量写入 | P1 | 1天 |
| 4. Redis 缓存工人信息 | P2 | 半天 |

### 4.2 场景2: 订单批量发布（一次 200 个工单）

```
当前瓶颈分析:
┌─────────────────────────────────────────────────────────────┐
│ 200 工单同时发布 → 200 次 INSERT + 200 次工序生成          │
│     ↓                                                       │
│ 每工单 8 次 DB 查询 → 1600 次往返                          │
│     ↓                                                       │
│ 单事务耗时 ~30s → 锁等待超时                                │
└─────────────────────────────────────────────────────────────┘
```

**优化建议**:
| 措施 | 优先级 | 工作量 |
|------|--------|--------|
| 1. 批量 INSERT（INSERT VALUES） | P1 | 2小时 |
| 2. 工序生成异步化 | P1 | 1天 |
| 3. 分批提交（50工单/事务） | P1 | 1小时 |

### 4.3 场景3: 质检数据实时统计

```
当前瓶颈分析:
┌─────────────────────────────────────────────────────────────┐
│ Dashboard 大屏 → 实时查询统计                                │
│     ↓                                                       │
│ SELECT COUNT/GROUP BY → 全表扫描                            │
│     ↓                                                       │
│ 每次刷新 5-10s → 大屏卡顿                                   │
└─────────────────────────────────────────────────────────────┘
```

**优化建议**:
| 措施 | 优先级 | 工作量 |
|------|--------|--------|
