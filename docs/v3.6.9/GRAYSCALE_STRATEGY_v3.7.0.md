# 灰度上线策略 - v3.7.0 重构

> **创建日期**: 2026-06-28
> **关联版本**: v3.7.0 架构重构（18周计划）
> **性质**: P0 文档，Week 0 启动前必须签字
> **审计来源**: 4专家审计（小曦PM）→ C-1

---

## 一、核心原则

1. **安全第一**: 任何阶段出问题，立即回滚，不影响工厂生产
2. **小步快跑**: 每次放量不超过 30%，发现异常立即暂停
3. **可观测**: 每次放量必须配套监控，异常即告警
4. **可回滚**: 任何阶段都必须能在 10 分钟内回滚到上一稳定状态

## 三、五次放量策略（每批独立验收）

### 3.1 每次放量的标准流程

```
阶段验收签字（技术+PM）
    ↓
新版本打包（git tag v3.7.0-{批次}-{日期}）
    ↓
通知相关人（开发+PM+工厂对接人）
    ↓
放量前健康检查（4-gate全绿）
    ↓
放量10%生产流量（1小时观察窗口）
    ↓
├─ 异常 → 立即回滚（≤10分钟）→ 复盘 → 修复
└─ 正常 → 放量30%（4小时观察窗口）
           ↓
    ├─ 异常 → 回滚至10% → 修复 → 再试
    └─ 正常 → 放量60%（24小时观察窗口）
               ↓
        ├─ 异常 → 回滚至30% → 修复 → 再试
        └─ 正常 → 放量100%
```

## 四、放量触发条件（Gate标准）

### 4.1 通过条件（必须全部满足）

| 指标 | 阈值 | 检测方式 |
|------|------|---------|
| pytest通过率 | ≥ 95% | CI自动检测 |
| P99响应时间 | ≤ baseline + 200ms | APM监控 |
| API错误率 | ≤ 0.5% | 日志监控 |
| bandit安全扫描 | 0 HIGH漏洞 | CI自动检测 |
| 并发压测 | 100并发×10轮 零崩溃 | 性能测试 |

### 4.2 告警条件（满足任一即触发告警）

| 指标 | 告警阈值 | 处理方式 |
|------|---------|---------|
| API错误率 | > 0.5% | 观察15分钟，仍超则回滚 |
| P99响应时间 | > baseline + 500ms | 观察30分钟，仍超则回滚 |
| 服务不可用 | 任意服务5xx | 立即回滚 |
| 数据库连接数 | > max_connections 80% | 观察10分钟，仍超则回滚 |
| 特定接口错误 | 单一接口错误率 > 5% | 立即回滚该接口 |

## 五、回滚机制

### 5.1 回滚触发条件

满足以下任一条件，必须立即回滚：

| 优先级 | 条件 | 允许时间 |
|---|---|
| P0 | 工厂工人手机报工失败率 > 1% | 5分钟内回滚 |
| P0 | 老板浏览器无法查看数据 | 5分钟内回滚 |
| P1 | API错误率 > 5% 持续 > 15分钟 | 30分钟内回滚 |
| P1 | P99响应时间 > 3秒 持续 > 30分钟 | 30分钟内回滚 |
| P2 | 非核心接口错误率 > 10% | 1小时内决策 |

### 5.2 回滚执行流程

```
触发回滚
    ↓
通知：开发负责人 + PM + 工厂对接人（微信群）
    ↓
执行回滚：git revert {bad_commit} && 重新部署
    ↓
验证：健康检查端点 + 核心API抽检
    ↓
确认：PM签字确认"回滚完成，生产正常"
    ↓
复盘：24小时内输出复盘报告
    ↓
决策：修复后重新放量 / 放弃该批次
```

### 5.3 回滚执行人

| 角色 | 职责 |
|------|------|
| **开发负责人** | 执行 git revert + 重新部署 |
| **运维（若存在）** | 执行 k8s rollout undo / docker-compose restart |
| **PM（小曦）** | 确认工厂生产状态，决策是否继续放量 |
| **AI（当前）** | 记录复盘报告，更新风险矩阵 |

## 六、灰度前置条件（v3.7.1版）

> ⚠️ **脱水说明**：v3.7.0的灰度Gate写了很多指标，但没有说清楚"谁来量这些指标"。
> 工厂系统没有Prometheus/Grafana，以下是真实的最低可执行方案。

### 6.0 灰度放量前必须满足的3个前提

```
【每个G放量签字前，必须确认以下3项】

1. 监控脚本可运行
   $ python scripts/monitor.py
   输出包含：DB连接数、API成功率、P99延迟
   → 如果脚本跑不起来，灰度Gate无法判断

2. 企微Webhook配置完成
   monitor.py 的 WECHAT_WEBHOOK 配置有效
   → 测试发一条消息，确认能收到

3. 监控数据有baseline对比
   放量前状态记录：当前DB连接数、API成功率、P99
   → 没有baseline，增量异常无法判断

如果以上3项不满足，禁止放量签字。
```

## 七、监控指标体系

### 7.1 核心监控指标（每次放量必须观察）

| 指标 | 正常范围 | 告警阈值 | 监控工具 |
|------|---------|---------|---------|
| API成功率 | ≥ 99.5% | < 99% | 日志聚合 |
| P50响应时间 | ≤ 200ms | > 500ms | APM |
| P99响应时间 | ≤ 1000ms | > 1500ms | APM |
| MySQL连接数 | ≤ 80% max | > 90% max | 数据库监控 |
| 错误日志速率 | ≤ 1条/分钟 | > 10条/分钟 | 日志聚合 |
| 特定接口错误 | 0 | > 0 | API监控 |

### 7.2 工厂业务指标（每日检查）

| 指标 | 正常范围 | 异常表现 |
|------|---------|---------|
| 工人手机报工成功率 | ≥ 99% | 工人反馈"报不了" |
| 老板浏览器数据加载 | ≤ 3秒 | 老板反馈"看不到" |
| 质检提交成功率 | ≥ 99% | 质检员反馈"提交不了" |
| 订单状态同步延迟 | ≤ 30秒 | 桌面端与移动端数据不一致 |
