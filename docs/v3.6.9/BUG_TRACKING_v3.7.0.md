## 二、Bug优先级定义

| 优先级 | 定义 | 修复时限 | 阻塞上线 |
|--------|------|---------|:-------:|
| **P0** | 工厂完全无法使用 / 数据损坏 / 安全漏洞 | 24小时内 | ✅ 阻塞 |
| **P1** | 功能不可用但有绕过方案 | 72小时内 | ✅ 阻塞 |
| **P2** | 体验问题，不影响核心功能 | 2周内 | ❌ 不阻塞 |
| **P3** | 优化建议，cosmetic | 纳入v3.8.0 | ❌ 不阻塞 |

每个Bug必须包含以下字段：

```markdown

**发现来源**: [Layer1-代码Review / Bug清单已有 / 灰度放量 / 工厂反馈 / P0修复]
**发现批次**: [G1 / G2 / G3 / G4 / G5 / Week 0-19]
**优先级**: P0 / P1 / P2 / P3

### 位置信息
**文件**: `mobile_api_ai/xxx.py`
**行号**: L123
**函数/路由**: `/api/xxx`

### 问题描述
[用大白话描述问题，不是技术细节]

### 复现步骤
1. [步骤1]
2. [步骤2]
3. [预期结果 vs 实际结果]

### 修复方案
[用大白话描述修复思路]

### 关联测试用例
- [ ] 测试用例1: `test_xxx_功能名`
- [ ] 测试用例2: `test_xxx_边界情况`

### 状态
- [ ] **Open**（待修复）
- [ ] **In Progress**（修复中，关联commit: `xxx`）
- [ ] **Closed**（已修复，验证commit: `xxx`，验证人: xxx）
```

## 四、Bug流转状态机

```
[New] → [Triaged] → [Open] → [In Progress]
   ↓        ↓          ↓          ↓
[Duplicate] [Won't Fix]  [Closed]  [Open ← 回退]
                                          ↓
                                    [Closed（修复后）]
                                          ↓
                                    [Reopened（如有新问题）]
```

### 状态说明

| 状态 | 定义 | 操作人 |
|------|------|--------|
| **New** | 刚发现，待分类 | 发现人 |
| **Triaged** | 已分类，确认优先级 | 开发+PM |
| **Open** | 确认修复，分配给开发 | 开发 |
| **In Progress** | 正在修复 | 开发 |
| **Closed** | 修复完成，通过验证 | 开发+品控 |
| **Reopened** | 修复后仍有问题 | 品控 |
| **Duplicate** | 重复Bug，关联到已有Bug | 开发 |
| **Won't Fix** | 不修（设计如此或影响太小） | PM签字 |

### 5.1 Layer1开发期间（Week 3-8）：Bug随发现随录入


## 七、Bug关闭标准（3层验证）

| 验证层 | 验证方式 | 通过标准 |
|--------|---------|---------|
| **技术验证** | 跑关联测试用例 | pytest PASS + 手动复现确认 |
| **业务验证** | PM确认业务场景 | 工厂实际使用无异常 |
| **异常验证** | 边界值/压力测试 | 极端输入不崩溃 |

**关闭流程**：
1. 开发自测 → 关联测试用例全部PASS
2. PM业务验收 → 工厂实际场景确认
3. 品控签字 → Bug清单状态改为Closed
