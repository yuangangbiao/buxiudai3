## 一、原始需求

用户期望实现**全面模块化**的系统架构：

1. **模块化操作**：所有数据操作都变成可插拔的模块
2. **统一入口**：所有数据库写入走统一入口
3. **统一存储**：消除跨库直接查询
4. **数据分层**：
   - steel_belt：桌面端权威数据（只写核心状态）
   - container_center：完整业务数据（读/写）
5. **实时同步**：手机端报工后桌面端立即可见
6. **性能优先**：前端加载速度快

---

## 二、需求理解

### 2.1 当前架构问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 跨库直接查询 | `_core.py` 直接连接 steel_belt | 加载慢、耦合高 |
| 多处直接写 steel_belt | `_sync_to_mysql()`, `sync_bridge.py` | 架构混乱 |
| 无统一入口 | 报工、状态更新各自写 | 状态不一致 |
| 无模块化 | 代码散落在各处 | 难维护 |

### 2.2 目标架构特征

| 特征 | 说明 |
|------|------|
| **模块化** | 数据操作封装为独立模块，可插拔 |
| **统一入口** | 所有 steel_belt 写入走 8008 桥接 |
| **SSOT** | 单一数据源，orders_local 等本地表作为读取缓存 |
| **分层存储** | steel_belt 只存核心状态，container_center 存完整数据 |
| **实时同步** | 报工后 < 3秒 同步到 steel_belt |

## 四、数据分层设计

### 4.1 数据库职责

| 数据库 | 职责 | 数据类型 |
|--------|------|----------|
| **steel_belt** | 桌面端权威数据 | 核心状态（订单状态、生产状态） |
| **container_center** | 容器中心完整数据 | 全部业务数据 |
| **inventory_db** | 库存数据 | 库存、出入库记录 |

### 4.2 同步策略

| 数据类型 | 写入位置 | 同步方向 | 同步方式 |
|----------|----------|----------|----------|
| 订单基础信息 | container_center | → steel_belt | 异步（队列） |
| 生产状态 | container_center | → steel_belt | 实时 |
| 报工记录 | container_center | → steel_belt | 异步（队列） |
| 质检记录 | container_center | → steel_belt | 异步（队列） |

## 五、API 统一入口

### 5.1 8008 统一写入 API

| API | 方法 | 功能 | 模块 |
|------|------|------|------|
| `/api/dal/order/create` | POST | 创建订单 | OrderModule |
| `/api/dal/order/status` | POST | 更新订单状态 | OrderModule |
| `/api/dal/process/report` | POST | 报工 | ProcessModule |
| `/api/dal/process/publish` | POST | 发布工序 | ProcessModule |
| `/api/dal/quality/report` | POST | 质检上报 | QualityModule |
| `/api/dal/sync-to-steelbelt` | POST | 同步到 steel_belt | DAL内部 |

### 5.2 模块调用示例

```python
# 调用方代码（简洁统一）
from dal import OrderModule, ProcessModule

# 创建订单
OrderModule.create({
    'order_no': 'ORD-202606130001',
    'customer_name': '客户A',
    'product_name': '不锈钢网带',
    'quantity': 100
})

# 报工
ProcessModule.report({
    'order_no': 'ORD-202606130001',
    'step_name': '编织',
    'quantity': 50,
    'operator': '张三'
})
```

## 六、边界确认（硬性约束）

### 6.1 硬性约束

| 约束 | 说明 | 影响 |
|------|------|------|
| 🚫 **不改名** | 现有所有函数名、API 名、字段名、变量名保持不变 | 改造只能新增，不能修改名称 |
| 🚫 **不改前端** | 前端 HTML、JS、Vue、CSS、URL 都不变 | 前端调用方式不变 |
| 🚫 **不改展示** | 工序、报工、质检等前端展示页面不变 | 只动后端逻辑 |
| ✅ **steel_belt 只存核心** | order_no, status, plan_start, plan_end, updated_at | 避免大字段 |
| ✅ **不乱存大容量** | 报工记录、质检详情等存 container_center | 减轻 steel_belt 压力 |
| 🚫 **云端不调用 8008** | 云端服务（cloud_*.py）不直接调用 8008 桥接 | 必须通过 5003 消息入口 |
| 🚫 **5003 消息入口** | **所有消息类调用**必须经过 5003 调度中心 | 5003 是消息总线，非 API Gateway |

### 6.2 改造范围

| 包含 | 不包含 |
|------|--------|
| 数据访问层（DAL）封装 | 桌面端 PyQt5 改造 |
| 存储层（STO）封装 | 桌面端数据库 |
| 8008 统一入口（内部） | 前端 HTML/JS/Vue |
| SSOT 本地表 | API 名称、字段名、URL |
| 内部调用重构 | 前端展示页面 |

### 6.3 技术约束

| 约束 | 说明 |
|------|------|
| Python 3.14.3 | 固定版本 |
| Flask 3.1.3 | Web 框架 |
| MySQL | 数据库 |
| 不破坏现有功能 | 渐进式改造 |
| 现有 API 100% 兼容 | 接口名、参数、返回值不变 |

## 六点五、改造原则

### 原则 1: 新增优于修改
- 优先新增模块/函数
- 现有函数内部重写，签名保持不变
- 保持 100% 向后兼容

### 原则 2: 内部重构
- 现有 `_sync_to_mysql()` 内部改为走 8008
- 函数名、入参、出参都不变
- 调用方完全无感知

### 原则 3: 前端零侵入
- 所有 URL 保持不变
- 所有 API 名称不变
- 所有返回数据格式不变
- 前端展示效果不变

### 原则 4: steel_belt 轻量化
| 字段 | 写入 | 大小 |
|------|------|------|
| order_no | ✅ | 字符串 |
| status | ✅ | 字符串 |
| plan_start | ✅ | 时间 |
| plan_end | ✅ | 时间 |
| updated_at | ✅ | 时间 |
| ~~content~~ | ❌ | 大字段 |
| ~~metadata~~ | ❌ | 大字段 |
