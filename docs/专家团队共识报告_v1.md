## 三、修复方案共识（争议最小项）

### 3.1 已达成共识的 P0 修复方案

| Bug | 共识修复方案 | 专家来源 |
|-----|-------------|---------|
| **P0-H import random** | `desktop_web/server.py:3144` 顶部加 `import random` | 小圣确认 |
| **P0-E 订单状态机** | 在 `OrderDAO.update_status` 加 `ALLOWED_TRANSITIONS` 白名单校验 | 小贺 + 小圣共识 |
| **P0-F 工序假成功** | 在 `api_process_complete` 和 `api_process_reset` 加 `if cur.rowcount == 0: return 400`，**不改动 _call_dispatch** | 小圣（反对原方案）|
| **P0-J 外协/维修/排产撤回无锁** | `SELECT ... FOR UPDATE` → `UPDATE is_deleted=1` 包裹事务 | 小圣 + TRAE 共识 |
| **P0-G 并发报工** | **原子 UPDATE**（乐观锁）：`UPDATE ... SET completed_qty=completed_qty+%s WHERE id=%s AND completed_qty+%s <= plan_qty`，`rowcount==0` 说明超计划 | 小圣（反对 SELECT+check+UPDATE）|
| **P0-D dispatch_token** | 已在昨晚 P0 修复中完成（base64 对齐 5003）| 小贺 P0 报告 |

### 3.2 存在分歧的 P0 修复方案（需决策）

#### 分歧 1：P0-I 跨库同步原子保证

| 立场 | 方案 | 理由 |
|------|------|------|
| **小圣** | outbox 模式：5001 写 process_records **同事务** 写 outbox_events 表，后台 worker 异步推 5003 | 保证最终一致性，进程崩溃也不丢 |
| **小曦** | 保持分离 + 强化同步（指数退避 3 次）| 工厂断网是常态，同步不是业务阻塞；合并库风险大 |
| **共识** | **分两步走**：短期（1 天）加 service token + 告警；中期（3 天）加 outbox | 平衡即时安全 + 长期架构 |

#### 分歧 2：P0-K outbox 死信告警

| 立场 | 方案 | 理由 |
|------|------|------|
| **小圣** | 加监控 + 告警（写入告警表 + 微信通知）| 死信=数据丢失，必须有人知道 |
| **小曦** | 重试 5 次后写入 retry_queue，5003 恢复后批量补偿 | 工厂弱网需要更多重试机会 |
| **共识** | **outbox 模式 + 5 次指数退避** | 结合两者优点 |
