## 二、认证机制深度对比

### 2.1 桌面端认证

**认证位置**: [models/operator.py#L67-115](file:///d:/yuan/不锈钢网带跟单3.0/models/operator.py#L67-115)

```python
@staticmethod
def login(operator_id, password):
    cursor.execute("""
        SELECT id, operator_id, name, role, status, password, password_salt
        FROM operators WHERE operator_id=%s AND status='正常'
    """, (operator_id,))  # ✅ 参数化查询

    if verify_password(password, stored_password, stored_salt):  # ✅ bcrypt验证
        return {'id': row['id'], 'operator_id': row['operator_id'], ...}
```

**特点**:
- 直接查询数据库操作员表
- 使用 bcrypt 盐值哈希验证密码
- 登录成功后更新 `last_login` 时间戳
- 无网络传输，本地应用

### 2.2 web5001 认证

**认证位置**: [desktop_web/server.py#L603-618](file:///d:/yuan/不锈钢网带跟单3.0/desktop_web/server.py#L603-618)

```python
@app.route('/api/login', methods=['POST'])
def api_login():
    data = request.get_json(silent=True) or {}
    r = requests.post(f'{DISPATCH_BASE}/api/login', json=data, timeout=5)  # 代理到5003
    if r.status_code == 200 and body.get('code') == 0:
        user = body['data']
        token = _secrets.token_hex(32)  # 32字节强随机
        session['dispatch_token'] = token
        session['dispatch_user'] = user
        session['csrf_token'] = _secrets.token_hex(16)  # CSRF令牌
```

**特点**:
- 代理登录请求到 5003 调度中心
- session 存储用户信息
- CSRF Token 生成（16字节）
- **JWT_SECRET_KEY 环境变量强制检查**

**认证装饰器**:
```python
# server.py#L26-33
def require_auth(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if 'dispatch_user' not in session:
            return jsonify({'code': 401, 'message': '未登录'}), 401
        return f(*args, **kwargs)
    return decorated
```

### 2.3 认证差异总结

| 维度 | 桌面端 | web5001 |
|------|--------|---------|
| 验证方式 | 数据库+bcrypt | 代理5003+session |
| 会话保持 | 内存变量 | Flask session |
| 密码传输 | 本地 | HTTP明文（需HTTPS） |
| CSRF | 不需要 | 必须（新增） |
| 攻击面 | 仅本地 | 网络+CSRF |

## 四、SQL 注入防护分析

### 4.1 安全区域（参数化查询）

大部分 SQL 使用参数化查询，安全：

```python
# ✅ 安全示例 - server.py#L214
cur.execute("SELECT status FROM orders WHERE id=%s", (order_id,))

# ✅ 安全示例 - server.py#L650
cur.execute("UPDATE orders SET is_deleted=1 WHERE order_no=%s", (order_no,))
```

### 4.2 风险区域（动态字段拼接）

#### 风险1: 工序更新动态字段（L2191）

```python
# server.py#L2157-2192
_PROCESS_UPDATE_FIELDS = {'process_name', 'worker', 'planned_qty', ...}
update_fields = []
update_values = []
# ... 省略构建逻辑 ...
cur.execute(
    "UPDATE process_records SET {} WHERE id=%s".format(','.join(update_fields)),  # ⚠️ 风险
    update_values + [process_id]
)
```

**风险分析**:
- `_PROCESS_UPDATE_FIELDS` 是白名单，控制了可用字段
- 但 `update_fields` 列表由 `'key=%s'` 拼接，理论上可注入
- 攻击面：若攻击者能控制 `_PROCESS_UPDATE_FIELDS` 或字段名

**实际风险**: 🟡 中（白名单缓解）

#### 风险2: 质检记录更新（L2856-2861）

```python
# server.py#L2856-2861
fields = []
for key in data:
    if key in _QUALITY_UPDATE_FIELDS:
        fields.append(f"{key}=%s")  # ⚠️ 风险
sql = "UPDATE quality_records SET {} WHERE id=%s".format(', '.join(fields))
cur.execute(sql, params)
```

**风险分析**:
- 同上，`_QUALITY_UPDATE_FIELDS` 是白名单
- 动态拼接 `key=%s`，但 key 来自白名单

**实际风险**: 🟡 中（白名单缓解）

#### 风险3: 发货单更新（L3171）

```python
# server.py#L3171
fields.append(f"{key}=%s")
```

**实际风险**: 🟡 中（白名单缓解）

## 五、XSS 防护分析

### 5.1 前端模板防护

大部分使用 `escapeHtml()` 函数转义：

```javascript
// process_admin.html#L264
`<th onclick="${c.noSort ? '' : `sortBy('${c.key}')`}">${escapeHtml(c.label)}${arrow}</th>`

// process_admin.html#L305-319
`<td>${escapeHtml(p.process_name || '-')}</td>`
```

### 5.2 未转义区域检查

```javascript
// process_admin.html#L383 ⚠️
select.innerHTML = '<option value="">-- 请选择工单 --</option>' + orders.map(o => {
    return `<option value="${o.id}" data-order-id="${o.order_id || o.id}">${escapeHtml(label)}</option>`;
});
```

**分析**: `o.id` 和 `o.order_id` 可能是数字，未转义风险较低

### 5.3 Jinja2 自动转义

服务器端模板默认开启自动转义：

```html
<!-- templates/*.html -->
{{ user_input }}  <!-- Jinja2 自动转义 -->
```

**评估**: ✅ XSS 防护整体较好
