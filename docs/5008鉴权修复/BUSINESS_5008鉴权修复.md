# 业务影响报告 - 5008 mobile API 写接口鉴权修复

## 1. 用户场景对比

| # | 用户角色 | 改善前（痛点） | 改善后（价值） |
|---|---------|---------------|---------------|
| 1 | 工厂老板 | 任何人能匿名调用审批/质检/报工写接口，可能被恶意刷数据或改单 | 未授权请求 401，必须登录后才可操作 |
| 2 | 车间操作员 | 自己 token 泄露后，别人能用同 token 写数据但无法识别身份 | 强制鉴权后，所有写入有 user_id 审计追溯 |
| 3 | 外部攻击者 | 直接 `curl -X POST` 即可触发报工/审批/质检，零门槛 | 缺 token 一律 401，业务接口被锁 |

## 2. 业务能力新增

| 业务流 | 新增/优化功能 | 影响范围 |
|--------|--------------|---------|
| 鉴权 | mobile API 写接口统一 JWT 鉴权（19 个） | 优化 |
| 鉴权 | 优先 `Authorization: Bearer`，fallback `X-Auth-Token` | 新增 |
| 鉴权 | 失败返 HTTP 401（非 200+code:401） | 修复 |
| 鉴权 | 自动设置 `request.user_id`/`user_name`/`user_role`/`user_team` | 新增（下游可读） |
| 鉴权 | cost + reports 9 个接口蓝图未加载（pre-existing `utils.validators` 问题） | ⚠️ 暂未生效 |

## 3. 不变更部分（防回归保护清单）

| # | 模块/功能 | 保护措施 | 验证方式 |
|---|----------|---------|---------|
| 1 | `/api/auth/login` 登录接口 | 未加装饰器（白名单） | E2E test_01 通过 |
| 2 | `/api/dispatch-center/*` 内部转发 | 未加装饰器（仅本地服务调用） | E2E test_05 通过（伪造 token 拒绝） |
| 3 | GET 类只读接口 | 未加装饰器（保持可用） | 业务流不中断 |
| 4 | 现有 session/cookie 机制 | 未改动 | E2E test_03/04/06/07 通过 |
| 5 | JWT 密钥 `JWT_SECRET_KEY` 来源 | 复用 `core.config` | 与 `auth.py` 同一密钥 |

## 4. 一句话总结

本次改动让 5008 mobile API 的 19 个写接口（审批/质检/报工/考勤/AI/报表/成本/监控/消息）从"匿名零鉴权可写"变为"必须登录 token 验证后才可写"，业务侧 E2E test_02 等 8 项鉴权测试全部通过。

## 已知风险/未闭环

- ⚠️ **9 个 cost + reports 写接口仍 404**：`services/factory` 模块缺失导致 cost/reports 蓝图未注册（pre-existing，E2E 不涉及但功能未恢复）
- ⚠️ **历史遗留进程**：之前残留 4788/24376/22352 等 5008 旧进程未完全清理，可能影响后续重启
- ⚠️ **登录 token 已包含 role/team 信息**但下游客服端代码可能未使用 `request.user_role`（建议后续 P1 接入 RBAC）
