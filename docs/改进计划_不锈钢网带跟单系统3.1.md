## 一、现状概述

### 1.1 系统定位

**不锈钢网带跟单系统 3.1**（简称"主系统"）是生产管理核心系统，负责：
- 订单管理（创建、确认、排产、跟踪）
- 生产工序管理（报工、质检、外协）
- 库存管理（原材料、成品）
- 报表与数据分析

**运行位置**：Windows 桌面端（车间办公室）

**技术栈**：Python 3.x + Tkinter（GUI）+ MySQL/SQLite

### 1.2 当前架构

```
┌─────────────────────────────────────────┐
│           桌面端主系统 3.1               │
├─────────────────────────────────────────┤
│  主窗口 (main_window.py)                │

│  ├─ 工序视图 (process_view.py)          │

├─────────────────────────────────────────┤

└─────────────────────────────────────────┘
```

### 1.3 当前痛点

| 痛点 | 说明 | 影响 |
|------|------|------|
| **事件驱动缺失** | 工序报工后主系统不主动通知外部 | 移动端无法实时获取状态变更 |
| **同步机制薄弱** | 桌面端与移动端数据不同步 | 报工后主系统任务单未减少 |
| **代码耦合度高** | DAO 直接操作 GUI，缺乏中间层 | 修改成本高，扩展困难 |
| **日志分散** | 主软件、云服务器日志各自独立 | 问题追溯困难 |
| **配置硬编码** | 端口、阈值、路径存在硬编码 | 部署不灵活 |
| **监控缺失** | 无 API 健康检查、熔断、限流 | 云端服务稳定性不足 |
| **备份粒度粗** | 全量备份，无增量/差异备份 | 备份效率低，存储浪费 |
| **前端架构旧** | Tkinter 可视化能力有限 | 大屏展示、仪表盘能力弱 |

## 三、改进计划

### 阶段一：架构解耦（基础改进）

**目标**：将主系统重构为事件驱动架构，打通与移动端的数据同步闭环。

#### 1.1 事件驱动集成

| 任务 | 文件 | 改进内容 | 优先级 |
|------|------|----------|--------|
| 主系统集成 EventBus | `core/event_bus.py` | 工序创建/报工/完成触发事件 | P0 |
| 集成 ContainerEventListener | `container_event_listener.py` | 监听主系统事件并转发至移动端 | P0 |
| 集成 DesktopContainerIntegration | `desktop_container_integration.py` | 接收移动端任务回执并更新状态 | P0 |
| 工序视图事件埋点 | `views/process_view.py` | 在报工/状态变更处触发事件 | P0 |
| 订单视图事件埋点 | `views/order_view.py` | 在订单创建/确认/变更处触发事件 | P1 |

**事件类型定义**：

```python
# core/events.py（新建）
class EventType:
    ORDER_CREATED = 'order.created'
    ORDER_CONFIRMED = 'order.confirmed'
    ORDER_SHIPPED = 'order.shipped'
    PROCESS_STARTED = 'process.started'
    PROCESS_REPORTED = 'process.reported'    # 工序报工
    PROCESS_COMPLETED = 'process.completed'  # 工序完成
    QC_PASSED = 'qc.passed'                  # 质检通过
    QC_REJECTED = 'qc.rejected'              # 质检拒绝
    INVENTORY_LOW = 'inventory.low'          # 库存预警
```

**数据流向（改进后）**：

```
工人扫码报工
  → 移动端 API 接收
    → 容器中心处理
      → 主系统 EventBus 触发 PROCESS_REPORTED
        → 所有监听器收到通知
          → 移动端任务池更新
          → 主系统工序视图刷新
          → 云端审计日志写入
```

#### 1.2 生产排单界面 - 自动发布任务开关

**需求背景**：

当前生产排单流程中，工序/工单排产确认后，需要人工手动点击"发布任务"按钮才能将任务推送至移动端（容器池）。工人经常因未及时发布任务而导致报工延误，且无法适应"确认即发布"的自动化场景需求。

**改进方案**：

在生产排单界面增加「自动发布任务」开关（默认关闭），由用户自行选择模式：

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| **手动模式**（默认） | 排产确认后 | 显示发布按钮，用户手动点击后才推送任务 |
| **自动模式** | 排产确认后 | 自动发布任务，无需人工干预 |

**界面改动**：

```
┌─────────────────────────────────────────────────────────────┐
│  生产排单                                                    │
├─────────────────────────────────────────────────────────────┤
│  ☑ 自动发布任务   [说明：当开启时，排产确认后自动推送任务]   │
│                                                             │
│  工单号：WB-2025-0501                                       │
│  工序：裁剪 → 编织 → 定型 → 包装                            │
│                                                    [发布任务]│
└─────────────────────────────────────────────────────────────┘
```

**数据流（自动模式）**：

```
排产确认（工序分配完成）
  → 检测「自动发布任务」开关状态
    → 若开启：自动调用 container_center.collect_report() 发布任务
    → 若关闭：等待人工点击「发布任务」按钮
      → 成功后通知「任务已发布至工人端」
```

**实现要点**：

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 排单界面增加 Toggle 开关 | 位置：生产排单界面顶部，与发布按钮同行 | P1 |
| 开关状态持久化 | 保存至主系统配置文件（`config.ini` 或 `window_config.json`），重启后记忆用户偏好 | P1 |
| 自动发布逻辑挂载 | 排产确认事件触发后，根据开关状态判断是否自动调用发布接口 | P1 |
| 发布结果反馈 | 自动发布后界面显示发布成功/失败提示（气泡或状态栏） | P1 |
| 日志记录 | 自动/手动发布均记录操作日志，便于追溯 | P1 |

**相关文件**（预估）：

| 文件 | 改动 |
|------|------|
| `views/order_view.py` 或 `views/schedule_view.py` | 增加 Toggle 控件 + 事件绑定 |
| `models/production_dao.py` 或 `services/production_service.py` | 增加 `publish_to_container()` 发布方法 |
| `config.ini` / `window_config.json` | 增加 `auto_publish_tasks: bool` 配置项 |
| `models/database.py` | 如需记录发布历史，扩充发布日志表 |

> **注意**：此功能仅涉及主系统（跟单3.1）自身 UI 改动，不依赖移动端改造，可独立开发和测试。

#### 1.3 数据同步闭环

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 报工后主动回执 | 移动端报工成功后，主动回调主系统更新任务池 | P0 |
| 主系统状态变更同步 | 主系统工序状态变更后同步到移动端任务池 | P0 |
| 订单数据定时拉取 | 移动端定时从主系统同步最新订单状态 | P1 |
| 冲突检测机制 | 同一工序被多方操作时的冲突检测与处理 | P1 |

#### 1.4 代码规范统一

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 消除所有裸 `except:` | 全局搜索并替换为 `except Exception as e: logger` | P0 |
| 统一日志管理 | 所有 `print()` 替换为 `logger` | P0 |
| 配置集中管理 | 硬编码阈值/路径收拢至 `config.py` | P1 |
| 统一异常类 | `core/exceptions.py` 补充业务异常定义 | P1 |

### 阶段二：增强模块集成（运维改进）

**目标**：为主系统补齐生产级运维能力（监控、熔断、审计、备份）。

#### 2.1 运维增强模块清单

| 模块 | 来源 | 用途 | 优先级 |
|------|------|------|--------|
| CircuitBreaker | mobile_api_ai 已有 | 防止外部 API 调用级联失败 | P1 |
| QueueManager | mobile_api_ai 已有 | 消息排队、延迟监控、死信队列 | P1 |
| HealthChecker | mobile_api_ai 已有 | 依赖组件健康检查（MySQL/Redis） | P1 |
| EnhancedBackup | mobile_api_ai 已有 | 增量备份、校验和、时间戳命名 | P1 |
| EnhancedAuditLogger | mobile_api_ai 已有 | 含 IP 追踪、请求 ID、数据快照 | P2 |
| ClockSync | mobile_api_ai 已有 | 主系统与云端时钟同步 | P2 |
| DataIntegrity | mobile_api_ai 已有 | 数据完整性校验（哈希/校验和） | P2 |
| DeploymentManager | mobile_api_ai 已有 | 配置版本管理、自动回滚 | P2 |

#### 2.2 审计日志增强

**现状**：主系统操作日志分散，无统一格式。

**改进方案**：
- 集成 `enhanced_audit_logger.py`
- 所有关键操作（订单创建/删除/修改、工序报工、库存变动）记录审计日志
- 日志格式含：时间戳、操作用户、操作类型、IP、数据变更前后快照
- 支持导出为 CSV/Excel 便于追溯

#### 2.3 备份策略优化

**现状**：全量备份，无增量。

**改进方案**：
- 集成 `enhanced_backup.py`
- 实现增量备份（仅备份变更部分）
- 备份文件命名规范化：`backup_{type}_{timestamp}.zip`
- 备份后自动校验完整性
- 保留策略：每日增量 + 每周全量 + 保留 30 天

### 阶段二补充：材料备料界面 - 发布用料需求按钮

**需求背景**：

当前材料备料界面仅展示备料清单，工人/仓管完成备料后，需要人工在移动端另行提交用料需求，或依赖系统定时检测库存不足才触发通知。缺乏在备料完成后直接在主系统一键发布用料需求到移动端的能力，导致用料需求传递滞后。

**改进方案**：

在材料备料界面增加「发布用料需求」按钮，仓管确认备料完成后可直接一键发布用料需求至移动端（容器池），触发采购或领料流程。

**界面改动**：

```
┌─────────────────────────────────────────────────────────────┐
│  材料备料                                                    │
├─────────────────────────────────────────────────────────────┤
│  工单：WB-2025-0501   工序：编织   负责人：张三             │
│                                                             │
│  ┌──────────────┬────────┬────────┬────────┬─────────────┐  │
│  │ 物料名称     │ 需求量  │ 已备量  │ 状态    │ 操作        │  │
│  ├──────────────┼────────┼────────┼────────┼─────────────┤  │
│  │ 不锈钢丝     │ 500kg  │ 300kg  │ 进行中  │ [勾选完成] │  │
│  │ 编织网       │ 200m   │ 200m   │ 已完成  │ ✓          │  │
│  └──────────────┴────────┴────────┴────────┴─────────────┘  │
│                                                             │
│                              [ 发布用料需求 ]  [ 刷新 ]     │
└─────────────────────────────────────────────────────────────┘
```

**数据流（改进后）**：

```
仓管点「发布用料需求」
  → 收集当前工单/工序所有「已勾选完成」的备料项
    → 调用 container_center 或 directly publish_to_container()
      → 发布用料需求任务至移动端（容器池）
        → 工人收到领料通知
          → 仓管收到发布成功确认
```

**实现要点**：

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 备料界面增加发布按钮 | 位置：备料清单底部，与刷新按钮同行 | P1 |
| 备料勾选完成功能 | 每行物料增加勾选框，支持批量勾选 | P1 |
| 发布逻辑挂载 | 点击后收集已勾选物料，调用发布接口 | P1 |
| 发布结果反馈 | 界面气泡提示「已发布 X 项用料需求」或失败原因 | P1 |
| 与自动发布任务开关联动 | 若「自动发布任务」开关开启，备料完成后自动触发发布 | P2 |

**相关文件**（预估）：

| 文件 | 改动 |
|------|------|
| `views/production_view.py` | 在「材料备料」跳转逻辑处新增发布按钮 |
| `views/inventory_view.py`（若独立） | 独立备料界面中增加按钮和勾选功能 |
| `services/production_service.py` | 增加 `publish_material_requirements()` 方法 |
| `models/production_dao.py` | 扩充备料记录状态字段（待备料/已备料/已发布） |
| `config.ini` / `window_config.json` | 增加 `auto_publish_material: bool` 配置项 |

> **注意**：此功能与「自动发布任务开关」（1.2节）属于同一类「一键发布」场景，可复用同一套发布接口，仅触发时机不同。备料发布侧重物料侧，工序发布侧重任务侧。

### 阶段三：云端融合（功能扩展）

### 阶段四：现代化（体验提升）

## 六、约束与前提

1. **Tkinter 保留**：主系统 GUI 保持 Tkinter，不进行大规模重写，避免引入过大风险
2. **向后兼容**：所有改造必须保持与现有数据库结构兼容，禁止破坏性变更
3. **移动端配合**：阶段一和阶段三需要移动报工系统（mobile_api_ai）同步改造，需协调推进
4. **测试覆盖**：每个 P0/P1 任务完成后必须验证主流程不受影响
5. **文档同步**：代码变更后同步更新 `docs/说明文档.md`
