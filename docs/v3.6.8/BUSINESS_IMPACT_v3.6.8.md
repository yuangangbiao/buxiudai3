# 业务影响报告 - v3.6.8 P0 修复（P0-K + P0-C）

## 1. 用户场景对比

> 改善前（痛点） → 改善后（价值）表格

| # | 用户角色 | 改善前（痛点） | 改善后（价值） |
|---|---------|---------------|---------------|
| 1 | **生产工人**（手机端报工）| 报工提交后偶尔出现"5003 没收到同步"，车间调度看不到，报工丢失要补录 | 报工后 5003 直连同步（< 10ms）；若断网，outbox 自动兜底重试（30s 内） |
| 2 | **车间班组长**（桌面端）| 任意登录用户都能删除/重置订单和工序，误操作易发生且无法追溯 | 删除/重置类操作需 admin 或 manager 角色，权限边界清晰 |
| 3 | **系统运维**（凌晨值班）| 5003 同步失败无人值守，业务数据永久漂移 | 失败 5 次触发微信告警（`notify_sync_failed`），值班可及时处理 |
| 4 | **质检员 / 质检主管**（桌面端）| 任何登录用户都能删除质检记录，质量追溯链断裂 | 删除质检记录需 admin 或 inspector，追溯链受保护 |
| 5 | **发货员 / 管理员**（桌面端）| 任何人都能删除发货记录，导致已发货物丢失发运凭证 | 删除发货记录需 admin，发运凭证受保护 |
| 6 | **人事管理员**（桌面端）| 任何人都能停用操作员账号，导致操作员被恶意踢出系统 | 停用操作员需 admin，人事变更受保护 |

## 2. 业务能力新增

> 按业务流分类

| 业务流 | 新增/优化功能 | 影响范围 |
|--------|--------------|---------|
| **报工** | outbox 兜底重试（30s 内自动重试 5 次） | 5008 报工 → 5003 桌面端同步全链路 |
| **跨库同步** | `bridge/dispatch_center_sync.py` 统一 outbox 入口（所有 send_* 调用方受益）| material / repair / outsource / quality / sub-step-report 5 个端点 |
| **死信告警** | P0-M 联调：5 次失败触发微信告警 | 全部跨库同步事件 |
| **角色权限** | 8 个高危接口加 RBAC 校验 | 删除/重置订单/操作员/物料/工序/质检/发货 |
| **权限分级** | 区分 admin / manager / inspector 三级权限 | 桌面端 5001 全站 |
| **CSRF 防护** | 8 个接口全部走 `@verify_csrf_token` | 桌面端跨站请求防护 |

## 3. 不变更部分

> 防回归保护清单

| # | 模块/功能 | 保护措施 | 验证方式 |
|---|----------|---------|---------|
| 1 | 报工实时性（直连路径） | `send()` 改造保留直连成功路径，outbox 仅作失败兜底 | mock 测试：直连成功时 outbox 不被调用 |
| 2 | 业务事务原子性 | `save_process_sub_step_with_pkg_update` 事务边界不变 | 既有单测覆盖（v3.6.5 重构单测）|
| 3 | 5003 调度中心协议 | `/api/dispatch-center/sync/*` 接口契约不变 | 既有 5003 单测覆盖 |
| 4 | 普通用户登录功能 | `require_auth` 装饰器不变 | 既有登录流程不变 |
| 5 | 只读接口（GET）| 未加角色校验（不必要）| L559/L775 等查询接口保持原样 |
| 6 | P0-A/B JWT Secret + dispatch_token | 上一轮已修复，本轮无变更 | 启动时强制环境变量检查 |
| 7 | P0-D/E/F/G/H/I/J/L/M 全部 P0 修复 | 上一轮已完成，本轮无变更 | 见 `docs/v3.6.8/ACCEPTANCE_v3.6.8.md` |

## 4. 一句话总结

> 本次改动让 5008 报工和 5001 桌面端的**跨库同步**与**删除/重置操作**从"任意登录用户可触发、失败后数据漂移"变为"分级授权保护、失败自动兜底 + 告警"，8 个高危接口的越权风险被彻底堵住。

## 附：变更影响范围

### 文件变更清单
1. `mobile_api_ai/bridge/dispatch_center_sync.py` — `send()` 函数加 outbox 兜底
2. `desktop_web/server.py` — 8 个路由加 `@require_role()` 装饰器

### 调用方影响（自动受益）
- `mobile_api_ai/app.py` 中 11 处 `send_sub_step_report / send_quality_record / send_material_update / send_repair_update / send_outsource_update` 调用
- 桌面端 8 个高危接口的调用方（前端 JS 调用方式不变，403 响应需前端处理）

### 部署注意
- **无需**新增数据库表（`sync_outbox` 表已存在）
- **无需**新增 worker 进程（`outbox_worker.py` 已在运行）
- 建议下次部署后用 worker 角色登录验证 403 响应
