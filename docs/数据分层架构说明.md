# 数据分层架构说明

## 📋 概述

本次优化实现了数据分层架构，确保扫码界面和企业微信机器人的数据来源于**中间容器池**，而不是直接访问主数据库。

---

## 🏗️ 架构图

```
┌─────────────────┐
│   桌面端系统   │
│  (主数据管理)  │
└────────┬────────┘
         │
         │ [收集任务]
         │
         ↓
┌─────────────────┐
│  中间容器池    │ ← 数据来源！
│(wechat_container.db)│
└────────┬────────┘
         │
         │ [获取数据]
         │
    ┌────┴────┐
    ↓         ↓
┌─────────┐ ┌─────────┐
│ 网页端 │  │ 企微机器人│
│ (手机/PC) │ └─────────┘
└─────────┘
```

---

## 📝 优化内容

### 1. api/scan.py 优化

**优化前：**
- 直接连接并查询主数据库（MySQL）
- 依赖 models.database

**优化后：**
- 从容器中心获取数据
- 新增函数：
  - `get_container_center()`: 获取容器中心实例
  - `parse_qr_data()`: 解析二维码数据
  - `find_task_in_container()`: 在容器中查找任务
  - `format_task_data()`: 格式化任务数据
  - `create_sample_task()`: 测试接口，创建示例任务

---

### 2. wechat_work_bot_v2.py 优化

**优化前：**
- 扫码功能直接查询主数据库
- 依赖 models.database

**优化后：**
- 从容器中心查找任务
- 如果任务还没分配，自动分配给当前操作员
- 不再依赖主数据库连接

---

### 3. 新增测试功能

**API 新增：**
- `POST /api/scan/test/create-sample`: 创建示例任务到容器

**测试脚本：**
- `quick_test.py`: 完整测试流程，从创建测试数据到扫码、确认、报工

---

## 📊 数据流向

### 桌面端 → 容器池（数据收集）

```
桌面端任务创建
    ↓
container_center.collect_report(...)
    ↓
数据存入容器池 (wechat_container.db)
```

### 网页/手机端 → 容器池（数据获取）

```
用户扫码输入工单
    ↓
/api/scan/task 接口
    ↓
find_task_in_container() 查找
    ↓
返回容器中的任务数据
```

## 💡 使用提示

### 桌面端需要先做的

1. 在任务创建时调用 `container_center.collect_report()`
2. 确保 `content` 字段包含必要信息：
   - `order_no`: 工单号
   - `order_no`: 订单号
   - `customer_name`: 客户名
   - `product_type`: 产品类型
   - `quantity`: 数量
   - `unit`: 单位
   - `process_name`: 工序名
   - `planned_qty`: 计划数量
   - `voice_text`: 语音播报文本

### 容器数据格式示例

```json
{
  "id": "xxx",
  "title": "报工任务-ORD202604001-编织",
  "status": "pending",
  "content": {
    "order_no": "WO202604001",
    "order_no": "ORD202604001",
    "customer_name": "上海机械厂",
    "product_type": "不锈钢编织网",
    "quantity": 100,
    "unit": "米",
    "process_name": "编织",
    "planned_qty": 100,
    "voice_text": "订单ORD202604001，工序编织，数量100..."
  },
  "related_order": "ORD202604001"
}
```
