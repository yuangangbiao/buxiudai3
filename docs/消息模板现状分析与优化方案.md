## 二、问题诊断

### 2.1 重复定义（最严重问题）

同一消息类型在 **3～4 个不同文件** 中各自硬编码，且格式不同：

| 消息类型 | dispatch_center.py 模板 | services/notifier.py | integration/wechat_notifier.py | schedule_flow.py |
|---------|----------------------|---------------------|------------------------------|-------------------|
| 任务分配 | `tmpl_task_assigned` | `notify_task_assigned()` (140行) | `notify_task_assigned()` (96行) | — |
| 任务完成 | `tmpl_task_completed` | `notify_task_completed()` (185行) | `notify_task_completed()` (124行) | — |
| 库存预警 | `tmpl_material_lowstock` | `notify_low_stock()` (230行) | — | — |
| 报工提交 | — | `notify_report_submitted()` (311行) | `notify_report_success()` (217行) | — |
| 排产通知 | `tmpl_schedule_notify` | — | — | `schedule_flow.py` (521行) |
| 工单发布 | — | — | — | `schedule_flow.py` (375行) |
| 排产确认 | `tmpl_schedule_complete` | — | — | `schedule_flow.py` (729行) |
| 排产拒绝 | — | — | — | `schedule_flow.py` (742行) |
| 新任务广播 | — | — | — | `send_broadcast_notification.py` (102行) |

**重复率**：8 种消息类型中有 **6 种存在冗余定义**。

---

### 2.2 格式不一致

三套消息格式比较（以"任务分配"为例）：

| 对比项 | dispatch_center.py | services/notifier.py | integration/wechat_notifier.py |
|--------|-------------------|---------------------|-------------------------------|
| 标题 | `📋 新任务通知` | `📢 任务分配通知` | `(无标题)` |
| 分隔线 | `━━━━━━━━` | 无 | `━━━━━━━━` |
| 字段样式 | `**订单号**：{value}` | `**订单号**：{value}` | `任务ID: {value}` |
| 尾部 | `请及时确认处理！` | `请及时确认！` | `发送"确认"或"确认 {id}"` |
| Markdown类型 | markdown | text | text |

同样的问题出现在所有重复的消息类型中。用户在不同场景收到同一类通知时，格式完全不同，**体验割裂**。

---

### 2.3 硬编码 vs 模板化

| 对比维度 | 模板引擎方式 (dispatch_center.py) | 硬编码方式 (其他 10 个文件) |
|---------|--------------------------------|--------------------------|
| 修改消息内容 | 改模板数据（JSON/API） | 改代码 → 重启 |
| 运营人员可配 | ✅ 通过 API | ❌ 必须开发介入 |
| 多语言支持 | 框架已有（中英文变量双向映射） | ❌ 不支持 |
| 变量校验 | 有变量列表定义 | 无 |
| HTML/Markdown混用 | ❌ | ✅ HTML标签在微信中可能解析异常 |
| 模板版本管理 | 内存缓存 + JSON持久化 | 无 |

核心问题：dispatch_center.py 已经有了一套模板系统，但 **services/notifier.py** 完全绕过了它，自己重复构建了一套硬编码方案。

---

### 2.4 废弃代码未清理

- `mobile_api_ai/integration/wechat_notifier.py` 明确标注 `@deprecated`（第 6 行）
- 但 `container_center_v5.py` 和 `timeout_reminder.py` 仍然引用它
- 该文件底部还创建了全局单例 `wechat_notifier = WeChatNotifier()`（第 326 行）
- `TaskPollingNotifier` 类依然在等待被调用

**结论**：标记了 deprecated 但从未真正下线。

---

### 2.5 模板引擎自身缺陷

1. **无模板语法**：仅支持 `{var}` 字符串替换，无条件判断、循环、过滤器
2. **无模板继承**：每条模板完全独立，修改公共结构需逐个改 30 条
3. **无版本管理**：修改模板后无回滚能力
4. **存储方式**：模板存在 JSON 文件 + 内存缓存中，非数据库，多实例间不同步
5. **变量对照表散落在代码里**（`_resolve_variables()` 和 `get_template_variables()` 各维护一份）
6. **成本核算模板中使用 Python 格式化语法**（`¥{总成本:.2f}`），与其他模板的纯字符串替换不一致

## 三、优化方案

### 3.1 总体目标

> **一个引擎、一份模板、一条链路**

将所有消息发送统一到调度中心的模板引擎，消除所有硬编码消息。

---

### 3.2 优化步骤

#### Phase 1：清理废弃代码（投入：0.5 人天）

1. **删除** `mobile_api_ai/integration/wechat_notifier.py`
2. 改造 `container_center_v5.py` 和 `timeout_reminder.py` 中对旧 `WeChatNotifier` 的引用，改为调用 `services/notifier.py`
3. 删除 `scripts/cloud/send_broadcast.py` 和 `send_broadcast_notification.py`（移到 `_archive/`）

#### Phase 2：统一模板接入（投入：2 人天）

1. **改造 `services/notifier.py`**：
   - 每个 `notify_*` 方法不再硬编码 f-string，改为调用 `_render_template(template_id, variables)`
   - 将现有的 5 种消息格式沉淀为调度中心模板，补全缺失的模板 ID（如报工提交 `tmpl_report_submitted`）
   - 所有消息通过 `send_message()` API 统一发送

2. **改造 `scripts/cloud/schedule_flow.py`**：
   - 删除 6 处硬编码消息
   - 改为调用调度中心 `/api/dispatch-center/messages/send` 接口
   - 新工单发布 → 使用 `tmpl_process_start`
   - 排产通知 → 使用 `tmpl_schedule_notify`
   - 排产确认 → 使用 `tmpl_schedule_complete`
   - 排产拒绝 → 新增 `tmpl_schedule_reject` 模板

3. **改造 `alert.py`**：
   - 微信企业号渠道改为调用 `send_message()` API
   - 钉钉/邮件保留独立实现（因为它们不走微信通道）

#### Phase 3：模板引擎增强（投入：1.5 人天）

1. **引入 Jinja2 轻量模板引擎**替代 `{var}` 字符串替换：
   - 支持 `{% if %}`, `{% for %}`, `{{ var | filter }}`
   - 成本核算模板中的 Python 格式化语法可以统一
2. **增加模板预览/测试 API**：`POST /messages/templates/<id>/preview`
3. **增加模板版本历史**：每次 PUT 更新保留前一版本
4. **将 VAR_EN_TO_CN / VAR_CN_TO_EN 映射表** 收敛到一个配置文件 `configs/template_variables.json`

#### Phase 4：配置化与可观测（投入：1 人天）

1. **模板存储迁移到 MySQL**，多实例共享（不再依赖 JSON 文件同步）
2. **增加发送日志持久化**：每次 send_message 调用写入 `message_send_log` 表
3. **增加消息发送监控看板**：
   - 今日发送量
   - 发送成功率
   - 按渠道/模板/类型的分布

## 四、总结

当前系统的消息模板体系处于 **"有引擎但没人用"** 的状态。调度中心 dispatch_center.py 已经实现了 30 个模板 + CRUD API + 变量体系，但核心业务代码（notifier.py、schedule_flow.py、alert.py）全部绕过了它，自己硬编码了一套。这导致了：

- **8 种消息类型有 3～4 套不同实现**
- **用户收到的同类通知格式完全不统一**
- **修改消息内容必须改代码重启**

优化方向明确：**一刀切 — 所有消息发送都走 dispatch_center.py 的统一发送接口 `POST /messages/send`**，模板缺失就补模板，变量缺失就补变量。整个改造控制在 5 人天内，风险可控。
