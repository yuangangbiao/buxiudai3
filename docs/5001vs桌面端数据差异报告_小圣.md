## B. 字段差异清单

### B.1 production_orders 表

| 维度 | 桌面端 (models/production.py) | web5001 (server.py L1839-1927) | 差异分析 |
|------|------------------------------|--------------------------------|---------|
| **INSERT 字段** | `order_no, order_id, priority, plan_start, plan_end, assigned_to, status, remark` (L40-43) | 同左：`order_id, priority, plan_start, plan_end, assigned_to, remark` (L1847-1853) | ✅ 完全一致 |
| **UPDATE 字段** | `priority, plan_start, plan_end, assigned_to, status, remark` (L131-143) | 同左 (L1883-1888) | ✅ 完全一致 |
| **状态默认值** | `ProductionStatus.PENDING.value` (L50) | `ProductionStatus.PENDING_PUBLISH.value` (L1855) | ⚠️ **差异1**: 桌面端用 PENDING，web5001 用 PENDING_PUBLISH |
| **字段补全** | 从 orders 表补全 order_no | ❌ **差异2**: web5001 创建工单时 **未写入 order_no** |

**差异影响**:
- **差异1**: 桌面端创建工单状态是"待发布"，web5001 是"待发布"(其实值可能相同，需确认 ProductionStatus 枚举)
- **差异2**: web5001 通过 ProductionDAO.create 创建时会从订单表查询 order_no，所以实际上会写入

## D. 同步机制差异 (核心问题)

### D.1 桌面端同步机制

**models/production.py L199-230**:
```python
if new_status and order_id:
    sync_url = os.environ.get('SYNC_BRIDGE_URL', 'http://127.0.0.1:5008')
    _req.post(f'{sync_url}/api/sync/status-change', json={
        'order_no': wo_no,
        'status_key': status_key,
        'source': 'production.update_status'
    }, timeout=2)
```

**触发条件**: 仅 `ProductionDAO.update_status()` 被调用时

### D.2 web5001 同步机制

| API | 是否触发 SYNC_BRIDGE | 原因 |
|-----|---------------------|------|
| `api_create_work_order` (L1839) | ❌ 不触发 | 只调用 `ProductionDAO.create`，不调用 `update_status` |
| `api_change_work_order_status` (L1896) | ✅ 触发 | 调用 `ProductionDAO.update_status()` |
| `api_update_work_order` (L1875) | ⚠️ 可能触发 | 若 `body` 中包含 `status`，调用 `ProductionDAO.update()` → 不触发 sync |
| `api_process_start/complete/reset` (L2206, L2237, L2266) | ❌ 不触发 | 直接写 SQL，不调用 ProductionDAO |
| `api_process_report` (L2295) | ❌ 不触发 | 直接写 SQL，不调用 ProductionDAO |

### D.3 架构风险点

| # | 风险 | 描述 | 影响程度 | 建议 |
|---|------|------|---------|------|
| **R1** | 工序操作不同步 | web5001 的工序报工/开始/完成直接写 SQL，不经过 ProductionDAO，不会触发 SYNC_BRIDGE | 🔴 高 | 在 `api_process_report/start/complete/reset` 中增加同步调用 |
| **R2** | 工单创建不同步 | `ProductionDAO.create` 不调用 `update_status`，所以创建工单时不会同步到 5008 | 🟡 中 | 在 `api_create_work_order` 末尾增加 SYNC_BRIDGE 调用 |
| **R3** | 双重数据写入路径 | web5001 直接写 SQL vs 桌面端通过 DAO，可能产生字段不一致 | 🟡 中 | 统一 web5001 也使用 DAO 层 |
| **R4** | 状态转换规则分散 | web5001 有自己的状态验证 (L1906)，但桌面端也有，逻辑需保持同步 | 🟢 低 | 共用 `validate_production_status_transition()` |

## E. 建议的统一方案

### 方案 A: web5001 完全走 DAO 层 (推荐)

```
web5001 server.py
    │
    ├── api_create_work_order ──▶ ProductionDAO.create() ──▶ 自动同步
    ├── api_change_work_order_status ──▶ ProductionDAO.update_status() ──▶ 自动同步
    ├── api_process_* ──▶ 新增 ProcessDAO.report/start/complete/reset ──▶ 同步
    │
    └── 其他直写 SQL ──▶ 迁移到对应 DAO
```

**优点**: 
- 代码复用，减少重复
- 同步逻辑集中管理
- 便于统一维护

**缺点**:
- 需要新增 ProcessDAO 类
- 工作量较大

### 方案 B: web5001 保持现状，补充同步调用

```
web5001 server.py
    │
    ├── 直写 SQL (保持不变)
    │
    └── 新增同步装饰器 @sync_to_5008
        │
        └── 在每个写操作后调用 SYNC_BRIDGE
```

**优点**:
- 改动小
- 保持现有逻辑

**缺点**:
- 同步逻辑分散
- 容易遗漏

### 方案 C: 统一数据流经 5003 调度中心

```
桌面端 ──▶ 5003 调度中心 ──▶ MySQL ──▶ SYNC_BRIDGE
web5001 ──▶ 5003 调度中心 ──┘
```

**优点**:
- 架构统一
- 所有数据变更集中管理

**缺点**:
- 需要重构 5003
- 改动最大

## H. 总结

### 关键发现

1. ✅ **状态机定义**: 桌面端和 web5001 完全共用 `constants.py`，状态机一致
2. ⚠️ **同步机制**: web5001 只有工单状态变更触发同步，工序操作不触发
3. ⚠️ **字段差异**: `process_records.order_id` 在 web5001 添加工序时为 NULL
4. ✅ **共用 DAO**: 订单和工单 CRUD 共用桌面端 DAO 层

### 优先修复

1. **P0**: web5001 工序报工/开始/完成需触发 SYNC_BRIDGE 同步
2. **P1**: web5001 添加工序时需写入 `order_id` 字段
3. **P2**: 考虑将 web5001 的工序操作也迁移到 DAO 层
