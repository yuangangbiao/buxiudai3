## 二、会议讨论与争议解决

### 争议 1：P0 数量——11 项还是更多？

| 立场 | 观点 |
|------|------|
| **小贺** | 应加入物料新增即锁定（P1→升P0）；理由：用户无法编辑新添加的物料 |
| **小圣** | 物料锁定是 UX 问题，不是数据安全问题；维持 P1 |
| **小曦** | 同意小圣；但补充：若工厂高峰期（每月20号）出现此问题，影响面大 |
| **小钰** | 安全角度：不影响鉴权，维持 P1 |

**会议决议**：

- 物料新增即锁定：**🟠 P1**，但增加"**高峰期事故预案**"（临时解锁按钮）
- 升 P0 的原则：**影响数据安全 / 影响生产核心流程 / 可导致数据永久丢失**才升

---

### 争议 2：P0-I 跨库同步——outbox 还是暂缓？

| 立场 | 理由 |
|------|------|
| **小圣** | 必须 outbox；工厂断网是常态，否则数据永久丢失 |
| **小曦** | 工厂实际：断网时工人不报工（手机离线），不是"报完工但同步失败" |
| **小贺** | 从日志看：5008 报工是工人手机端操作，断网时工人不会提交 |

**会议决议**：

- 短期（本周）：**加 service token + 告警表**（2h）；outbox 列为 🟠 P1
- 工厂实际场景：断网时工人无法提交，5008 不存在"报完工但同步失败"场景
- 5001（桌面端）才是真正的同步风险点

---

### 争议 3：P0-4 SYNC_BRIDGE——修还是不修？

| 立场 | 理由 |
|------|------|
| **小圣** | 必须修；5003 是调度核心 |
| **小曦** | 工厂实际：5003 调度功能用得少；先修 5008 |
| **小贺** | 5003 每天只用 2-3 次；不是生产阻塞点 |

**会议决议**：

- SYNC_BRIDGE_URL 降为 **🟡 P2**（不影响生产）
- 优先修：5008 的 RE-001 事务一致性（影响数据正确性）
- 5003 的 service token 列入 🟠 P1

---

### 争议 4：测试框架——先修测试还是先修主应用？

| 立场 | 理由 |
|------|------|
| **小圣** | 测试框架 A1 循环导入（30min）优先级高于部分 P0 |
| **小贺** | 主应用 P0 直接影响生产；测试框架修不修不影响今天 |
| **小曦** | 测试框架修好才能跑 CI；CI 修好才能验证 P0 |

**会议决议**：

- 测试框架 A1（conftest 循环导入）：**🔥 立即修**（30min，不阻塞主应用）
- 主应用 P0 和测试框架并行推进（两组人）
- CI 可执行（A2）列入 **🟠 P1**
