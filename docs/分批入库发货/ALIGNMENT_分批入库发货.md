# ALIGNMENT - 分批入库/发货

## 原始需求
主软件（桌面端 Tkinter）增加分批入库和分批发货功能，按数量分别完成步骤，联动调度中心展示子步骤。

## 需求理解

### 当前状态
- **调度中心流程步骤：** `完工入库` → `发货` 为二进制步骤（一步到位）
- **主软件（桌面端）：** ShipmentView 管理发货单和成品库存，FinishedProductStatsView 统计成品
- **成品库存：** 按订单显示总库存量（无分批概念）
- **发货单：** 一次发货一条记录（无分批概念）

### 目标状态
- **分批入库：** 仓库员可多次入库，每次录入部分数量，累计达到订单总量后步骤自动完成
- **分批发货：** 销售/仓库可多次发货，每次录入部分数量，累计达到订单总量后步骤自动完成
- **子步骤同步：** 每批操作在调度中心生成一条子步骤记录，可展开查看明细
- **全局同步：** 成品统计、发货管理、订单详情等所有相关界面同步显示累计量和剩余量

### 边界确认
| 项目 | 包含 | 不包含 |
|------|------|--------|
| 主软件（桌面端） | 分批入库UI、分批发货UI、进度展示 | 不修改现有核心业务逻辑 |
| 调度中心（Web） | 子步骤列表展示、进度条 | 不修改流程推进按钮逻辑 |
| 后端API | 子步骤CRUD、累计量查询 | 不重构现有API架构 |
| 数据库 | 新增 process_sub_steps 表 | 不修改现有表结构（仅扩展） |

### 流程图示

```
订单 1000米
├── 生产 → 质检 → 报工完成
├── 完工入库 (累计入库: 700/1000)
│   ├── 批次#1  300米  [已完成]
│   ├── 批次#2  400米  [已完成]
│   └── ... (待入 300米)
└── 发货 (累计发货: 400/1000)
    ├── 批次#1  400米  [已完成]
    └── ... (待发 600米)
```

## 决策记录

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 联动方式 | 联动调度中心 | 用户明确要求每批操作在调度中心可见 |
| 数据展示 | 全局同步 | 所有相关视图显示累计量和剩余量 |
| 实现模式 | 子步骤模式 | 新增 process_sub_steps 表，每批独立记录 |
| 主步骤完成条件 | 累计量 >= 订单数量时自动完成 | 无需人工干预 |
