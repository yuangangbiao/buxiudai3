# 跨层共通模式分析

> **目的**：识别主应用审计与测试框架审计中的**共通架构反模式**
> **分析日期**：2026-06-24
> **被分析文档**：
> - `架构问题临时文档_20260624.md`（测试框架 10 项 A1-A10）
> - `专家团队共识报告_v1.md`（主应用 29 项 P0/P1/P2）
> - `漏洞探查报告_4专家_20260623.md`（主应用 39 个漏洞）

---

## 一、发现 5 个跨层共通反模式

### 🔁 反模式 1：循环依赖 / 紧耦合

| 层级 | 位置 | 表现 |
|------|------|------|
| **测试框架** | `conftest.py:23` ↔ `health.py:9` | pytest 启动即崩 |
| **主应用** | `dispatch_center` ↔ `container_center` | 跨服务调用无服务发现 |
| **主应用** | `models/inventory.py` ↔ `steel_belt` | 旧库/新库混用 |

**根本原因**：快速迭代时"先跑起来再说"，模块边界模糊。

---

### 🔒 反模式 2：并发安全缺失 / 竞态条件

| 层级 | 位置 | 表现 | 证据 |
|------|------|------|------|
| **测试框架** | `conftest.py:155-239` | pytest -n 2 时 2 个 worker 互相 DELETE 对方数据 | A5 + A10 |
| **主应用** | `server.py:1586/1732/1831` | 撤回操作无 FOR UPDATE，并发竞态 | P0-J |
| **主应用** | `server.py:1200` | 物料编辑无锁，与 add-qty 不一致 | P1-D |
| **主应用** | `mysql_storage.py:1193` | `package_exists` 漏 key，重复派发 | P1-E |

**根本原因**：单线程写时没问题，上并发就崩；测试没覆盖并发场景。

---

### 📊 反模式 3：一致性校验缺失 / 盲目信任

| 层级 | 位置 | 表现 | 证据 |
|------|------|------|------|
| **测试框架** | `ai_e2e.yml` 全文 | CI 配置了但全是无效步骤（文件不存在）| A2 |
| **主应用** | `models/order.py:316` | 订单状态机零校验，任意跳级 | P0-E |
| **主应用** | `server.py:2415-2418` | reset 后 completed_qty 不清零 | P1-B |
| **主应用** | `models/order.py:370` | 软删除假实现（is_deleted 从未更新）| P1-C |
| **主应用** | `server.py:2421` | 报工数量无上限，可负数 | P1-F |
| **主应用** | `routes_core.py` 等多处 | 审计 history 不完整（5008 四处缺 INSERT）| P1-G |

**根本原因**：相信"数据是对的"而不验证输入；修复时只管 happy path。

---

### 🔌 反模式 4：跨系统集成无原子保证

| 层级 | 位置 | 表现 |
|------|------|------|
| **测试框架** | 4 个 conftest 分散 | 加载顺序不确定，fixture 冲突 |
| **主应用** | `server.py:2454` | 报工先 commit 再同步 5008，失败不回滚 |
| **主应用** | `sync_bridge.py:299` | outbox 死信无告警 |
| **主应用** | `5001` ↔ `5003` | 数据库分离，跨端 ID 不互通 |

**根本原因**：分布式系统"先干活再说"，同步失败无降级策略。

---

### 🏗️ 反模式 5：重复造轮子 / 架构碎片化

| 层级 | 位置 | 表现 |
|------|------|------|
| **测试框架** | A3 + A4 | 与已有 e2e/unit 测试重复，边界不清 |
| **测试框架** | A9 | 自写报告系统与 pytest-html 功能重叠 70% |
| **测试框架** | A8 | 自写 4 层 fixture vs 官方 pytest-playwright plugin |
| **主应用** | 分散审计 | 5008 审计日志分散各处 vs 5010 统一 log_operation |
| **主应用** | 多套缓存 TTL | 7 套散落不一致（P3-B）|

**根本原因**：没先审计"已有的"，直接起新炉灶。

---

## 二、根因分析

```
快速迭代压力
    ↓
先跑起来 / 先修 Bug / 先加功能
    ↓
忽略：架构边界 / 并发安全 / 输入校验 / 跨系统一致性
    ↓
上线后遇到问题
    ↓
修一个问题引入两个新问题（A2 CI 修一次发现 7 个不存在）
    ↓
审计时发现系统性反模式
```

## 三、跨层共通修复原则

基于两套审计的共通反模式，总结出 **5 条跨层通用修复原则**：

| # | 原则 | 主应用示例 | 测试框架示例 |
|---|------|-----------|------------|
| **1** | **先审计再动手**：修 Bug 前先 grep 确认"已有的"在哪里 | 修 dispatch_token 前先确认 5003 的格式 | 建 L2 测试前先审计 unit/e2e 覆盖了什么 |
| **2** | **并发必须测**：所有写操作上线前用 pytest -n 4 跑过 | 撤回/编辑操作加 FOR UPDATE 后跑并发测试 | conftest fixture 加 scope='worker' |
| **3** | **输入必校验**：所有外部输入（用户/DB/跨服务）必须验证后再用 | 报工数量校验 > 0 <= 计划数 | fixture 参数类型校验 |
| **4** | **跨系统有退路**：跨库/跨服务调用必须有 fallback 或 outbox | 5008→5003 加 service token + outbox | 多 conftest 加 load_order 注解 |
| **5** | **变更有审计**：所有数据变更必须能追溯（history 表 / 审计日志）| 修 P1-G 给 5008 四处补 INSERT data_regression_history | — |

## 四、验证检查清单（下次审计必跑）

主应用审计 + 测试框架审计时，共同检查：

```
□ grep -rn "FOR UPDATE" <target>     # 无锁 = 并发风险
□ grep -rn "rowcount" <target>       # 无 rowcount = 假成功
□ grep -rn "import random" <target>  # 缺 import = 运行时崩
□ grep -rn "conn.commit()" <target>  # commit 后才同步 = 跨库不一致
□ grep -rn "is_deleted" <target>     # 假软删除
□ ls tests/conftest.py tests/e2e/conftest.py  # 多 conftest = 冲突风险
□ grep "pytest --" .github/workflows/  # CI 步骤文件存在性
□ grep -rn "soft_delete\|restore" <target>  # 无 history = 无审计轨迹
```

## 五、结论

测试框架审计（10 项 A 系列）和主应用审计（29 项 P 系列）**揭示了同一个根因**：快速迭代时跳过了架构纪律。

| 反模式 | 主应用出现次数 | 测试框架出现次数 | 合计 |
|--------|:------------:|:---------------:|:----:|
| 循环依赖/紧耦合 | 3 | 1 | 4 |
| 并发安全缺失 | 4 | 2 | 6 |
| 校验缺失 | 6 | 1 | 7 |
| 跨系统无原子 | 3 | 1 | 4 |
| 重复造轮子 | 2 | 3 | 5 |
| **合计** | **18** | **8** | **26** |
