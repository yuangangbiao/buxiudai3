## 二、P0-1 修复验证（实测完成）

### 问题 4 原结论（已修正）

**原审计声称**：
> `/api/dispatch-center/production/list` **不存在**
> `/api/dispatch-center/material/list` **不存在**

**实测结果**：❌ **误判，端点实际存在**。

进一步检查发现真正 bug 在 `_call_dispatch`：

```python
# 修复前 - query string 被忽略
qs = request.query_string.decode('utf-8')  # "status=all&limit=3"
r = requests.get(url, headers=headers, params=kwargs.get('params'), timeout=10)
# params=None，requests 收到 url=".../production/list"（无参数）

# 修复后
if r_params is None and '?' in url:
    r_url = url.split('?', 1)[0]
    r_params = url.split('?', 1)[1]
r = requests.get(r_url, headers=headers, params=r_params, timeout=10)
```

**修复结果**：影响全部 9 个含 query string 的 GET 代理端点：
- `/api/production/list` ✅ 参数正确传递
- `/api/material/list` ✅
- `/api/quality/list` ✅
- `/api/shipment/list` ✅
- `/api/shipment/pending` ✅
- `/api/shipment/tracking-list` ✅
- `/api/shipment/finished-goods` ✅
- `/api/report-queue/list` ✅
- `/api/orders/query` ✅

**问题 4 结论修正**：
- 端点路径正确 ✅
- bug 原因：`_call_dispatch` 不处理 URL 内嵌 query string
- 已修复 ✅

## 三、🔴 严重问题（仍存在）

### 问题 1：生产排单数据写入路径冲突 ⚠️

桌面端 `production_view.py` 用 `ProductionDAO` **直接写库**，而 5003 的 `/production/list` 端点用 `core.db_compat.get_conn()` 直连同一库。

**当前架构**：

```
desktop_web (server.py)
    └── GET  /api/production/list  → 5003 dispatch_center_bp
                                    └── core.db_compat.get_conn() → MySQL steel_belt
    └── POST /api/production/orders → ?（方案未规划）

desktop (Tkinter)
    └── ProductionDAO → pymysql → MySQL steel_belt（直接写）

5003 dispatch_center
    └── schedule_bp → pymysql → MySQL（独立读写）
```

**方案中的写入路径未规划**：生产排单的 CRUD（创建工单、修改状态）走哪个路径？
- 走 5003 → 需要 5003 实现对应端点（当前 `/api/production/orders` POST 不存在）
- 走 `ProductionDAO` 直写 → 绕过了 5003，状态不同步

**建议**：生产排单写入走 `ProductionDAO` 直写（保持与桌面端一致），发布任务走 5003 `schedule_bp`。

**风险等级**：🟠 中 — 需要明确数据写入路径
