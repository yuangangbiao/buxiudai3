# 调度中心功能清单(按业务归属严格分类)

> **版本**: v2.0(归属修订版)
> **修订日期**: 2026-06-20
> **数据来源**: 基于 `mobile_api_ai/` 实际代码扫描(Grep + Read + 蓝图注册处逐行确认)
> **核心改动**: v1.0 把"注册到 5003 的 Flask app 上"等同于"调度中心的功能",**这是错误归类**。v2.0 严格按业务归属拆分为 §A/§B/§C 三段。
> **适用读者**: 维护者 / 新成员 / 架构师 / 稳定性加固负责人

---

## 0. 修订说明(为什么改)

### 0.1 v1.0 的错误

v1.0 把 `standalone_dispatch_server.py` 注册的所有蓝图/路由都计为"调度中心功能"。但仔细审查 `standalone_dispatch_server.py:118~160` 的注册注释后,发现:

```python
# 行 137: 同步桥蓝图(云端去除调度中心功能, TASK-8) — 16 端点本地 5003 入口
# 行 145: 企业微信机器人蓝图(群hook、应用回调、消息代理)
# 行 154: 系统配置蓝图
# 行 333: 注册报工/外协记录管理路由(app.py 中 @app.route,调度中心也需用)
```

这三类注释明确表明:**它们不属于调度中心**,只是被"寄生"在 5003 这个常驻 Flask 服务上。

### 0.2 v2.0 的归属三段式

| 段 | 名称 | 占比 | 含义 |
|----|------|------|------|
| **§A** | **真正属于调度中心** | **~70%** | 业务定位是"派单/流程/告警/模板",目录在 `dispatch_center/` |
| **§B** | **寄生在 5003 的其他系统** | **~20%** | 同步桥 / 企业微信 / 系统配置 / 报工记录 / 登录认证 |
| **§C** | **5003 调用的依赖库(非端点)** | **~10%** | `bots/*` / `MessageHub` / `Dispatcher` 等 |

### 0.3 业务影响

- **稳定性加固**(`docs/调度中心稳定性加固.md`):真正要加固的是 §A 的 ~148 端点,不是 v1.0 虚高的 205 端点
- **微服务拆分**:§B 的 ~56 端点应该拆出去,5003 只保留调度核心
- **业务影响报告**:不能把 §B 的功能算到"调度中心改进了什么"

## 2. 归属判断标准

| 维度 | §A 调度核心 | §B 寄生业务 | §C 依赖库 |
|------|-------------|-------------|-----------|
| **目录** | `dispatch_center/*.py` | `mobile_api_ai/*.py`(顶层) | `bots/` / `container/` / `bridge/` |
| **业务** | 派单/流程/告警/模板/排产/发货 | 同步/企业微信/系统配置/报工记录 | Bot/派发器/同步客户端 |
| **代码注释** | 业务定义即"调度中心" | 注释明确"非调度"(如 TASK-8、复制自 app.py) | 不是 5003 端点,只是被调用 |
| **拆分必要性** | 必须保留在 5003 | 应该拆出去 | 跟随 §A 或单独库 |

#### A.8.1 `production` 生产流程(7 步)

```

生产执行(生产部) → 报工完成(生产部) → 质检审核(质检部) → 完工入库(仓库)
```

#### A.8.2 `material_purchase` 物料流程(6 步)

```
物料需求(生产部) → 库存检查(系统) → 采购审批(主管) →
采购下单(采购部) → 到货入库(仓库) → 领料出库(生产部)
```

#### A.8.3 `quality` 质检流程(4 步)

```
接收质检任务(质检部) → 检测结果判断(质检部) → 审核放行(质检部) → 入库(仓库)
```

#### A.8.4 `repair` 维修流程(5 步)

```

```

#### A.8.5 `outsource` 外协流程(7 步)

```
外协发单(计划部) → 外协确认(外协厂) → 外协生产(外协厂) →
外协质检(质检部) → 外协回厂(仓库) → 质检审核(质检部) → 入库(仓库)
```

## 9. 业务影响(用于稳定性加固与微服务拆分)

### 9.1 稳定性加固范围

**正确范围**:§A 的 ~148 端点 + 4 个调度蓝图 + 5 类流程模板。

**不包括**:
- §B 的 56 端点(同步桥/企业微信/系统配置/报工/外协/质检/物料记录管理)— 它们各自有独立的稳定性需求,不应混入调度中心加固
- §C 的依赖库 — 它们是被调用方,稳定性由其自身决定

### 9.2 微服务拆分建议

| 服务 | 应包含 | 端点数 |
|------|--------|--------|
| **dispatch-center:5003**(精简后) | §A 全部 | ~148 |
| **sync-bridge:5004**(独立) | §B.1 同步桥 | 21 |
| **wechat-bot:5005**(独立) | §B.2 企业微信 + §C.1 Bots | 6 + §C |
| **config-center:5006**(独立) | §B.3 系统配置 | 15 |
| **main-service:5000**(原 app.py) | §B.4 报工/外协/质检/物料记录管理 + §B.4.1/2/3 | 14 |

**拆分后 5003 端口负载降低 ~28%**(=56/204)。
