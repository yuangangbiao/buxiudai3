## 一、问题背景

### 1.1 用户痛点

手机报工应用中，排产列表和任务列表加载速度慢，影响用户体验：
- 首次打开页面需要等待 3-5 秒
- 切换筛选条件时页面卡顿
- 数据量大时白屏时间过长

### 1.2 影响范围

| 页面/功能 | 接口 | 影响用户 |
|-----------|------|----------|
| 排产列表 | `/api/schedule/list` | 生产主管、调度员 |
| 任务列表 | `/api/task/list` | 一线操作员 |
| 工序详情 | `/api/process/detail` | 一线操作员 |

### 3.2 详细方案

#### P1: 算法优化（核心，收益最高）

**P1.1 哈希分组替代嵌套循环**

```python
# 当前：O(n*m) 嵌套循环
for p in processes:           # n=1000
    for pkg in packages:     # m=2000
        if pkg.get('related_order') == p['order_no']:
            ...

# 优化后：O(n+m) 哈希分组
pkg_by_order = defaultdict(list)
for pkg in packages:
    pkg_by_order[pkg.get('related_order', '')].append(pkg)

for p in processes:
    order = p['order_no']
    for pkg in pkg_by_order.get(order, []):
        ...
```

**P1.2 条件前移到 SQL 层**

```python
# 当前：Python 过滤
packages = storage.get_packages(limit=2000)  # 加载全部
for pkg in packages:
    if pkg.get('status') != status_filter:  # Python 过滤
        continue

# 优化后：SQL WHERE 过滤
packages = storage.get_packages(
    limit=2000,
    status=status_filter,  # 传入 SQL WHERE 条件
    data_type=data_type_filter
)
```

#### P2: 数据库优化

**P2.1 字段裁剪**

```python
# 当前
"SELECT * FROM data_packages"

# 优化后（列表页只取必要字段）
SELECT 
    id, data_type, title, status, priority,
    related_order, related_process, target_operator,
    created_at, distributed_at, completed_at, completed_qty
FROM data_packages WHERE ...
```

**P2.2 添加联合索引**

```sql
-- 新增索引
ALTER TABLE data_packages 
ADD INDEX idx_order_status (related_order, status, created_at DESC);

-- 新增索引
ALTER TABLE data_packages 
ADD INDEX idx_type_status (data_type, status, created_at DESC);
```

#### P3: 缓存优化

**P3.1 智能缓存策略**

```python
# 当前：固定 5 分钟 TTL
CACHE_TTL = 300

# 优化后：根据数据变化频率动态调整
def get_cache_ttl(data_type):
    if data_type in ('process_report', 'process_task'):
        return 60   # 生产数据变化频繁，短 TTL
    elif data_type in ('flow_production',):
        return 180  # 排产数据中等频率
    else:
        return 300 # 其他数据标准 TTL
```

**P3.2 增量缓存更新**

```python
# 当前：全量刷新
def invalidate_work_order_cache():
    self.work_order_cache['data'] = None

# 优化后：增量更新
def update_package_in_cache(package_id, updates):
    for pkg in self.work_order_cache['data']:
        if pkg['id'] == package_id:
            pkg.update(updates)
            break
```

#### P4: 前端优化

**P4.1 虚拟列表**

```javascript
// 移动端只渲染可视区域的数据
const VirtualList = {
    render() {
        const visibleItems = this.getVisibleItems();
        return visibleItems.map(item => this.renderItem(item));
    },
    getVisibleItems() {
        const start = this.scrollTop / this.itemHeight;
        const end = start + this.visibleCount;
        return this.items.slice(start, end);
    }
};
```

**P4.2 分页加载**

```javascript
// 当前：一次加载 2000 条
const items = await fetch('/api/schedule/list').then(r => r.json());

// 优化后：分页加载，每页 50 条
const items = await fetch('/api/schedule/list?page=1&page_size=50').then(r => r.json());
```

## 八、总结

本计划针对排产与任务列表加载慢问题，从算法、数据库、缓存、前端四个层面进行优化：

1. **核心优化**：哈希分组将 O(n*m) 降为 O(n+m)
2. **数据库优化**：字段裁剪 + 联合索引
3. **缓存优化**：智能 TTL + 增量更新
4. **前端优化**：分页加载 + 虚拟滚动

预计整体优化后：
- 首次加载：3-5 秒 → < 1 秒
- 内存占用：减少 60%
- 响应体积：减少 70%
