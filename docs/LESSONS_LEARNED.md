## 2. 我们做错的 5 件事

### ❌ 错 1：只信单元测试

```python
# 测试里这样写
mock = MagicMock()
mock.publish_report_task.return_value = 'task_123'
service.do_publish(mock)
assert service.success
# 通过！✅
```

**为什么错**：mock 不验证 publisher.py 实际有 publish_report_task 方法。

**真相**：`ReportPublisher` 类根本没 `publish_report_task` 方法。运行时调用会 AttributeError。

### ❌ 错 2：只看 1 个文件，不读全链路

**我做的工作**：只读了 `publisher.py` 344 行

**应该读**：
- ✅ `publisher.py`（新代码）
- ✅ `desktop_container_integration.py`（被替换的）
- ✅ 5 个 service 文件（谁在调用）
- ✅ `standalone_dispatch_server.py`（5003 端口）
- ✅ `container_center_v5.py`（本地存储）

### ❌ 错 3：API 签名不匹配

**服务调用**：
```python
self._integration.publish_report_task(
    order_no='WO-001',
    process_name='拉丝',
    ...
)
```

**新 publisher 提供**：
```python
def publish(self, payload: Dict) -> bool  # 签名不一致
```

### ❌ 错 4：文档与代码不一致

| 文档声称 | 实际 |
|---------|------|
| "Q-B6 迁移 100%" | ❌ 5 个 service 仍调用旧 API |
| "124 测试通过" | ✅ 但测试是 mock 的 |
| "web 化完成" | ❌ 测试 4 fail + 10 error（未运行）|
| "92 处 str(e) 替换" | ❌ 实际 70 处（68 新增 + 2 新增）|

### ❌ 错 5：commit scope 蔓延

**原 commit 2599c47d**：
- 标题："fix: P2 security & perf fixes"
- 实际内容：98.2% 是 web 化骨架（3745 行新代码）
- **真实 P2 修复**：仅 60-70 行

## 3. 应该有的 5 步学习法

### ✅ 步骤 1：看完整代码链

```bash
# 改一个方法前，至少读 3 个相关文件
git grep "调用方法名" -- '*.py'
git grep "import 模块名" -- '*.py'
```

### ✅ 步骤 2：跑测试

| 类型 | 必须有 | mock 算不算 |
|------|--------|:-----------:|
| 单元测试 | ✅ | ✅ |
| 集成测试 | ✅ | ❌ |
| e2e 测试 | ✅ | ❌ |

### ✅ 步骤 3：真实点一次按钮

```
1. 启动所有依赖（docker-compose up）
2. 真实点一次业务按钮
3. 看数据落到哪（DB？HTTP？）
4. 看板能看到
```

### ✅ 步骤 4：多人多视角审计

```
✅ 4 人独立审计
✅ 至少 1 人必须得出不同结论
❌ 4 人结论一致 = 审计失真
```

### ✅ 步骤 5：文档与代码对比

```
✅ 数字必须精确（不要四舍五入）
✅ API 名必须一字不差
✅ 行号必须准确
✅ 文档的"已完成"必须真有测试覆盖
```

## 4. 3 条铁律（会议通过）

### 铁律 1：测试绿 ≠ 系统可工作

```
✅ 必须有：单元 + 集成 + e2e
✅ 集成测试必须真启动依赖
❌ mock 不算集成测试
```

### 铁律 2：改动前必读全链路

```
改 1 个方法，至少读：
- 被改的方法
- 调它的上游（service）
- 它调的下游（DB / HTTP / 另一个服务）
- 至少 3 个相关文件
```

### 铁律 3：审计必有人唱反调

```
✅ 4 人独立审计
✅ 至少 1 人必须得出不同结论
❌ 强行达成共识 = 失真
```
