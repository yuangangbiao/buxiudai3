## 修复详情

### P0-1: 工序列表性能优化

**问题**：工序列表 API `/processes` 响应时间 14.73 秒

**根因**：
1. `list_processes()` 函数中调用了两次 `_get_cached_work_orders(page=1, size=2000)`
   - 第3592行：用于计算 `order_task_count`
   - 第3720行：用于填充工序任务列表
2. 每次调用都触发数据库查询，无缓存复用

**修复方案**：
- 预加载工单数据到 `_cached_orders` 变量
- 复用预加载数据，避免重复查询
- 减少数据库往返次数

**修改文件**：[dispatch_center/_core.py](file:///d:/yuan/不锈钢网带跟单3.0/mobile_api_ai/dispatch_center/_core.py)

**预期效果**：响应时间从 14.73s 降至 <2s

---

### P1-1: process_code/process_name 数据填充

**问题**：工序列表返回的 `process_code` 和 `process_name` 字段为空

**根因**：
- `processes` 数据从 `_dispatch_cache.get_data()['processes']` 获取
- 原始数据不包含 `process_code` 和 `process_name` 字段

**修复方案**：
- 在返回数据前，实时计算 `process_code` 和 `process_name`
- 使用 `core.config.get_process_code(current_step)` 获取工序代码
- 使用 `current_step` 作为工序名称

**修改文件**：[dispatch_center/_core.py](file:///d:/yuan/不锈钢网带跟单3.0/mobile_api_ai/dispatch_center/_core.py)

**修复位置**：`list_processes()` 函数末尾排序前

---

### P1-2: task_by_status 统计数据修复

**问题**：`task_by_status` 和 `task_by_type` 统计数据为空

**根因**：
- `_get_client().query_documents('work_order', all=True)` 返回空结果
- 兜底逻辑只处理了 `status_stats`，未处理 `task_stats`

**修复方案**：
- 增强兜底逻辑：当 `status_stats` 为空或 `task_stats` 全为0时触发
- 从 `process_records` 表获取状态统计
- 根据 `flow_type` 推断 `task_type` 并填充 `task_stats`

**修改文件**：[dispatch_center/_core.py](file:///d:/yuan/不锈钢网带跟单3.0/mobile_api_ai/dispatch_center/_core.py)

**修复位置**：`get_stats()` 函数的兜底逻辑部分

---

### P1-3: 排班列表/待排班超时修复

**问题**：`/schedule/list` 和 `/schedule/pending` 接口超时（10秒）

**根因**：
1. `_query_mysql_workorders()` 使用 `SELECT *` 查询全字段
2. `LIMIT 200` 返回数据量大
3. 无查询超时设置

**修复方案**：
1. 字段裁剪：只查询必要字段（id, order_no, product_name, quantity, unit, customer_name, status, created_at, updated_at）
2. 减少 LIMIT：从 200 降至 100
3. 添加查询超时：`SET SESSION MAX_EXECUTION_TIME=5000`（MySQL 8.0+）

**修改文件**：[dispatch_center/schedule_routes.py](file:///d:/yuan/不锈钢网带跟单3.0/mobile_api_ai/dispatch_center/schedule_routes.py)

**修复位置**：`_query_mysql_workorders()` 函数
