## 五、P0 风险应对详情

### R1: 字段映射错误

**风险描述**：目标表字段名与 data_packages 不一致，导致查询返回 null 或报错

**概率**：30%（较高）
**影响**：前端显示异常，数据无法保存

**应对方案**：
```python
# 1. 编写字段映射验证 SQL
SELECT 
    dp.id,
    dp.related_order,
    sr.order_no,
    CASE WHEN dp.related_order = sr.order_no THEN 'OK' ELSE 'MISMATCH' END as status
FROM data_packages dp
LEFT JOIN schedule_records sr ON dp.id = sr.id
WHERE dp.data_type = 'schedule'
LIMIT 100;

# 2. 部署前验证
- 导出 100 条 data_packages 数据
- 执行映射后的 SQL
- 对比结果集
```

**验收标准**：映射前后数据一致率 ≥ 99%

---

### R2: 前端兼容性问题

**风险描述**：前端使用 `dp.related_order` 等字段，迁移后字段名变化导致 undefined

**概率**：20%
**影响**：前端页面显示 undefined

**应对方案**：
```python
# 返回前做字段别名映射
def _map_schedule_record(row):
    return {
        'id': row['id'],
        'data_type': 'schedule',
        'related_order': row['order_no'],  # 映射
        'title': row['product_name'],      # 映射
        'target_operator': row['operator_id'],  # 映射
        ...
    }
```

**验收标准**：前端页面无 undefined 报错

---

### R3: 并发数据不一致

**风险描述**：两个请求同时修改同一记录，后面的覆盖前面的

**概率**：15%
**影响**：数据丢失，状态不一致

**应对方案**：
```python
# 使用 FOR UPDATE 行锁
cur.execute("SELECT * FROM schedule_records WHERE id=%s FOR UPDATE", (id,))

# 使用原子 UPDATE
cur.execute("""
    UPDATE schedule_records 
    SET status='withdrawn', updated_at=NOW() 
    WHERE id=%s AND status != 'withdrawn'
""", (id,))
```

**验收标准**：并发测试 10 线程无数据丢失
