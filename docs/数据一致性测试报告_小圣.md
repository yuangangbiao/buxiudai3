### 🔴 P0 漏洞 1: 5001 工序 API 返"成功"但 DB 状态未变更

**测试用例**: 工序 PENDING→IN_PROGRESS / IN_PROGRESS→COMPLETED

**真实证据**（命令: `& "C:\Users\lenovo\AppData\Local\Python\bin\python.exe" scripts\test_consistency_xiaosheng.py`，时间 2026-06-23 13:51:00~01）:

```
[1] PUT /api/process/371/start (5001)
❌ 工序 PENDING→IN_PROGRESS: API code=0 DB status='待开始' start_time=None
[2] PUT /api/process/371/complete (5001)
❌ 工序 IN_PROGRESS→COMPLETED: API code=0 DB status='待开始' end_time=None
```

| 工序 ID | 调 start 前 DB | 调 start 后 API 返回 | 调 start 后 DB 实际 |
|---------|----------------|---------------------|-------------------|
| 371 | `status='待开始'` | `code=0 message='工序已开始'` | **`status='待开始'`** ⚠️ |
| 371 | `status='待开始'` | `code=0 message='工序已完成'` | **`status='待开始'`** ⚠️ |

**根因**（基于源码 desktop_web/server.py:2228-2297）:
- `api_process_start()` 行 2245-2247：`conn.commit()` → `if cur.rowcount == 0: return 400` → return 200
- `api_process_complete()` 行 2278-2282：UPDATE WHERE id=%s **没有 rowcount 检查** → 任何状态都返 200
- 5001 用 `get_connection()` 拿独立连接 → commit 后连接归还连接池

**业务影响**: 工人报工后系统显示"成功"，但 DB 状态没改 → **报工数据 100% 丢失**，工人重复报工会绕过状态机约束。这是发往生产计划的核心数据流。

**修复建议**:
- 把 `complete`/`reset` 也加 `if cur.rowcount == 0: return 400`（最简方案）
- 长期方案：用 `UPDATE ... WHERE id=%s AND status='expected_old_status'` + 严格 rowcount 检查

### 🔴 P0 漏洞 3: 并发报工超额 60% — read-then-write TOCTOU 漏洞

**测试用例**: 总报工数 vs 计划数（无超额）

**真实证据**（5 线程真实并发，命令同前）:
```
[并发] 5 线程同时报工, 总耗时 0.02s
  线程 4: completed_qty=4.0  status=生产中
  线程 3: completed_qty=7.0  status=生产中
  线程 0: completed_qty=10.0 status=已完成
  线程 2: completed_qty=13.0 status=已完成
  线程 1: completed_qty=16.0 status=已完成
❌ 总报工数 vs 计划数: 期望=16.0 DB实际=16.0 计划数=10 Δ=+0.000
❌ 无负库存 + 在计划内: completed=16 planned=10 ❌ 异常
```

| 测试条件 | 值 |
|---------|---|
| 计划数 (planned_qty) | 10 |
| 并发线程数 | 5 |
| 每线程报工数 | 3 |
| 理论应得报工数 | 5 × 3 = 15（应被库存 10 截断） |
| 实际 DB 完成数 | **16** |
| **超额率** | **+60%** ⚠️ |

**根因**（基于源码 desktop_web/server.py:2335-2399）:
```python
# 当前实现（read-then-write, 非原子）
cur.execute("SELECT * FROM process_records WHERE id=%s FOR UPDATE", (process_id,))
old = cur.fetchone()
old_completed = float(old['completed_qty'] or 0)
new_completed = old_completed + qty  # ← 读到的"old"都是同一份, 5 线程都从 baseline 开始加
cur.execute("UPDATE process_records SET completed_qty=%s ...", (new_completed, ...))
```

**问题本质**: `SELECT FOR UPDATE` 锁住行但只在当前连接的当前事务有效；5 个线程独立连接（5001 `get_connection()` 每次新建），所以每线程都拿到同一份 baseline（completed_qty=1），各自 +3 = 4/7/10/13/16 累加写回。

**业务影响**:
- 计划数 10 的工序实际被报工 16 个 → **多生产 6 个 / 库存虚增 / 成本计算失真**
- 订单交付数量与生产数量对不上
- 任何"按计划数锁库存"逻辑全部失效

**修复建议**（用数据库原子加法）:
```python
# 替换 read-then-write 模式
cur.execute("""
    UPDATE process_records
    SET completed_qty = completed_qty + %s,
        qualified_qty = qualified_qty + %s,
        work_hours = work_hours + %s,
        status = CASE WHEN completed_qty + %s >= planned_qty THEN '已完成' ELSE '生产中' END,
        worker = %s,
        record_date = NOW()
    WHERE id = %s
""", (qty, qualified, hours, qty, worker, process_id))
# 然后 SELECT 拿最新 completed_qty 用于响应
```

### 🔴 P0 漏洞 4: SYNC_BRIDGE 跨系统同步 0 生效 — 修复空跑

**测试用例**: 5003 队列有新增: sync_log / sync_logs / report_queue / sync_queue

**真实证据**:
```
[预检] os.environ SYNC_BRIDGE_URL = ''
[预检] .env 文件 SYNC_BRIDGE_URL = ''
❌ SYNC_BRIDGE_URL 配置: ❌ 环境变量和 .env 文件都未配置 SYNC_BRIDGE_URL
[基线] 5003 同步队列: {'sync_log': 12, 'sync_logs': 369, 'report_queue': 82, 'sync_queue': 25}
[触发] 4 个工序 API: [start=200/0, complete=200/0, reset=200/0, report=200/0]  ← 5001 全部返成功
[触发后] 5003 同步队列: {'sync_log': 12, 'sync_logs': 369, 'report_queue': 82, 'sync_queue': 25}
                                                    ↑                ↑                ↑                ↑
                                                   Δ=0             Δ=0             Δ=0             Δ=0
```

| 同步目标 | 基线 | 触发后 | Δ | 结论 |
|---------|------|--------|---|------|
| container_center.sync_log | 12 | 12 | **0** | 5001 触发 4 个工序 API 但 sync_log 无变化 |
| container_center.sync_logs | 369 | 369 | **0** | 同上 |
| container_center.report_queue | 82 | 82 | **0** | 同上 |
| container_center.sync_queue | 25 | 25 | **0** | 同上 |

**根因**（基于源码 desktop_web/server.py:73-84）:
```python
def _sync_bridge(data):
    url = os.getenv('SYNC_BRIDGE_URL')
    if not url:
        return  # ← 静默 return, 无日志无告警
    data['source'] = 'web5001'
    requests.post(url, json=data, timeout=5)
```

- **5001 启动环境的 `SYNC_BRIDGE_URL` 未配置**（os.environ 和 .env 都没有）
- 函数无 try 外的日志（只 catch 异常，URL 缺失直接 return）
- P0 同步修复报告"4 个工序 API 已加同步调用"代码层面 OK，**但配置层没补齐，0 真实同步发生**
- 即使配置补上，5003 端点 `/api/sync-bridge/receive` **不存在**（实测 GET 5003 端点表只有 `/api/sync/*`（sync_bp）和 `/api/sync-queue/*`）

**业务影响**:
- 5001 报工/工序推进 100% **不会通知到 5003/5008/桌面端**
- 5003 调度中心看不到 5001 的任何变更
- "web5001 与桌面端数据一致"是假象 — 单边写入，无同步链路
- 这是 P0 修复要解决的核心问题，**实际未解决**

**修复建议**:
1. `.env` 补齐 `SYNC_BRIDGE_URL=http://127.0.0.1:5003/api/sync/receive`（5003 真实端点见 sync_bp.py）
2. 5001 `_sync_bridge()` 在 `url` 缺失时打印 warning（`logger.warning('[SYNC_BRIDGE] URL 未配置, 同步已跳过')`），便于部署时发现
3. 5003 新建 `/api/sync-bridge/receive` 端点，接收 `process_*` 事件并落 `sync_log` 表
