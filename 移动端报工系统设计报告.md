## 1. 项目概述

### 1.1 项目背景

不锈钢网带跟单系统3.0目前为桌面端应用（Python+Tkinter），运行于Windows系统，连接MySQL数据库。随着业务发展，车间工人、班组长、管理人员对移动办公的需求日益迫切，需要一款移动端报工系统来实现：

- 扫码快速报工
- 实时查看工序进度
- 移动质检录入
- 消息实时推送
- 移动审批流
- 生产数据看板

### 1.2 项目目标

| 目标 | 描述 |
|------|------|
| 提升报工效率 | 扫码即可报工，减少手工录入时间 |
| 数据实时同步 | 移动端与桌面端共用同一数据库，实时同步 |
| 覆盖全员场景 | 工人扫码报工、班组长审批、管理层看板 |
| 微信原生 | 基于企业微信小程序框架，充分利用微信能力 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 复用优先 | 移动端复用桌面端工序模板、质检配置、数据表 |
| 简洁高效 | 扫码→填报工→提交，三步完成报工 |
| 数据一致 | 报工数据直接写入现有表，桌面端实时可见 |
| 微信原生 | 基于企业微信小程序框架，充分利用微信能力 |

## 2. 需求分析

### 2.1 业务现状分析

#### 2.1.1 现有系统核心数据表

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `orders` | 订单主表 | order_no, customer_name, product_type, material, quantity, status |
| `production_orders` | 生产工单 | order_no, order_id, assigned_to, priority, status |
| `process_records` | 工序记录 | process_name, process_seq, planned_qty, completed_qty, qualified_qty, worker, status |
| `quality_records` | 质检记录 | inspection_type, inspection_items, result, defect_qty, handling_method, inspector |
| `operators` | 操作员 | operator_id, name, role |

#### 2.1.2 生产工序流程（7步）

```
原材料准备 → 编织成型 → 热处理 → 整形校直 → 抛光处理 → 质量检验 → 包装入库
```

> **注意**：工序流程通过工序模板管理，可在桌面端自定义。排产时选择模板生成工序记录。

#### 2.1.3 质检类型

| 类型 | 说明 |
|------|------|
| 首检 | 订单开始生产前的首件检验 |
| 巡检 | 生产过程中的定期抽检 |
| 终检 | 订单完成前的最终检验 |

#### 2.1.4 质检结果

| 结果 | 处理 |
|------|------|
| 合格 | 继续生产或进入下一工序 |
| 不合格 | 根据处理方式（返工/降级/报废）处理 |
| 待复检 | 暂缓，等待复检结果 |

### 2.2 功能需求

| 功能 | 描述 | 优先级 |

| 移动审批 | 急单加急、外协申请、异常处理审批 | P1 |
| 生产看板 | 个人日报、班组看板、管理看板 | P1 |

#### 2.2.3 增值功能（Nice to Have）

| 功能 | 描述 | 优先级 |

### 2.3 用户角色

| 角色 | 操作 | 权限 |
|------|------|------|
| 车间工人 | 扫码报工、查看我的任务 | 报工、查看 |

### 2.4 业务流程

#### 2.4.1 扫码报工流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                         扫码报工流程                                  │
└─────────────────────────────────────────────────────────────────────┘

   ① 工人扫工单二维码（工单码）
         │
         ▼
   ② 系统解析二维码 → 获取 order_no
         │
         ▼
   ③ 查询生产工单 + 订单信息
         │
         ▼
   ④ 查询工序记录列表（process_records）
         │
         ▼
   ⑤ 显示界面：
       - 订单信息（客户、产品、数量）
       - 工序进度列表
       - 当前应报工的工序
       - 报工表单
         │
         ▼
   ⑥ 工人填写报工数据
       - 完成数量
       - 合格数量
       - 工时
       - 备注（可选）
         │
         ▼
   ⑦ 提交报工
         │
         ├─── 当前工序 ≠ 质量检验 ───┐
         │                              │
         ▼                              │
   更新 process_records ────────────→ ⑧ 报工完成，显示下一工序
         │                              │
         │  当前工序 = 质量检验          │
         ▼                              │
   ⑨ 弹出质检表单                      │
         │                              │
         ▼                              │
   ⑩ 填写质检数据 ─────────────────────┘
         │
         ▼
   ⑪ 提交质检 → 写入 quality_records
         │
         ▼
   ⑫ 判断终检结果
         │
         ├── 终检+合格 → 触发订单完成流程
         │
         └── 其他 → 按规则处理状态流转
```

```

│                         质检录入流程                                  │
└─────────────────────────────────────────────────────────────────────┘

   质检触发条件：当前工序 = "质量检验"

   ┌─────────────────────────────────────────────┐
   │              质检表单（移动端）               │
   ├─────────────────────────────────────────────┤
   │                                             │
   │  质检类型: ○ 首检  ●巡检  ○终检             │
   │                                             │
   │  质检项目（可多选）:                         │
   │  ┌─────────────────────────────┐           │
   │  │ ☑ 外观检验          [合格] │           │
   │  │ ☑ 尺寸检测          [合格] │           │
   │  │ ☑ 拉力测试          [待复检]│           │
   │  │ ☑ 表面处理          [合格] │           │
   │  └─────────────────────────────┘           │
   │                                             │
   │  不良数量: [ 5 ]                           │
   │  不良描述: [ 边缘有轻微划痕 ]                │
   │                                             │
   │  处理方式: ○ 返工  ●降级  ○报废  ○特采      │
   │                                             │
   │  质检结论: ○ 合格  ●不合格  ○ 待复检        │
   │                                             │
   │  质检员: [ 王五 ]                          │
   │  备注:   [________________]                │
   │                                             │
   │  [📷 拍照上传（可选）]                      │
   │                                             │
   └─────────────────────────────────────────────┘

         │
         ▼

   提交质检 → QualityDAO.create()

         │
         ▼

   状态流转规则（与桌面端一致）：
   ┌─────────────────────────────────────────────┐
   │  终检 + 合格  → 订单完成（成品入库）          │
   │  终检 + 不合格 → 保持状态，等待处理           │
   │  首检/巡检 + 合格 → 更新为「质检中」          │
   │  任意类型 + 待复检 → 更新为「质检中」         │
   └─────────────────────────────────────────────┘
```

#### 4.2.4 mobile_notifications（消息通知表）- 新增

```sql
CREATE TABLE mobile_notifications (
    id INT PRIMARY KEY AUTO_INCREMENT,
    recipient_id VARCHAR(50) NOT NULL,
    type VARCHAR(30) NOT NULL,
    title VARCHAR(100) NOT NULL,
    content TEXT,
    related_order_id INT,
    related_production_id INT,
    is_read TINYINT DEFAULT 0,
    read_time DATETIME,
    created_at DATETIME DEFAULT NOW(),
    expires_at DATETIME,
    FOREIGN KEY (related_order_id) REFERENCES orders(id),
    FOREIGN KEY (related_production_id) REFERENCES production_orders(id)
);

CREATE INDEX idx_recipient_read ON mobile_notifications(recipient_id, is_read);
CREATE INDEX idx_created ON mobile_notifications(created_at);
```

**消息类型（type）**：

| 类型 | 说明 |
|------|------|
| REPORT_REMIND | 报工提醒 |
| REPORT_COMPLETE | 报工完成 |
| OVERDUE_WARNING | 逾期预警 |
| DUE_WARNING | 即将到期 |

## 6. 功能模块设计

### 6.1 扫码报工模块

#### 6.1.1 功能描述

工人通过扫描工单二维码，快速查看订单信息和工序进度，填报工数据并提交。

#### 6.1.2 二维码生成

**员工二维码**：
- 内容格式：`WORKER:{operator_id}`
- 示例：`WORKER:OP001`

**工单二维码**：
- 内容格式：`WORKORDER:{order_no}`
- 示例：`WORKORDER:GY2026042901`

#### 6.1.3 报工规则

| 规则 | 说明 |
|------|------|
| 只能报当前工序 | 只能填报当前应报工的工序 |
| 数量校验 | 完成数量 ≤ 计划数量 |
| 合格数量 ≤ 完成数量 | 合格数量不能大于完成数量 |
| 工时录入 | 可选，建议填写 |
| 自动开始 | 第一次报工时自动记录开始时间 |
| 自动完成 | 报工状态=完成时自动记录结束时间 |

### 6.2 质检模块

#### 6.2.1 功能描述

当报工工序为"质量检验"时，弹出质检表单，填写质检数据后提交。

#### 6.2.2 质检规则

| 规则 | 说明 |
|------|------|
| 终检+合格 | 自动创建成品入库，更新订单为已完成 |
| 终检+不合格 | 保持当前状态，等待处理 |
| 首检/巡检+合格 | 更新订单状态为"质检中" |
| 待复检 | 更新订单状态为"质检中" |

### 6.3 消息通知模块

| 类型 | 标题 | 触发条件 | 接收人 |
|------|------|----------|--------|
| REPORT_REMIND | 报工提醒 | 工序开始前5分钟 | 工序负责人 |
| REPORT_COMPLETE | 报工完成 | 工人提交报工 | 班组长 |
| OVERDUE_WARNING | 逾期预警 | 订单逾期 | 管理员 |
| DUE_WARNING | 即将到期 | 交货日期前3天 | 班组长 |
| QC_RESULT | 质检结果 | 质检完成 | 申请人 |
| ORDER_COMPLETE | 订单完成 | 终检合格 | 相关人员 |
| APPROVAL_RESULT | 审批结果 | 审批完成 | 申请人 |

### 6.4 移动审批模块

#### 6.4.1 审批类型

| 类型 | 说明 | 审批人 |
|------|------|--------|
| URGENT_ORDER | 急单加急 | 管理员 |
| OUTSOURCE | 外协申请 | 管理员 |
| EXCEPTION | 异常处理 | 班组长/管理员 |
| ORDER_CHANGE | 订单变更 | 管理员 |
| WORKTIME_ADJUST | 工时调整 | 班组长 |

### 6.5 生产看板模块

#### 6.5.1 看板类型

| 看板 | 用户 | 内容 |
|------|------|------|
| 我的日报 | 工人 | 今日报工统计、工序明细 |
| 我的周报 | 工人 | 本周每日统计、周排名 |
| 班组看板 | 班组长 | 班组产量、成员排名 |
| 生产看板 | 管理员 | 全厂订单、交付率、逾期 |

#### 6.5.2 统计规则

| 统计项 | 计算方式 |
|--------|----------|
| 完成数量 | SUM(completed_qty) |
| 合格数量 | SUM(qualified_qty) |
| 合格率 | 合格数量 / 完成数量 × 100% |
| 工时 | SUM(work_hours) |
| 准时率 | 按时完成订单数 / 总订单数 × 100% |
