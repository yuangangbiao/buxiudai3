## P0 — 致命问题 (4 项)

### P0-1: app.py 报工程序直接操作 SQLite + cc_ 前缀
- **文件**: `mobile_api_ai/app.py`
- **行号**: 159, 164-174, 198
- **问题**:
  1. 第 159 行 `cc.storage._conn.cursor()` — 走的是 ContainerCenter 的内部 SQLite 连接，不是 MySQL
  2. 第 164 行 `row['content']` — 从 SQLite dict 游标读取
  3. 第 173 行 `UPDATE cc_data_packages` — 写回 SQLite（表名也有 cc_ 前缀）
  4. 第 180-212 行 — 另开 MySQL 连接同步 `UPDATE cc_process_records`（另一个 cc_ 前缀）
- **影响**: 直接报工写 SQLite > 延迟同步 MySQL > 数据一致性问题
- **修复方向**: 报工程序应直接写 MySQL，不再通过 SQLite

### P0-2: sync_bridge.py 仍在桥接 SQLite→MySQL
- **文件**: `mobile_api_ai/sync_bridge.py`
- **行号**: 383
- **问题**: `sql_conn = sqlite3.connect(cc_db)` — 从 SQLite 的 `data_packages` 表读数据再写入 MySQL
- **影响**: 如果 Container Center 已直接写 MySQL，此桥接多余；如果未写，则存在数据延迟和双写风险
- **修复方向**: 明确单一数据源，废弃此桥接层

### P0-3: container_center/router.py 仍用 SQLite 做路由器存储
- **文件**: `mobile_api_ai/container_center/storage/router.py`
- **行号**: 127, 142
- **问题**: 核心存储路由 `sqlite3.connect(db_path)` — 不使用 MySQL
- **影响**: 容器中心内部路由数据不经过 MySQL，形成数据孤岛

### P0-4: 数据库名不一致
- **问题**: 
  - `container_center_v5.py MYSQL_CFG` → `database='container_center'`
  - `app.py:182` → `database=CONTAINER_MYSQL_DATABASE='container_center'`
  - `storage_mysql.py MYSQL_CFG` → `database=steel_belt`
  - `dispatch_center.py MYSQL_CFG` → `database=steel_belt`
- **影响**: 不同服务连不同数据库，跨服务查询失败、数据分散
- **修复方向**: 明确 container_center 库 vs steel_belt 库的职责分界，统一规范

## 架构符合性检查

### 已正确实现的项目 ✅
- `container_center_v5.py` 使用 `CONTAINER_STORAGE_TYPE` 动态选择存储
- `storage_mysql.py` 有完整的 TABLES_DDL 和 _ensure_tables()
- `dispatch_center._sync_to_mysql()` 正确写入 MySQL
- `container_config._load_operators_from_db()` 从 MySQL 加载操作员
- `server_launcher.py` 向子进程传递 MYSQL_PASSWORD

### 架构不符的地方 ❌
1. 数据流向未统一：SQLite ↔ MySQL 双向存在
2. 配置分散：9 个 MYSQL_CFG 定义，互不一致
3. 跨库混乱：steel_belt 和 container_center 职责不清晰
4. sync_bridge 仍运行在 SQLite→MySQL 模式下
5. 报工程序 (app.py) 同时操作 SQLite 和 MySQL
