## 四、技术约束（修正后）

### 4.1 禁止项

| 禁止行为 | 说明 |
|---------|------|
| 修改生产代码提高覆盖率 | 不得为了测试而改动业务逻辑 |
| 删除已有测试 | 存量测试不得删除或注释 |
| 修改 `steel_belt` 数据库 | 桌面库只读 |
| **新建 SQLite 连接用于生产代码路径** | 生产代码的导入链不应依赖 SQLite（注：`conftest.py` 中的 `empty_sqlite` fixture 作为测试基础设施是允许的。**本约束适用于生产代码，不禁止测试 fixture 中使用 SQLite 临时数据库**） |
| 编造不符合业务逻辑的测试数据 | 测试数据必须符合业务约束 |
| 跳过 pytest 全量验证 | 每次批量改动后必须 `pytest -q` 全绿 |
| 写无断言的空测试 | 每个测试至少包含 1 个有意义的 assert |

> 🔴 **修正（QA-005）**：明确了"禁止新建 SQLite 连接"的范围，同时新增"空测试禁止"约束。

### 4.2 测试代码质量门禁（新增）

| 要求 | 说明 |
|------|------|
| 每个测试至少 1 个业务行为断言 | 禁止仅验证类型或 `assert result is not None` 类伪测试 |
| 分支覆盖率 ≥80% | 新增测试的 if/else 分支应尽量覆盖 |
| 禁止 try-except-pass 模式 | 不允许空异常捕获来刷覆盖率 |
| 参数化测试优先 | 对边界条件使用 `@pytest.mark.parametrize` |
| 测试文件独立可运行 | `pytest tests/unit/test_xxx.py -v` 应能单文件通过 |

> 🔴 **新增（AUDIT-016/AUDIT-018）**：明确测试代码质量要求，防止伪测试刷覆盖率。

### 4.3 必须遵守

| 要求 | 说明 |
|------|------|
| 使用项目现有测试框架 | pytest + mock，沿袭 `tests/` 目录现有风格 |
| 每个新测试文件应可独立运行 | `pytest tests/unit/test_xxx.py -v` |
| 使用 conftest.py 的 mock_db fixture | 强依赖数据库的模块优先使用已有 fixture（标准 `unittest.mock.patch` 也可） |
| **模块级 conftest.py** | 新增模块的特定 fixture 放在对应目录，避免修改全局 conftest.py |
| 写完后运行全量测试 | 确保 1300+ 用例全绿 |

## 八、方案完整性补充

### 8.1 测试数据工厂（新增）

为减少重复造数据和数据不一致，建议在 `tests/conftest.py` 或新建 `tests/factories/` 目录下建立以下默认工厂：

```python
# tests/factories/order_factory.py 示例
def make_order(**kwargs):
    """创建默认订单测试数据"""
    default = {
        "order_no": "ORD-TEST-001",
        "customer": "测试客户",
        "status": "active",
        "created_at": "2026-01-01",
    }
    default.update(kwargs)
    return default

def make_process(**kwargs):
    """创建默认工序测试数据"""
    default = {
        "process_code": "P001",
        "process_name": "测试工序",
        "order_no": "ORD-TEST-001",
        "sequence": 1,
    }
    default.update(kwargs)
    return default
```

### 8.2 新增代码覆盖率处理（新增）

开发期间若新增生产代码：
1. 新增代码须同时编写测试，覆盖率 ≥80%
2. 每日跟踪覆盖率时同时记录代码总量变化
3. 目标动态调整：`目标覆盖行数 = (当前总语句 + 预估新增语句) × 90%`

### 8.3 services/ 目录处理说明（新增）

修正 `_coverage_parse.py` 的 PKG_PREFIX_MAP 后，`services/` 下的代码将被正确统计。建议操作顺序：
1. 修正 `_coverage_parse.py` → 增加 `'services': 'services/'` 映射
2. 重新运行 `python _coverage_parse.py` → 获得准确覆盖率数据
3. 评估 services/ 的代码是否需要纳入 90% 目标
4. 若需要，当前 90% 目标的难度将增加（约 9 个文件 + 数千行语句）
5. 根据新数据 **重新评估本方案的目标和阶段划分**
