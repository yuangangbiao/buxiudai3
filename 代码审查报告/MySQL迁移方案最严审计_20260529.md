## P0 — 致命缺陷 (4 项)

### P0-1: Router 改造方案不可行 — 影响面被严重低估

**方案说**: "保留 ROUTING_TABLE 路由逻辑，底层全部走 MySQL"

**实际代码**: `router.py` 的 DB_FILE_MAP 定义了 **12 个独立 SQLite 文件**（orders.db, production.db, equipment.db...），ROUTING_TABLE 将 40+ 个 doc_type 路由到这些文件。

**致命问题**: 
- Router 并非只被 container_center_v5.py 使用，还被 **5 个存储模块** 直接依赖：
  - `container_center/storage/document_store.py`
  - `container_center/storage/config_store.py`  
  - `container_center/storage/index_store.py`
  - `container_center/storage/alert_store.py`
  - `container_center/storage/__init__.py`
- `container_center_api.py` 也直接 import DatabaseRouter
- `dispatch_center.py` 通过 container_center 间接调用
- **文件列表只列了 6 个文件，实际需改 11+ 个**

**正确方案**: Router 不应该是"改造"而应**整体废弃**。所有 doc_type 对应的数据写入 `container_center` MySQL 库的同名表，通过 `storage_mysql.py` 的 `_ensure_tables()` 自动建表。

### P0-2: 双库事务跨越不可行

**方案说** (1.2.4):
```
BEGIN TRANSACTION (container_center)
UPDATE data_packages
UPDATE process_records (steel_belt)  
COMMIT
```

**致命问题**: pymysql 的 `BEGIN/COMMIT` 是**单库事务**，不能跨越 `container_center` 和 `steel_belt` 两个 MySQL 数据库。需要 **XA 分布式事务**或**两阶段提交 (2PC)**，代码复杂度剧增。

**正确方案**: 
- 方案A: 合并两库为单库（消除跨库问题）
- 方案B: 先写 container_center → 成功后写 steel_belt → steel_belt 失败则补偿回滚 container_center（最终一致性）
- **推荐方案A**: 审计中双库边界其实没有强隔离理由，合并简化一切

### P0-3: SchedulerManager 误判为业务数据

**方案说**: L4496/4510/4523 的内部 helper 类改为 MySQL 连接

**实际代码**: 这三行在 `SchedulerManager._init_db/_get_config/_save_config` 中，操作的是 `scheduler_configs` 表——**调度器本地配置**（定时任务启用/间隔），存储在 `data/scheduler_configs.db`。

**致命问题**: 这是**非业务本地状态**。如果把调度器配置存 MySQL：
- MySQL 断连 → 调度器无法读取配置 → 所有定时任务停止
- 与方案自身 1.2.7 节（SQLite 保留范围中明确写"调度器配置保留 SQLite"）**自相矛盾**

**正确方案**: 这三行不纳入迁移，保持 SQLite。

### P0-4: USE_SQLITE 回退开关不存在

**方案说**: `.env` 中 `USE_SQLITE=true` 可紧急回退

**实际代码**: 搜索整个项目，`USE_SQLITE` 只存在于 `.env` 的一条注释行（`USE_SQLITE=false`），**没有代码读取此变量**。实际控制存储类型的是 `CONTAINER_STORAGE_TYPE`（容器中心）和 `MYSQL_CFG`（其他模块）。

**致命问题**: 回退方案引用了一个**不存在的机制**。如果 MySQL 故障，无法通过改配置回退。

**正确方案**: 统一使用 `CONTAINER_STORAGE_TYPE=sqlite|mysql` 控制所有模块的存储选择。

## P1 — 严重问题 (5 项)

### P1-1: wechat_message_store DDL 缺失

**方案说**: 改为 `CREATE TABLE IF NOT EXISTS` MySQL DDL

**实际代码**: `storage_mysql.py` 的 `TABLES_DDL` 中没有 `wechat_messages_*` 表定义。按月分表的 DDL 需要动态生成。

**风险**: 首次写入消息时表不存在 → 报错。

**修复**: 在 `_ensure_tables()` 或 `wechat_message_store` 初始化中增加动态建表逻辑。

### P1-2: 任务依赖图有误 — T04 应依赖 T02

**方案说**: T04 只依赖 T01

**实际情况**: T04 的 sync/handlers 需要读写 container_center 库的表（workers, attendance 等），这些表在 T02 改造 router → 统一建表后才就绪。

**修复**: T04 依赖改为 `T01 + T02`。

### P1-3: 文件清单遗漏 5 个存储模块

**方案 2.1 节** 列出需修改文件 18 个，但遗漏：

| 遗漏文件 | 原因 |
|----------|------|
| `container_center/storage/document_store.py` | 直接 import DatabaseRouter |
| `container_center/storage/config_store.py` | 同上 |
| `container_center/storage/index_store.py` | 同上 |
| `container_center/storage/alert_store.py` | 同上 |
| `container_center/storage/__init__.py` | 导出 DatabaseRouter |

### P1-4: cc_ 清理重复分配

**T03** 和 **T05** 都列了 `container_center_api.py` 的 cc_ 清理。应只在 T03 做一次，T05 只做验证。

### P1-5: sync/handlers 改造缺少函数级方案

**方案仅说**: `sqlite3.connect(db_path)` → `pymysql.connect(**MYSQL_CFG)`

**实际问题**: 
- `worker_handler.py` 有 `_cs_cursor()` 上下文管理器（返回 SQLite cursor）
- `sub_step_handler.py` 有 `_cs_cursor()` 和 `_container_cursor()` 两个 helper
- 改为 MySQL 后，`.fetchone()` 返回 tuple 而非 Row 对象（除非设 `cursorclass=DictCursor`）

**修复**: 需要为每个 handler 提供具体的替换代码段。

## 正确方案修正建议

### 1. 合并双库为单库（解决 P0-2）

```
steel_belt 统一数据库：
├── orders, process_records, process_sub_steps (生产核心)
├── data_packages, dispatch_commands (容器中心)
├── sync_logs, data_flow_logs, data_collection_records (数据流)
├── workers, attendance (人员)
├── wechat_messages_YYYY_MM (消息)
├── operators, enterprise_personnel, enterprise_structure (组织)
└── operation_logs (日志)
```

好处：消除跨库事务问题、简化配置（只需一个 MYSQL_CFG）、减少连接数。

### 2. 废弃 Router 而非改造（解决 P0-1）

Router 完全移除。所有 doc_type → 表名直接映射，通过 `storage_mysql.py._ensure_tables()` 统一管理 DDL。

### 3. 修正依赖图

```
T01(配置) ──→ T02(存储+router废弃) ──→ T03(报工直写)
    │                                        │
    └──→ T04(调度+handler改造) ←─────────────┘
              │
              └──→ T05(清理验证)
```

### 4. 统一回退开关

使用 `CONTAINER_STORAGE_TYPE=sqlite|mysql` 统一控制，移除不存在的 `USE_SQLITE`。

### 5. 修正文件清单（+9 个文件）

新增：
```
container_center/storage/document_store.py   # router 重依赖
container_center/storage/config_store.py     # 同上
container_center/storage/index_store.py      # 同上
container_center/storage/alert_store.py      # 同上
container_center/storage/__init__.py         # 路由导出
mobile_api_ai/container_center/api/__init__.py  # router import
mobile_api_ai/container_center/api/app.py       # router import
mobile_api_ai/container_center/__init__.py      # router import
tests/test_migration.py                      # 新增：迁移验证测试
```
