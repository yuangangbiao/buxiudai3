### QA-002 🔴 引用的 Mock 基础设施不存在

**风险等级**: 高  
**涉及章节**: 三（3.3 强依赖数据库模块）  
**问题描述**: 方案第 3.3 节的示例代码引用了一个不存在的 mock 基础设施：

```python
# 方案中的代码：
from unittest.mock import patch
@patch('tests.mocks.mock_storage.MockMySQLStorage')
def test_save_order(mock_storage_cls):
    ...
```

**实际验证**:
- `tests/mocks/` 目录 **不存在**
- `tests/mocks/mock_storage.py` **不存在**
- `MockMySQLStorage` 类 **不存在**
- `storage.mysql_storage` 模块位于 `mobile_api_ai/storage/mysql_storage.py`，**不在主项目的 import 路径中**

当前项目实际使用的测试模式是 `conftest.py` 中的 `mock_db` fixture（直接 mock `get_connection` 返回 MagicMock），与方案描述的 mock 模式完全不同。

**影响**: 执行者无法直接复制方案中的测试代码运行，所有 3.3 节的示例代码均不可执行。

**建议**: 
1. 删除/替换 3.3 节的错误示例代码
2. 明确指示使用 `conftest.py` 中已有的 `mock_db` fixture 模式
3. 如需提供 MockMySQLStorage，需先在 `tests/` 下创建该基础设施

### QA-003 🟡 _database_legacy.py 行数严重低估

**风险等级**: 中  
**涉及章节**: 二（Phase 2c）  
**问题描述**: 方案称 `_database_legacy.py` 为 491 行、估时 15h，但实际文件为 **1625 行**（约 3.3 倍）。

**连锁影响**:
- 未覆盖行：按比例估算为 1625 × 90.22% ≈ **1466 行未覆盖**（而非方案中的 443 行）
- 估时：按方案标准至少需 **45-50h**（而非 15h）
- Phase 2 的覆盖率增量计算将不准确
- 方案声称"Phase 2 完成后可达 90%+"，但仅此一个模块的偏差就可能影响到 90% 目标是否可达

**建议**: 
1. 重新测量该模块的真实覆盖率并更新方案数据
2. 重新评估 Phase 2c 是否真的能达 90%
3. 将该模块单独列为 Phase 2d 或 Phase 3（高风险区）

### QA-004 🟡 "假测试"刷覆盖率风险

**风险等级**: 中  
**涉及章节**: 三（3.1、3.2）  
**问题描述**: 方案中的示例测试仅验证返回值类型而非业务逻辑，存在刷覆盖率但无防护价值的风险。

**方案中的示例**:
```python
def test_clean_old_logs():
    result = clean_old_logs(days=30)
    assert isinstance(result, int)   # 仅验证类型，未验证删除逻辑是否正确
```

**风险分析**:
- 若函数内部有 bug（如误删了不该删的日志），此测试仍然通过
- 若函数抛异常但返回了默认值（如 `return 0`），此测试仍然通过
- 这类测试对代码行覆盖率有贡献，但对回归防护价值极低
- 6 个推荐模块若全部按此模式编写，将产生约 100+ 行"伪覆盖"

**建议**:
1. 明确要求每个测试至少验证 1 个业务行为（如"30 天以前的日志被精确删除"）
2. 对纯数据/逻辑类模块，要求参数化测试覆盖边界条件（如 days=0, days=365, days=None）
3. 在方案中增加"伪覆盖"的警示说明

### QA-005 🟡 SQLite fixture 违反自身"禁止新建 SQLite 连接"约束

**风险等级**: 中  
**涉及章节**: 四（4.1 禁止项）  
**问题描述**: 方案 4.1 明确禁止"新建 SQLite 连接"，但 `tests/conftest.py` 中现有的 `empty_sqlite` fixture 正是创建 SQLite 数据库的。

**矛盾分析**:
- 方案说：禁止新建 SQLite 连接，必须使用 `storage.mysql_storage`
- 实际情况：现有测试大量使用 `sqlite3.connect` 创建临时数据库
- 如果严格执行方案禁止项，现有 `conftest.py` 中约 80 行 SQLite 代码将被废弃
- 如果保留现有 SQLite 代码，则方案 4.1 的约束形同虚设

**建议**:
1. 明确方案中"禁止新建 SQLite 连接"的适用范围（是仅针对新增测试，还是包括存量测试？）
2. 如果允许 SQLite，删除或修改 4.1 中的禁止项
3. 考虑是否将现有 `empty_sqlite` fixture 作为标准做法写入方案

### QA-007 🟢 core/app.py 在 Phase 1 和 Phase 2a 重复列出

**风险等级**: 低  
**涉及章节**: 二  
**问题描述**: Phase 1（P3）已将 `core/app.py`（60 行）列为 0% 模块，但 Phase 2a 的性价比表中再次列出 `core/app.py`（同样是 60 行，但标注为 0.00%）。

- 数据矛盾：Phase 1 完成后标注覆盖率 50.44%，但如果 app.py 已算入 Phase 1，这个数字的计算逻辑不透明

**建议**: 明确 `app.py` 到底归哪个阶段，调整 Phase 1 描述和覆盖率预测。

### QA-008 🟢 Phase 2 总估时远超可用时间

**风险等级**: 低  
**涉及章节**: 二  
**问题描述**: Phase 2 的分阶段估时合计远超方案声称的"第 3-10 天"可用时间。

**计算**:
- Phase 2a 估时：~19h
- Phase 2b 估时：~27.5h  
- Phase 2c 估时：~109h
- **合计**: ~155.5h
- 8 天可用工时（按每天 8h 计）：64h
- 缺口：91.5h（约 11 个工作日）

即便 Phase 2c 标注为"第 9-15 天"（7 天 = 56h），但 Phase 2a+b+c 合计 155.5h > 120h（15 天 × 8h）。

**建议**: 
1. 重新评估每个模块的估时（部分模块可能高估）
2. 明确 Phase 2c 中哪些是"必做"、哪些是"选做"
3. 添加缓冲时间（测试调试、bug 修复等）
4. 或将 Phase 2c 拆为独立阶段

### QA-009 🟢 缺少测试数据工厂标准

**风险等级**: 低  
**涉及章节**: 三  
**问题描述**: 方案提到"利用 fixture 工厂模式复用"测试数据，但未提供具体的工厂模式标准和示例。

**问题点**:
- 现有测试文件（如 `test_quality.py`、`test_common_queries_complete.py`）使用了不同的测试数据构造方式
- 无统一的数据工厂，每个开发者可能按自己的习惯造数据
- 24+ 个模块若各自造数据，将导致大量重复代码和测试数据不一致
- 未来重构时，数据变更需要修改多处测试文件

**建议**: 
1. 在 `tests/conftest.py` 或 `tests/factories/` 中建立统一的数据工厂
2. 至少为订单、工序、质检等核心实体提供默认工厂
3. 在方案中增加数据工厂的使用规范

---

### QA-010 🟢 覆盖率目标未考虑增量代码

**风险等级**: 低  
**涉及章节**: 一  
**问题描述**: 方案基于当前静态覆盖率数据（10,220 行）计算目标，未考虑执行期间新增代码对覆盖率的影响。

**风险**:
- 开发过程中可能会有 bug 修复或新功能代码增加
- 新增代码初始覆盖率为 0%，会稀释整体覆盖率
- 如果执行期间增加 500 行新代码，目标 90% 需要覆盖的语句数将从 9,198 变为 9,648
- 方案中的"4200 行缺口"可能在实际执行中扩大

**建议**: 
1. 在方案中增加"新增代码覆盖率"要求（如新增代码须 ≥80% 覆盖）
2. 每日跟踪覆盖率时同时记录代码总量变化
3. 设定动态目标公式，而非固定 90%
