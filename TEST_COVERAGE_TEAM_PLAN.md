## 一、角色分工

### 1.1 小李 - 测试创建工程师

**职责**:
- 创建单元测试文件
- 补充测试用例
- 确保测试覆盖率提升
- 修复审计发现的问题

**工作原则**:
- 测试用例必须可独立运行
- 每个测试函数必须有清晰的docstring
- 使用有意义的断言消息
- Mock外部依赖
- 测试之间相互独立

**输出**:
- 测试文件：`tests/unit/{module}/test_{module}_complete.py`
- 测试用例数量
- 覆盖率提升数据

---

### 1.2 小王 - 测试审计工程师

**职责**:
- 独立审计小李创建的测试文件
- 检查测试质量
- 发现问题并提出整改建议
- 验证修复效果

**审计范围**:
- 测试用例正确性
- 测试覆盖完整性
- 代码质量（命名、规范）
- Mock使用合理性
- 边界情况覆盖

**工作原则**:
- 独立审计，不参考小李的代码
- 基于源码和测试结果进行审计
- 客观公正，不预设立场
- 发现问题必须提供具体修复建议

## 四、问题分类

### 4.1 问题严重程度

| 级别 | 描述 | 影响 |
|------|------|------|
| 🔴 P0 | 测试错误或遗漏关键功能 | 必须立即修复 |
| 🟡 P1 | 覆盖不完整或Mock不当 | 应尽快修复 |
| 🟢 P2 | 代码规范问题 | 建议修复 |

### 4.2 常见问题

**P0问题**:
- 测试用例失败
- 未测试核心功能
- Mock错误导致测试无意义

**P1问题**:
- 缺少边界情况测试
- 缺少异常情况测试
- Mock不完整

**P2问题**:
- 命名不规范
- 注释不清晰
- 代码格式问题

## 六、质量保证

### 6.1 测试文件命名规范

```
test_{模块}_{功能}_complete.py
```

示例：
- `test_db_complete.py`
- `test_order_complete.py`
- `test_process_service_complete.py`

### 6.2 测试函数命名规范

```
test_{功能}_{场景}
```

示例：
- `test_create_order_success`
- `test_create_order_with_invalid_data`
- `test_update_status_order_not_found`

### 6.3 Docstring规范

```python
def test_function_scenario(self):
    """测试[功能] - [场景]

    测试目的:
    - [检查点1]
    - [检查点2]

    预期结果: [描述]
    """
```
