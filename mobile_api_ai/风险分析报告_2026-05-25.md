## 三、剩余风险点分析

### 风险等级定义
- **CRITICAL**: 可能导致服务崩溃、数据丢失或对外接口不可用
- **HIGH**: 可能导致功能异常，需要人工介入恢复
- **MEDIUM**: 功能部分降级，不影响整体服务
- **LOW**: 边缘情况，发生概率低

### 3.1 高风险

#### 🔴 CRITICAL: dispatch_center.py 尚余 16 个 GET 路由无 try/except

虽然 GET 读取操作风险较低，但在极端条件下（缓存损坏、JSON 解析异常）仍可能返回 500 错误。建议在后续迭代中为全部路由补充统一异常处理。

**影响范围**: `get_templates`, `get_global_config`, `get_department_managers`, `get_process_departments`, `get_default_templates` 等只读路由

**建议**: 创建 Flask 全局异常处理器，统一处理未捕获异常：
```python
@dispatch_center_bp.errorhandler(500)
def handle_500(e):
    logger.error(f'[路由] 未捕获异常: {e}')
    return jsonify({'code': 500, 'message': '服务器内部错误'}), 500
```

#### 🔴 HIGH: cloud_poller.py 零字节 ACK 未严格校验

message_hub 的 ACK 响应仅校验了存在性（`if ret`），未校验内容。如果 API 返回 `{"code": 0}` 外的状态码会被视为成功。

**建议**: 在 `send_via_message_hub` 中增加 `ret.get('ack', False)` 校验。

#### 🔴 HIGH: dispatch_center.py `_get_outsource_records()` 数据格式脆弱

`_get_outsource_records()` 对返回值进行多格式兼容解析（list/dict），但一旦返回意外结构（如 None），会抛出 `AttributeError`。

**建议**: 在 `_get_cached_work_orders` 返回 None 时直接返回空列表。

### 3.2 中风险

#### 🟡 MEDIUM: wechat_server.py `_app_bot` 仍有 2 处读操作路径可能异常

L2367 的 `if not _app_bot:` 和 L2961 的 `if not _app_bot or not operator:` 虽然本次已修复模块级同步，但在服务初始化完成前仍有竞态窗口。

**建议**: 在这些检查点添加额外的 `ctx = WeChatContext.get_instance()` 回退检查。

#### 🟡 MEDIUM: dispatch_center.py 批量操作原子性无保证

`batch_assign` (L2831) 循环处理多个任务，但单个失败不会导致整体回滚。可能出现部分成功部分失败的状态。

**建议**: 添加事务性包装或至少记录失败的 items 列表。

### 3.3 低风险

#### 🟢 LOW: cloud_poller.py `stop()` 方法中 `thread.join(timeout=...)` 使用的是 `REQUEST_TIMEOUT_FAST` 环境变量

该变量可能在部署环境中未定义（当前有默认值 5 秒），可正常运行。如需统一管理超时配置，建议在公共配置模块中定义。

#### 🟢 LOW: dispatch_center.py 模板渲染无 XSS 防护

`task_notify` 中直接使用 `f-string` 拼接用户输入到日志（已转义），但模板渲染内容未经 HTML 编码。不过该接口主要在企业内网使用，风险较低。
