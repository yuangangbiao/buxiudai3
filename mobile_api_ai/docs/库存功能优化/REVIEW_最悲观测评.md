# 库存功能优化 — 最悲观测评报告（v2.0 → 100/100）

> 评估方法：对抗性思维，假设每条建议都失败，找出方案的**已知风险与边界**

## 一、v2.0 自我最悲观测评

### 1.1 软删除 + 唯一约束冲突（v3.0 仍存风险）

**问题场景**：
- 产品 A 的 code = "P001" 被软删除
- 新增产品 B 想用 code = "P001"
- 如果 products.code 唯一索引包含 deleted_at 列未处理 → 新增失败
- 如果 unique 索引只对 `code` 唯一 → 软删除的"P001"和新增的"P001"冲突

**解决**（已隐含但需明文）：
- MySQL 8.0+ 虚拟列方案：`code_active = CASE WHEN deleted_at IS NULL THEN code ELSE NULL END`，对 code_active 加唯一索引
- 或采用"软删除后追加后缀"：`code = CONCAT(code, '_del_', id)`

→ **风险等级：中**（业务必须明确取舍）

### 1.2 调拨"在途库存"数据一致性（v3.0 仍存风险）

**问题场景**：
- 调拨创建：A 仓 -100，在途 +100，B 仓未动 → 此时服务崩溃
- 重启后 in_transit 状态的 transfer 永远卡住

**未在 v2.0 中明确**：
- 定时任务扫描 in_transit > 24h 的 transfer，发邮件提醒人工介入
- 或"自动取消"策略：超时 24h 自动回滚（A 仓 +100）

→ **风险等级：高**（必须补充）

### 1.3 库存价值计算的"单价"口径

**问题**：报表需要"库存总价值"，但 products 表没有 price 字段

**v2.0 报告未明确**：
- 选项 A：加 `last_purchase_price` 字段（从入库单带过来）
- 选项 B：成本用 FIFO / 加权平均（需 transactions 表支持）
- 选项 C：先只做"数量趋势"，不做"价值趋势"

→ **必须在实施阶段明确**，否则报表数据会误导

### 1.4 单用户 vs 多用户的通知体系

**当前架构**：单用户（admin 一个账号登录）

**v2.0 通知方案**：
- notifications 表只有 is_read，没有 user_id
- 实际是"全局通知"而非"个人通知"

→ **如果未来多用户，notifications 需要重构**（加 user_id 字段）

### 1.5 灰度开关的真实可行性

**v2.0 灰度方案**：
- 在 config_center.py 加 FEATURE_FLAGS
- 但每条路由的 if FEATURE_FLAGS['x']: return ... 散落在 8 个文件中
- 部署后改 FEATURE_FLAGS → **需要重启服务**（不是热切换）

→ **真实灰度应该是数据库或 Redis 的开关**，可热切

### 1.6 扫码 CDN 内网部署

**html5-qrcode 走 CDN**：
- 离线 / 内网 / 局域网环境会失败
- 必须下载到本地 `inventory_web/static/vendor/`

→ **v2.0 实施阶段需补：本地化静态资源**

### 1.7 xlsx 导入大文件边界

**v2.0 性能预算**：
- 1000 行 < 10s
- 但未测试 10000 行场景
- openpyxl 内存占用大（行数 × 字段数），10000 行约 50MB 内存

→ **必须限制文件大小 + 提示用户分批**

### 1.8 抽盘"绝对差值 vs 百分比差值"双重判断

**v2.0 容差算法**：
- 只用 `abs(diff_qty) / expected_qty <= tolerance_pct / 100`
- 问题：expected_qty=1，actual=0，差异 100% → 算异常
- 但 expected_qty=10000，actual=9990，差异 0.1% → 算正常（但缺 10 个！）

**v2.0 应补充**：
- 双重判断：`abs(diff_qty) > 1 AND abs(diff_qty) > expected_qty * tolerance_pct / 100`
- 避免"差异绝对值很小"被百分比误判
