# 调度中心服务器无法长期运行 — 深度故障分析报告

## 执行摘要

经过三轮递进式代码审计（常规扫描 → 针对性挖掘 → 全量深度扫描），共发现 **24 个**导致"调度中心服务器无法长期运行"的独立问题。按严重程度分布：

| 严重等级 | 数量 | 核心特征 |
|:--------:|:----:|----------|
| **P0（致命）** | 2 | 服务器框架错误直接导致单线程阻塞 + 后台调度未启动 |
| **P1（严重）** | 10 | 资源泄漏（连接/句柄/内存）最终导致 OOM 或句柄耗尽 |
| **P2（一般）** | 9 | 架构缺陷累积效应，长期运行后爆发 |
| **P3（较低）** | 3 | 代码质量/规范性风险 |

---

## 一、P0 — 致命问题（服务器直接不可用）

### P0-1 ❌ Flask 开发服务器（已知）

| 属性 | 值 |
|------|-----|
| **文件** | `standalone_dispatch_server.py` |
| **行号** | `__main__` 段 |
| **根因** | `app.run(debug=False)` 使用 Flask 内置 Werkzeug 开发服务器，**单线程阻塞模型** |
| **后果** | 一个慢请求阻塞所有请求 → 前端超时 → 用户认为服务挂了 → 人工重启 |
| **修复** | 替换为 Waitress 生产级 WSGI 服务器 |

### P0-2 ❌ 后台调度从未启动（已知）

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py` |
| **行号** | ~3964 |
| **根因** | `start_background_scheduler()` 函数存在，但**没有入口调用它** |
| **后果** | AlertEngine（超时提醒）、CostChecker 永不运行，操作员收不到任务通知 |
| **修复** | 在 `create_app()` 中调用 `start_background_scheduler()` |

## 二、P1 — 严重问题（长期运行后资源耗尽）

### P1-1 🔥 MySQL 连接在异常路径上泄漏（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py` |
| **行号** | 814-862 |
| **根因** | `pymysql.connect()` 创建的连接只在 `try` 块尾部 `finally` 中 `close()`，但 `except` 中直接 `pass`，`ImportError` 和 `Exception` 路径均不关闭连接 |

```python
try:
    conn = pymysql.connect(...)
    c = conn.cursor()
    c.execute(...)
    conn.commit()
except ImportError:
    pass  # ⚠️ conn 泄漏
except Exception as e:
    pass  # ⚠️ conn 泄漏
finally:
    try:
        conn.close()  # 如果 conn 未定义 → NameError 被外层 except 吞掉
    except:
        pass
```

**影响**：每次同步 MySQL 数据失败时泄漏一个连接，MySQL 默认最大连接 151，泄漏 100+ 次后拒绝新连接。

### P1-2 🔥 `inventory_api_server.py` 全部 4 个路由 MySQL 连接泄漏（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `inventory_api_server.py` |
| **行号** | 82, 133, 164, 204 |
| **根因** | 4 个路由函数中，若 `cursor.execute()` 抛出异常，`conn.close()` 不会执行，连接直接泄漏 |

```python
try:
    conn = pymysql.connect(...)
    cursor = conn.cursor()
    cursor.execute(sql)  # ⚠️ 异常时跳到 except
    conn.close()         # ← 未执行
except Exception as e:
    return jsonify({'code': 500, 'message': str(e)})  # conn 未关闭
```

**影响**：每次查询异常泄漏 1 个 MySQL 连接，调用频率 x 异常率 x 运行时间 → 连接耗尽。

### P1-3 🔥 SQLite 连接在多处异常路径上泄漏（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py:1196-1217`, `dispatch_center.py:4525-4542` |
| **根因** | `sqlite3.connect()` 后直接操作，未用 `try/finally` 包裹，中间异常导致 `conn.close()` 跳过 |

```python
conn = sqlite3.connect(db_path)    # 打开
cursor = conn.cursor()
cursor.execute(...)                # ⚠️ 异常时跳到外部 except
conn.close()                       # ← 未执行
```

**影响**：长期运行后文件句柄占用持续增长，Windows 进程句柄上限 ~16000。

### P1-4 🔥 `schedule_flow.py` 多处 MySQL 连接异常泄漏（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `schedule_flow.py` |
| **行号** | `get_customer_from_group()`, `get_order_info_from_main_system()` |
| **根因** | `conn.close()` 只在 `try` 块最后一行，多个 `cursor.execute()` 和 `return` 在异常时跳过关闭 |

```python
try:
    conn = get_main_system_db_connection()
    cursor = conn.cursor()
    cursor.execute(...)  # ⚠️ 异常
    cursor.close()       # ← 未执行
    conn.close()         # ← 未执行
except Exception as e:

```

**影响**：配合 N1-1 的 `get_main_system_db_connection()` 每次返回 `PooledConnection`，连接池耗尽后无法获取新连接。

### P1-5 🔥 35 个 `update_data()` 调用 → 每个创建 1 个守护线程（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py` — `DispatchDataCache.update_data()` |
| **行号** | 665-669、及 35 处调用点 |
| **根因** | 每次 `update_data()` 创建一个 `Thread(target=self._persist, daemon=True)`，不检查前一个线程是否存活 |

```python
def update_data(self, updater):
    with self._lock:
        ...
    threading.Thread(target=self._persist, args=(data,), daemon=True).start()
    # ^ 每次写数据创建一个线程，高频操作下大量短生命周期线程
    return True
```

**影响**：
- 线程创建开销：每次 ~8KB 栈空间，1000 次调用 → 8MB 仅栈内存
- Windows 线程调度压力：大量休眠中线程仍占用调度器时间
- 无写入节流：1秒内 100 次 `update_data()` → 100 次磁盘写入

涉及调用点：规则保存、任务分配、流程推进、模板操作、操作员变更、日志追加等 **35 处**。

### P1-6 🔥 `requests.Session` 永不关闭 — HTTP 连接数不受限（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `container_center_client.py` |
| **行号** | `__init__()` 和所有 API 方法 |
| **根因** | `requests.Session()` 使用默认连接池，`pool_connections=10, pool_maxsize=10`，**从不调用 `session.close()`** |

```python
# ContainerCenterClient 初始化
self.session = requests.Session()
# ... 运行期间永不关闭 ...

# V5CompatibleClient 同上
class V5CompatibleClient:
    def __init__(self, ..., http_client=None):
        self.http_client = http_client  # ContainerCenterClient 实例
        # 此实例的 session 永不关闭
```

**影响**：
- 每个 `ContainerCenterClient` 实例持有 10+ 个 HTTP Keep-Alive 连接
- 连接进入 TIME_WAIT 后不回收（默认 TCP 状态保持 240 秒）
- 多次重建 Client（如断连重试）累积 TIME_WAIT 堆积

### P1-7 🔥 `ContainerCenterClient` 进程退出时连接不关闭（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `container_center_client.py` |
| **根因** | `__del__` 和 `close()` 均未实现，`atexit` 也未注册 |
| **后果** | 进程退出时 OS 强行关闭 TCP 连接 → 对端收到 RST → 可能导致数据丢失 |

### P1-8 🔥 `dispatch_center.js` 前端 `localStorage` 缓存无限增长（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `static/js/dispatch_center.js` |
| **行号** | 35, 47 |
| **根因** | `apiCacheSet()` 写入 `localStorage`，`apiCacheDel()` 只清理 `CONFIG.API_PREFIX` 前缀的 key，其他 key 永不清理 |


**影响**：

## 三、P2 — 一般问题（架构缺陷累积效应）

### P2-1 ⚠️ `cloud_backup.py` `threading.local` 连接永不关闭 + `row_factory` Bug（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `cloud_backup.py` |
| **行号** | 39-54 |
| **根因 1** | `_get_connection()` 创建的 SQLite 连接存储在 `threading.local()` 中，**永不关闭** |
| **根因 2** | `_local.row_factory = sqlite3.Row` 误赋给 `_local` 对象，**第 2 次调用起 `sqlite3.Row` 失效** |

```python
_local = threading.local()

def _get_connection():
    if not hasattr(_local, 'conn') or _local.conn is None:
        _local.conn = sqlite3.connect(db_path)
        _local.conn.execute("PRAGMA journal_mode=WAL")
        _local.row_factory = sqlite3.Row  # ⚠️ BUG: 应改为 _local.conn.row_factory
    return _local.conn  # 永不 close()
```

**影响**：
- 每新增一个线程 → 新增一个永不关闭的 SQLite 连接
- `row_factory` bug 导致非首次调用返回原始 tuple 而非 Row 对象，破坏调用方逻辑

### P2-2 ⚠️ CostChecker `while True` daemon 线程无可控停止机制（本轮新发现）

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py` |
| **行号** | 3949-3961 |
| **根因** | `start_cost_checker()` 创建 `while True` 无限循环守护线程，无停止信号标志 |

```python
def start_cost_checker(interval_seconds: int = 3600):

    if _cost_checker_thread and _cost_checker_thread.is_alive():
```

## 六、新增修复策略（补充到原有方案）

### 6.1 MySQL 连接使用 try/finally 强制关闭（覆盖 P1-1~P1-4, P1-6）

**模式**：所有 `pymysql.connect()` / `PooledConnection` 使用 `try/finally` 包裹

```python
# ✅ 正确模式
conn = None
try:
    conn = pymysql.connect(...)
    cursor = conn.cursor()
    cursor.execute(sql)
    conn.commit()
except Exception as e:
    logger.error(f"查询失败: {e}")
    raise  # 或返回错误
finally:
    if conn:
        try:
            conn.close()
        except Exception:
            pass
```

**涉及文件**：`dispatch_center.py`, `inventory_api_server.py`, `schedule_flow.py`

### 6.2 `requests.Session` 上下文管理器化（覆盖 P1-6, P1-7）

```python
# ✅ 实现 close() 方法并在 atexit 注册
class ContainerCenterClient:
    def close(self):
        if self.session:
            self.session.close()
            self.session = None
    
    def __enter__(self):
        return self
    
    def __exit__(self, *args):
        self.close()
```

### 6.3 前端 localStorage 缓存上限控制（覆盖 P1-8）

```javascript
// ✅ 写入前检查并清理过期缓存
function apiCacheSet(path, opts, data) {
    _cleanup_stale_cache();  // 先清理过期项
    try {
        localStorage.setItem(k, JSON.stringify({ v: data, t: Date.now() }));
    } catch (e) {
        if (e.name === 'QuotaExceededError') {
            _cleanup_all_cache();  // 配额满时全清
            localStorage.setItem(k, JSON.stringify({ v: data, t: Date.now() }));
        }
    }
}
function _cleanup_stale_cache() {
    const now = Date.now();
    for (let i = 0; i < localStorage.length; i++) {
        const key = localStorage.key(i);
        if (!key.startsWith(CONFIG.API_PREFIX)) continue;
        try {
            const raw = localStorage.getItem(key);
            const parsed = JSON.parse(raw);
            if (now - parsed.t > 3600000) localStorage.removeItem(key);  // >1h
        } catch(e) {}
    }
```
