## 一、严重问题（必须整改，否则影响执行）

### S1. pyproject.toml 在两个阶段重复出现

| 项目 | 内容 |
|------|------|
| **位置** | 阶段一 Q1.3 + 阶段三 M1.1 |
| **问题** | `pyproject.toml` 的创建在两个阶段中都有提及，职责不清 |
| **影响** | 执行阶段可能冲突，或在阶段三遗漏配置项 |
| **修改建议** | 阶段一 Q1.3 负责**创建** `pyproject.toml`（工具链骨架），阶段三 M1.1 改为**填充** `pyproject.toml` 的项目元数据（name/version/author等）。在 TASK 文档中明确此分工 |

### S2. P1.3 数据库连接池涉及文件指向错误

| 项目 | 内容 |
|------|------|
| **位置** | 阶段四 P1.3 |
| **问题** | `config.py` 中的 `MAX_CONNECTIONS=10` 是 **HTTP 连接池**（用于 ResourceManager），**不是 MySQL 连接池**。MySQL 连接池在 `core/database.py` 中管理 |
| **影响** | 执行时找不到正确的修改位置，浪费精力 |
| **修改建议** | 将涉及文件改为 `core/database.py`（修改 MySQL 连接池配置）+ `config.py`（增加 HTTP 连接池调优配置）。修改验收标准，区分 MySQL 和 HTTP 连接池 |

### S3. 缺少与现有规范的一致性检查

| 项目 | 内容 |
|------|------|
| **位置** | DESIGN 文档全文 |
| **问题** | 未验证优化方案与以下既有规范的兼容性：① `dispatch_center_refresh.md`（刷新规范）② `wechat_server_cloud_only.md`（云端专用约束）③ `业务领域概念定义.md`（Process/SubStep 定义）④ `版本归档管理.md`（已归档项目不可复用） |
| **影响** | 实际执行时可能违反既有规范 |
| **修改建议** | 在 DESIGN 文档中增加"与既有规范兼容性"章节，逐条确认兼容性。WeChat 文件优化需添加"不修改 wechat_server.py"约束 |

## 二、重要问题（建议整改）

### W1. 类型注解补全工作量估算偏紧

| 项目 | 内容 |
|------|------|
| **位置** | 阶段一 Q1.2，DESIGN 估算 6h |
| **问题** | 451个函数（dispatch_center 313 + wechat_work_bot 78 + face_checkin 60），6h = 每个函数不到 1 分钟（含上下文理解+类型推断），实际需要 10-12h |
| **建议** | 将估算调整为 10-12h，或缩小范围：优先 dispatch_center.py（313函数→50% ≈ 157个函数需注解），其余2个文件延后至阶段三 |

### W2. 缺少回退方案和 checkpoint

| 项目 | 内容 |
|------|------|
| **位置** | DESIGN 文档 |
| **问题** | 5个阶段都是线性依赖，没有设计「暂停点」和「回退方案」。如果一个子任务失败，整条链路阻塞 |
| **建议** | 每个阶段末尾增加 checkpoint：① 阶段一 checkpoint：代码质量工具链就绪后创建 git commit/tag ② 阶段二 checkpoint：pytest 基线通过后创建 git tag ③ Git 分支策略：每个阶段独立 feature branch |

### W3. M1.2 和 M1.3 验收标准主观

| 项目 | 内容 |
|------|------|
| **位置** | 阶段三 M1.2、M1.3 |
| **问题** | "至少2处重复代码" 未定义重复代码判定标准（多少行算重复？相似度阈值？）。"关键业务函数 top 50" 未定义"关键"标准 |
| **建议** | M1.2 定义：>10行完全相同的代码块或 >20行相似度≥80% 的代码块。M1.3 定义：直接处理 HTTP 请求、数据库查询、外部 API 调用的函数为"关键函数" |

### W4. 阶段四未涵盖 N+1 问题检测

| 项目 | 内容 |
|------|------|
| **位置** | 阶段四 DESIGN 目标中提到了"N+1问题"，但行动项 P1.4 只做了"慢查询分析" |
| **问题** | N+1 查询问题的检测需要 SQL 日志追踪 + ORM 调用分析，与简单的时间包裹不同 |
| **建议** | P1.4 增加子任务：在 dispatch_center.py 中检测循环内执行 SQL 查询的模式（for 循环 + get_cursor/get_by_id 组合），标记为 N+1 疑似点 |
