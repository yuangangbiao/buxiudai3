## 二、故障分析摘要

### 2.1 问题统计

经过三轮递进式代码审计，共发现 **24 个**导致"调度中心服务器无法长期运行"的独立问题：

| 严重等级 | 数量 | 核心特征 |
|:--------:|:----:|----------|
| **P0（致命）** | 2 | 服务器框架错误直接导致单线程阻塞 + 后台调度未启动 |
| **P1（严重）** | 10 | 资源泄漏（连接/句柄/内存）最终导致 OOM 或句柄耗尽 |
| **P2（一般）** | 9 | 架构缺陷累积效应，长期运行后爆发 |
| **P3（较低）** | 3 | 代码质量/规范性风险 |

### 2.2 P0 致命问题

#### P0-1: Flask 开发服务器

| 属性 | 值 |
|------|-----|
| **文件** | `standalone_dispatch_server.py` |
| **问题** | `app.run(debug=False)` 使用 Flask 内置 Werkzeug 开发服务器，**单线程阻塞模型** |
| **后果** | 一个慢请求阻塞所有请求 → 前端超时 → 用户认为服务挂了 → 人工重启 |
| **修复** | 替换为 Waitress 生产级 WSGI 服务器 |

#### P0-2: 后台调度从未启动

| 属性 | 值 |
|------|-----|
| **文件** | `dispatch_center.py` |
| **问题** | `start_background_scheduler()` 未被调用 |
| **后果** | 定时任务不执行，调度缓存不刷新 |
| **修复** | 在服务启动时调用 `start_background_scheduler()` |

## 三、整体架构方案

### 3.1 三层渐进加固策略

不搞大重构，采用 **"止血 → 治疗 → 强身"** 三层渐进式方案：

```
Layer 1: 止血（P0）→ 让服务器不停机
Layer 2: 治疗（P1）→ 让服务器不崩溃
Layer 3: 强身（P2）→ 让服务器可运维
```

### 3.2 修复清单

| 层级 | 问题 | 修复方案 |
|------|------|---------|
| L1-P0-1 | Flask 开发服务器 | 使用 Waitress 替换 `app.run()` |
| L1-P0-2 | 后台调度未启动 | 在 `server_launcher.py` 中调用 `start_background_scheduler()` |
| L2-P1-1 | SQLite 连接泄漏 | 添加连接上限 + 定期清理机制 |
| L2-P1-2 | 无进程看护 | 实现进程监控 + 自动重启 |
| L2-P1-3 | JSON 频繁 I/O | 批量写入 + 异步刷新 |
| L3-P2-1 | 嵌套锁死锁风险 | 统一锁获取顺序 |
| L3-P2-2 | 守护线程数据安全 | 改用非守护线程 + 优雅关闭 |
| L3-P2-3 | JSON 文件无限增长 | 实现数据归档和清理策略 |
