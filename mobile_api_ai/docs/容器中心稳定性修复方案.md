```
     ↓
API无try/except → 未预期异常返回500（无日志）

     ↓
OOM 或 超时 → 进程被杀死
     ↓
无自动重启 → 服务永久停摆
```

#### P0-1: 日志系统完全未初始化

**现状：**
```python
# config.py L32-L41 — 完整定义了日志配置
LOG_DIR = os.getenv('LOG_DIR', os.path.join(BASE_DIR, 'logs'))
LOG_LEVEL = os.getenv('LOG_LEVEL', 'INFO')
LOG_RETENTION_DAYS = int(os.getenv('LOG_RETENTION_DAYS', '30'))
LOG_FORMAT = os.getenv('LOG_FORMAT', '%(asctime)s [%(levelname)s] [%(name)s:%(lineno)d] %(message)s')
LOG_MAX_BYTES = int(os.getenv('LOG_MAX_BYTES', str(100 * 1024 * 1024)))
```
```python
# container_center_api.py L23 — 仅有 logger = logging.getLogger(__name__)，无任何初始化
logger = logging.getLogger(__name__)
```

**后果：**
- 所有 `logger.info()` / `logger.debug()` 调用全部**静默消失**
- 仅 WARNING+ 级别输出到 stderr，线上 `logs/container_api.err` 文件其实是 stderr 捕获，非应用日志
- Config 定义的 `LOG_DIR/LOG_LEVEL/LOG_FORMAT/LOG_MAX_BYTES` 配置**全部闲置**
- 无法按级别过滤、无法文件轮转、无法追溯历史
- 服务器异常无日志，问题排查完全靠猜

**修复方案：**

在 `container_center_api.py` 入口（`if __name__ == '__main__':` 最开头）添加日志初始化：

```python
import logging
from logging.handlers import RotatingFileHandler
from config import Config

os.makedirs(Config.LOG_DIR, exist_ok=True)
log_file = os.path.join(Config.LOG_DIR, 'container_center.log')

file_handler = RotatingFileHandler(
    log_file,
    maxBytes=Config.LOG_MAX_BYTES,
    backupCount=10,
    encoding='utf-8'
)
file_handler.setFormatter(logging.Formatter(Config.LOG_FORMAT, Config.LOG_DATE_FORMAT))

console_handler = logging.StreamHandler()
console_handler.setFormatter(logging.Formatter(Config.LOG_FORMAT, Config.LOG_DATE_FORMAT))

logging.basicConfig(
    level=getattr(logging, Config.LOG_LEVEL),
    handlers=[file_handler, console_handler]
)

logger.info('日志系统已初始化: 级别=%s, 文件=%s', Config.LOG_LEVEL, log_file)
```

**风险：** 低。仅在入口处初始化一次，不影响其他模块已有的 logger。

#### P0-2: SQLite 缺少 WAL 模式和 busy_timeout

**现状代码：**
```python
self._conn = sqlite3.connect(self.db_path, check_same_thread=False)
self._conn.row_factory = sqlite3.Row
```

**后果：**
- SQLite 默认 `journal_mode=delete`，写操作锁定整个数据库文件
- 多线程并发写（主请求 + 定时校对 + 启动推送 + 桌面端回调）时频繁 `database is locked`
- 写阻塞累积导致请求线程假死，服务不可用

**修复方案：**
```python
self._conn.execute('PRAGMA journal_mode=WAL')
self._conn.execute('PRAGMA busy_timeout=5000')
self._conn.execute('PRAGMA synchronous=NORMAL')
```

**风险：** 低。`PRAGMA` 语句幂等，对已有数据无影响。WAL 模式读不阻塞写。

#### P0-4: Thread.join(timeout=20) 阻塞请求线程

**现状代码：**
```python
t = threading.Thread(target=notify_async)
t.start()
t.join(timeout=20)  # ← 在 Flask 请求线程中阻塞等待最长20秒
```

**后果：**
- 每次 `dispatch_task()` 调用时，请求线程被阻塞最多20秒等待异步通知完成
- 若 `threaded=False`，这20秒内整个服务无法处理其他请求
- 即使 `threaded=True`，也占用了线程池中的一个线程20秒

**修复方案：**
```python
# 改为完全异步，不阻塞请求线程
t = threading.Thread(target=notify_async, daemon=True)
t.start()
# 移除 t.join() — 通知在后台异步完成
logger.debug("[3.01对接] 异步通知线程已启动（不阻塞）")
```

**风险：** 低。`notify_async` 内部已有完整的 try/except 保护，且是 fire-and-forget 语义。

#### P0-5: API 端点普遍缺少 try/except

**影响范围（约 35 个端点）：**

| 端点 | 当前状态 | 风险 |
|:-----|:--------:|:----:|
| `GET /api/processes` | 无 try | 查询失败返回 Flask 默认500，无日志 |
| `POST /api/processes` | 无 try | 创建失败同上 |
| `PUT /api/processes/<id>` | 无 try | 更新失败同上 |
| `PUT /api/processes/<id>/status` | 无 try | 同上 |
| `PUT /api/processes/<id>/step` | 无 try | 同上 |
| `DELETE /api/processes/<id>` | 无 try | 同上 |
| `GET /api/tasks` | 无 try | 同上 |
| `GET /api/tasks/<id>` | 无 try | 同上 |
| `POST /api/tasks/<id>/complete` | 无 try | 同上 |
| 外协相关4个端点 | 无 try | 同上 |
| ...约25个更多端点 | 无 try | 同上 |

**后果：**
- 任何未预期异常（如 SQLite 锁冲突、类型转换错误）返回 Flask 默认500 HTML
- **异常无任何日志记录**，配合 P0-1（日志未初始化）导致问题完全无法追溯
- 前端收到 500 但无法获知具体原因

**修复方案：**

使用统一装饰器批量保护：

```python
import functools

def api_route(f):
    """API端点异常包装器：捕获未预期异常，记录日志，返回统一错误格式"""
    @functools.wraps(f)
    def wrapper(*args, **kwargs):
        try:
            return f(*args, **kwargs)
        except Exception as e:
            logger.exception(f'[API] {request.method} {request.path} 未预期异常: {e}')
            return jsonify({'code': 500, 'message': '服务器内部错误'})
    return wrapper
```

然后在所有 `@app.route` 装饰的端点后追加 `@api_route`：

```python
@app.route('/api/processes', methods=['GET'])
@api_route
def get_processes():
    ...
```

另一种方案（不侵入现有函数）——在 `__main__` 中统一注册 after_request 异常处理：

```python
@app.errorhandler(Exception)
def handle_exception(e):
    logger.exception(f'[全局异常] {request.method} {request.path}: {e}')
    return jsonify({'code': 500, 'message': '服务器内部错误'}), 500
```

**风险：** 低。推荐使用全局 `@app.errorhandler(Exception)`，零侵入，自动覆盖所有端点。

#### P1-2: 共享 SQLite 连接跨线程无锁保护

已在诊断报告中分析。补充：

**影响范围：** 77 处 `self._conn.cursor()` 调用全部共享同一连接。

**修复方案：**
```python
class SQLiteStorage(BaseStorage):
    def __init__(self, config: Dict):
        self._write_lock = threading.Lock()
    
    def _execute_write(self, sql: str, params: tuple = None):
        """线程安全的写操作"""
        with self._write_lock:
            cursor = self._conn.cursor()
            if params:
                cursor.execute(sql, params)
            else:
                cursor.execute(sql)
            self._conn.commit()
            return cursor
```

```python
cursor.execute('CREATE INDEX IF NOT EXISTS idx_pkg_created ON data_packages(created_at)')
cursor.execute('CREATE INDEX IF NOT EXISTS idx_pkg_data_type ON data_packages(data_type)')
cursor.execute('CREATE INDEX IF NOT EXISTS idx_pkg_related_order ON data_packages(related_order)')
```

对 process_records 表同样检查，补充：

```python
cursor.execute('CREATE INDEX IF NOT EXISTS idx_proc_updated ON process_records(updated_at)')
```

#### P2-1: 后台 Daemon 线程脆弱无保护

已在诊断报告中分析。补充弹性线程实现：

```python
class ResilientThread(threading.Thread):
    """带自动重启的弹性后台线程"""
    
    def __init__(self, target, name=None, restart_delay=10, max_retries=0):
        """
        Args:
            target: 线程执行函数
            name: 线程名称（用于日志）
            restart_delay: 异常退出后重启等待秒数
            max_retries: 最大重试次数，0=无限重试
        """
        super().__init__(daemon=True)
        self._target_fn = target
        self._restart_delay = restart_delay
        self._max_retries = max_retries
        self.name = name or target.__name__
        self._retry_count = 0

    def run(self):
        while self._max_retries == 0 or self._retry_count < self._max_retries:
            try:
                logger.info(f'后台线程 [{self.name}] 启动')
                self._target_fn()
            except Exception as e:
                self._retry_count += 1
                logger.error(
                    f'后台线程 [{self.name}] 异常退出 (第{self._retry_count}次), '
                    f'{self._restart_delay}秒后重启: {e}'
                )
                time.sleep(self._restart_delay)

# 使用
_sync_thread = ResilientThread(target=_sync_check_sender, name='sync_check')
_sync_thread.start()
```

## 六、风险说明与回退方案

### 已知风险

| 风险项 | 级别 | 说明 | 回退方案 |
|:-------|:----:|:-----|:---------|
| WAL 模式对已有数据的影响 | 低 | 幂等操作，不影响已有数据 | `PRAGMA journal_mode=DELETE` 回退 |
| threaded=True 的资源消耗 | 低 | 线程池开销约数百KB/线程 | 改回 `threaded=False` |
| 移除了 t.join 的原子性保证 | 低 | notify_async 是 fire-and-forget | 恢复 `t.join(timeout=20)` |
| 全局异常处理覆盖了预期异常 | 低 | 优先使用局部 try/except | 移除 `@app.errorhandler` |
| 索引对写入性能的轻微影响 | 低 | 写入需更新索引树 | `DROP INDEX IF EXISTS` |
