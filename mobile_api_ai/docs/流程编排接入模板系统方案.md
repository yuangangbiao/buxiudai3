# 流程编排接入消息模板系统 — 方案

## 一、现状与问题

### 1.1 现有资源（已经有的）

| 资源 | 状态 | 说明 |
|------|------|------|
| **流程消息模板** | ✅ 已有 3 个预设 | `tmpl_process_start`、`tmpl_process_advance`、`tmpl_process_complete` |
| **模板可视化编辑** | ✅ 刚完成 | 变量工具栏 + 实时预览 + 另存为 |
| **变量替换引擎** | ✅ 已有 | `onSendTemplateChange()` 用 `{变量名}` 正则替换 |
| **发送 API** | ✅ 已有 | `POST /messages/send` 支持 `template_id` + `variables` 自动渲染 |

### 1.2 断点（自动通知没有走模板）

流程编排中有 **4 处自动通知**，目前都是**硬编码字符串**，用户无法自定义：

```
触发点                      代码位置                    硬编码内容片段
─────────────────────────────────────────────────────────────────────────
dispatch_task() 任务分配    dispatch_center.py:L683-695  f'📋 **生产任务通知**\n订单: {order_no}...'
reassign_task() 任务转派    dispatch_center.py:L755-758  f'🔄 **任务转派通知**\n任务: {task_title}...'
advance_process() 流程推进  dispatch_center.py:L1467-1470 f'✅ **流程推进**\n工单: {order_no}...'
reject_process_step() 退回  dispatch_center.py:L1501-1504 f'🔙 **流程退回**\n工单: {order_no}...'
```

**用户无法改变这些消息的内容和格式**，想改必须改代码。

## 二、方案设计

### 2.1 核心思想

**在流程编排页面增加「通知配置」面板，让用户为每个触发点绑定一个消息模板。**

```
┌─────────────────────────────────────────────────┐
│  流程编排                        [+ 新建流程]    │
├─────────────────────────────────────────────────┤
│  ┌─ 流程卡片 ──────────────────────────────┐    │
│  │  工单: WO20260513001  生产流程  进行中    │    │
│  │  [查看详情] [推进] [退回] [⚙ 通知配置]    │    │
│  └──────────────────────────────────────────┘    │
│  ...                                             │
└─────────────────────────────────────────────────┘
         │ 点击「⚙ 通知配置」
         ▼
┌─────────────────────────────────────────────────┐
│  流程通知配置                                    │
├─────────────────────────────────────────────────┤
│  场景         使用模板             变量          │
│  ─────────────────────────────────────────────  │
│  ✅ 任务分配  [任务分配通知 ▼]    {order_no}...  │
│  ✅ 流程推进  [流程推进通知 ▼]    {order_no}...  │
│  ✅ 流程退回  [流程退回通知 ▼]    {reason}...    │
│  ✅ 流程完成  [流程完成通知 ▼]    {order_no}...  │
│                                               │
│  [💾 保存配置]   [恢复默认]                     │
└─────────────────────────────────────────────────┘
```

### 2.2 数据模型变更

**流程数据中增加 `notification_bindings` 字段**：

```python
# 现有流程对象 + 新增字段
{
    # ... 已有字段
    'id': 'proc_abc123',
    'order_no': 'WO20260513001',
    'flow_type': 'production',
    'status': 'in_progress',
    'steps': [...],

    # ★ 新增：通知绑定配置
    'notification_bindings': {
        'task_assigned': {           # 任务分配时
            'template_id': 'tmpl_task_assigned',
            'enabled': True
        },
        'task_reassign': {           # 任务转派时
            'template_id': 'tmpl_task_transfer',
            'enabled': True
        },
        'process_advance': {         # 流程推进时
            'template_id': 'tmpl_process_advance',
            'enabled': True
        },
        'process_reject': {          # 流程退回时
            'template_id': None,     # None=使用默认硬编码内容
            'enabled': True
        },
        'process_complete': {        # 流程完成时
            'template_id': 'tmpl_process_complete',
            'enabled': True
        }
    }
}
```

**新增全局默认配置（`dispatch_center_data.json` 根级别）**：

```python
{
    # ... 现有规则、模板、消息等

    # ★ 新增：流程通知全局默认绑定
    'process_notification_defaults': {
        'task_assigned':   { 'template_id': 'tmpl_task_assigned' },
        'task_reassign':   { 'template_id': 'tmpl_task_transfer' },
        'process_advance': { 'template_id': 'tmpl_process_advance' },
        'process_reject':  { 'template_id': None },
        'process_complete':{ 'template_id': 'tmpl_process_complete' }
    }
}
```

**变量映射表（硬编码字符串 → 模板变量名）**：

| 触发点 | 可用变量 |
|--------|----------|
| `task_assigned` | `{operator_name}` `{task_title}` `{order_no}` `{process}` `{department}` |
| `task_reassign` | `{from_operator}` `{to_operator}` `{task_title}` `{order_no}` `{reason}` |
| `process_advance` | `{order_no}` `{step_name}` `{operator_name}` `{next_step}` `{process_name}` `{product_name}` |
| `process_reject` | `{order_no}` `{step_name}` `{reason}` `{operator_name}` `{process_name}` |
| `process_complete` | `{order_no}` `{product_name}` `{process_name}` `{completed_at}` |

### 2.3 后端变更

#### 新增 API 接口

```python
# 1. 获取某个流程的通知配置
GET /processes/<id>/notification-config
→ { code: 0, data: { bindings: {...}, available_vars: {...} } }

# 2. 保存某个流程的通知配置
PUT /processes/<id>/notification-config
→ body: { bindings: { ... } }
→ { code: 0, message: '通知配置已保存' }

# 3. 恢复全局默认
POST /processes/<id>/notification-config/reset
→ { code: 0, message: '已恢复默认通知配置' }
```

#### 渲染函数（新增核心工具函数）

```python
def _render_template_message(template_id: str, variables: dict) -> str:
    """从模板库加载模板，用变量替换，返回最终消息文本"""
    data = _load_json_data()
    templates = data.get('templates', [])
    tmpl = next((t for t in templates if t.get('id') == template_id), None)
    if not tmpl:
        return ''
    content = tmpl.get('content', '')
    for key, val in variables.items():
        content = content.replace('{' + key + '}', str(val))
    return content
```

#### 4 个触发点改造（统一替换硬编码）

```python
# 改造前（advance_process 为例）
_send_wechat_message(
    f'✅ **流程推进**\n━━━━━━━━━━━━━━\n工单: {order_no}\n完成步骤: {step["name"]}\n...',
    msg_type='markdown'
)

# 改造后
def _notify_process_event(process_id: str, event_type: str, extra_vars: dict):
    """统一的通知入口：查绑定 → 渲染模板 → 发送"""
    data = _load_json_data()
    process = _find_process(data, process_id)
    bindings = process.get('notification_bindings', {})
    default_bindings = data.get('process_notification_defaults', {})
    binding = bindings.get(event_type) or default_bindings.get(event_type, {})
    template_id = binding.get('template_id')
    if not binding.get('enabled', True) or not template_id:
        return  # 关闭通知或未配置模板，跳过

    # 构建变量字典（自动从 process 对象提取 + 额外变量）
    base_vars = {
        'order_no': process.get('order_no', ''),
        'process_name': PROCESS_FLOW_TEMPLATES.get(process.get('flow_type', ''), {}).get('name', ''),
        'product_name': process.get('product_name', ''),
        'operator_name': extra_vars.get('operator_name', '系统'),
        ...
    }
    base_vars.update(extra_vars)
    content = _render_template_message(template_id, base_vars)
    if content:
        _send_wechat_message(content, msg_type='markdown')

# 4 个触发点调用改为：
_notify_process_event(process_id, 'process_advance', {
    'step_name': step['name'],
    'next_step': steps[current_step + 1]['name'] if ... else '流程结束',
    'operator_name': operator_name
})
```

## 四、优势 vs 风险

| 维度 | 说明 |
|------|------|
| **用户价值** | ✅ 流程中的所有通知都可自定义，不再依赖改代码 |
| **与现有模板系统无缝衔接** | ✅ 复用已有模板库、变量系统、编辑弹窗 |
| **向后兼容** | ✅ 旧流程数据无 `notification_bindings` 字段时自动走全局默认或硬编码回退 |
| **风险** | 低 — 新增接口不影响现有功能，触发点改为调用统一入口，不改返回值 |
| **工作量估算** | 后端 ~100 行、前端 ~150 行，总计约 0.5 天 |
