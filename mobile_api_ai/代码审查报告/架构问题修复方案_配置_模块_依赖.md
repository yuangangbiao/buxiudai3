## 一、配置双轨并行 — 统一配置体系修复方案

### 1.1 现状

当前项目存在三套配置体系：

| 配置体系 | 文件 | 风格 | 被引用文件数 | 状态 |
|---------|------|------|:----------:|:----:|
| 旧配置 | `mobile_api_ai/config.py` | `class Config`, 63+ env vars | ~22 | 标记 DEPRECATED |
| 新配置 | `core/config.py` | `DatabaseConfig` dataclass 风格 | ~26 | 新功能首选 |
| 废弃配置 | `mobile_api_ai/settings.py` | 6个 dataclass 组 | 2 | 几乎未被采纳 |

**核心问题**:
1. 两套配置并存且交叉引用 — 例如 `container_center_client.py` 同时从 `core.config` 和 `config.Config` 导入配置
2. `settings.py` 是第三个配置体系，设计最合理（dataclass + `from_env()`）但仅2个自引用文件使用，是死的方案
3. 维护负担：改一个配置项可能需要改三处
4. 导入混乱：部分模块用 `from mobile_api_ai.config import Config`，部分用 `from core.config import xxx`

### 1.2 目标架构

```
统一前:                        统一后:
config.py (DEPRECATED) ──┐
                          ├──→ 各模块混用导入    core.config ──→ 所有模块统一导入
core.config ──────────────┤                      
                          │                     (config.py → 归档删除)
settings.py ──────────────┘                     (settings.py → 归档删除)
```

### 1.3 详细迁移步骤

#### 步骤 1-A: 盘点所有引用点（依赖分析）

**目标**: 找出每个旧配置引用点，确认对应新配置中的等价项

```bash
# 搜索旧 config 的所有引用
grep -rn "from.*config import\|import.*config\|Config\." mobile_api_ai/ --include="*.py"
grep -rn "settings import\|from.*settings" mobile_api_ai/ --include="*.py"
```

**核查清单（22个旧 config 引用文件）**:

| 文件 | 导入方式 | 使用项 | 新 core.config 等价项 |
|------|---------|--------|---------------------|
| `container_center_client.py` | `from config import Config` + `from core.config import ...` | `REQUEST_TIMEOUT_NORMAL` 等 | ✅ 已有 |
| `dispatch_center.py` | `from config import Config` / `from core.config import ...` | 混用 | ❌ 需逐项映射 |
| `services/session.py` | `from config import Config` | `SESSION_CONFIG` | ❌ 需添加 |
| `wechat_cloud.py` | `from core.config import ...` | ✅ 已在新体系 | — |
| `app.py` | `from config import Config` | `FLASK_` 系列 | ❌ 需添加 |
| ... (全量清单见后) | | | |

**操作**:
1. 对每个引用旧 config.py 的文件，逐项列出使用的配置键
2. 确认 core.config 中是否有对应项
3. 若无对应项，则在 core.config 中添加
4. 生成完整的映射表

#### 步骤 1-B: 补齐 core.config 缺失项

**当前 core.config 已有的配置组** (读取自 `core/config.py`):

```python
class DatabaseConfig:    # DB 连接配置
class WeChatConfig:      # 企业微信配置
class ServerConfig:      # 服务端口等
class LogConfig:         # 日志配置
# ... 需要盘点
```

**需要从旧 config.py 补充到 core.config 的配置项** (示例):

```python
# core/config.py 新增配置组

@dataclass
class AppConfig:
    """应用级配置 - 从旧 config.py 迁移"""
    FLASK_HOST: str = field(default_factory=lambda: os.getenv('FLASK_HOST', '0.0.0.0'))
    FLASK_PORT: int = int(os.getenv('FLASK_PORT', '5001'))
    FLASK_DEBUG: bool = os.getenv('FLASK_DEBUG', 'true').lower() == 'true'
    
    # Session 配置
    SESSION_TYPE: str = os.getenv('SESSION_TYPE', 'filesystem')
    SESSION_PERMANENT: bool = True
    SESSION_USE_SIGNER: bool = True
    SESSION_KEY_PREFIX: str = os.getenv('SESSION_KEY_PREFIX', 'session:')

# 全局单例
app_config = AppConfig()
```

**原则**:
- 所有配置项必须有环境变量对应，**不提供不安全的默认值**（如空密码）
- 组织方式按功能分组（dataclass），避免扁平化
- 保留 `@property` 风格兼容旧调用方式

#### 步骤 1-C: 逐个迁移引用文件

**迁移模式**:

```python
# 旧代码:
from config import Config
db_host = Config.MYSQL_HOST

# 新代码:
from core.config import database_config
db_host = database_config.HOST
```

**迁移优先级** (按引用类型):

| 优先级 | 引用类型 | 文件数 | 说明 |
|-------|---------|:-----:|------|
| P0 | 从旧 config 和 core.config 同时导入的文件 | ~5 | 交叉引用最需紧急处理 |
| P1 | 仅从旧 config 导入的文件 | ~12 | 主要迁移对象 |
| P2 | 仅从 core.config 导入的文件 | ~5 | 保持不动，只需验证 |
| P3 | settings.py 的引用 | 2 | 直接迁移 |

**每次迁移的原子操作**:
1. 修改 import 语句
2. 修改所有调用处的引用名
3. 运行该文件相关测试
4. 提交

#### 步骤 1-D: 验证并废弃旧文件

**废弃前验证**:
1. 确认项目所有 `.py` 文件中没有 `from config import` 或 `from.*config import Config`
2. 确认没有 `from.*settings import`
3. 全项目跑一次 `python -c "import mobile_api_ai.app"` 确认无 ImportError

**废弃操作**:

```python
# config.py 顶部标记为最终废弃
raise RuntimeError(
    "config.py 已废弃，请使用 core.config 模块。"
    "迁移参考: https://internal/wiki/config-migration"
)
```

`settings.py` 相同处理。

#### 步骤 1-E: 删除/归档

在确认所有模块运行正常后：
1. 删除 `config.py`
2. 删除 `settings.py`
3. 删除 `core/config.py` 中所有标记为 DEPRECATED_ 的兼容项

#### 2.2.1 重复对 1: `container_center_client.py` vs `container_center/client/`

**接口差异对比**:

| 维度 | `container_center_client.py` (旧) | `container_center/client/container_client.py` (新) |
|------|----------------------------------|--------------------------------------------------|
| 风格 | 单体大文件 + 7个保护模块硬编码 | 简洁类 + 构造注入 |
| 熔断 | 自实现熔断逻辑 | 无（依赖调用方） |
| 签名 | 自实现 `_build_signature()` | ✅ 也有 `_build_signature()` |
| 配置 | 混用 `config.Config` + `core.config` | 仅依赖传入参数 |
| 重试 | 自实现重试逻辑 | 无 |
| 缓冲 | 自实现缓冲队列 | 无 |
| 灵活性 | ⭐ 低（硬编码配置） | ⭐⭐⭐ 高（注入式设计） |

**合并策略**: **淘汰旧文件，保留子包客户端**

**具体步骤**:

1. **盘点旧文件的唯一功能** — `container_center_client.py` 中有哪些功能是子包没有的：
   
   - 熔断机制 → ❌ 已有 `modules/circuit_breaker.py`，可被调用方使用
   - 重试逻辑 → ❌ 已有模块可复用，或用 `tenacity` 库
   - 缓冲队列 → ❌ 非核心功能，可移除或由调用方自实现
   - 7个保护模块 → ❌ 非核心、设计冗余

2. **确认子包 `container_center/client/` 能力**：
   
   - 基础 CRUD ✅
   - 签名 ✅
   - 请求超时 ✅
   - 错误处理 ✅

3. **迁移所有引用点**:

   ```bash
   # 查找所有引用旧客户端的文件
   grep -rn "container_center_client\|from.*container_center_client" mobile_api_ai/ --include="*.py"
   ```

   预期发现引用文件：
   
   | 引用文件 | 当前使用 | 迁移方式 |
   |---------|---------|---------|
   | `wechat_server.py` | 旧客户端 | 替换为新客户端 |
   | `container_center_v5.py` | 旧客户端 | 替换为新客户端 |
   | ... | | |

4. **更新导入**:

## 三、API跨层直接依赖 — 服务接口抽象层引入方案

### 3.1 现状

当前 API 层（`api/`）直接依赖服务层实现（`services/`），具体表现为：

| 文件 | 行号 | 导入语句 | 问题 |
|------|:----:|---------|------|
| `api/stats.py` | L5 | `from services.stats_engine import get_stats_engine` | 顶层导入具体实现 |
| `api/cost.py` | L5 | `from services.cost_service import get_cost_service` | 顶层导入具体实现 |
| `api/reports.py` | L6 | `from services.stats_engine import get_stats_engine` | 顶层导入具体实现 |
| `api/reports.py` | L237 | `from services.scheduler import _scheduler_instance` | **导入私有对象**（最严重） |
| `api/reports.py` | L247 | `from services.scheduler import start_scheduler` | 顶层导入具体实现 |
| `api/reports.py` | L255 | `from services.scheduler import _scheduler_instance` | 再次导入私有对象 |

**后果**:
1. **可测试性差** — 无法 mock 服务层，测试时必须真实启动服务
2. **紧耦合** — 服务层实现变更（如 `StatsEngine` 构造函数变化）必须同步修改 API 层
3. **启动时加载** — 顶层导入在模块加载时就运行，增加启动时间
4. **破坏分层** — API 路由应只依赖接口，不依赖实现细节

### 3.2 目标架构

```
当前:                           目标:
api/stats.py                    api/stats.py
  └── import services.stats_engine   └── import services.interfaces
                                        └── services/interfaces.py  (抽象接口)
                                              ▲ implements
                                              │
                                        services/stats_engine.py  (具体实现)
```

### 3.3 方案选择

#### 方案 A：服务工厂模式 ✅ 推荐

**原理**: API 层只依赖接口类和工厂函数，不依赖具体实现

**新增文件**: `services/interfaces.py`

```python
# services/interfaces.py — 服务接口抽象层
from abc import ABC, abstractmethod
from typing import Any, Optional


class StatsServiceInterface(ABC):
    """统计服务接口"""
    
    @abstractmethod
    def get_dashboard_stats(self, start_date: str, end_date: str) -> dict:
        ...
    
    @abstractmethod
    def get_production_stats(self, **filters) -> dict:
        ...
    
    @abstractmethod
```

## 四、风险评估与回滚策略

### 4.1 风险矩阵

| 风险 | 概率 | 影响 | 等级 | 缓解措施 |
|------|:---:|:---:|:---:|---------|
| 配置迁移遗漏某个引用 | 中 | 高 | **高** | 逐文件核查 + 全局 grep + CI 检查 |
| 配置默认值变更导致行为差异 | 中 | 高 | **高** | 迁移前后对比所有默认值 |
| 重复模块删除后引用报错 | 低 | 高 | **中** | 删除前全项目 grep 验证 |
| API 导入改工厂后返回 null | 低 | 高 | **中** | 工厂实现加入 null check + 单元测试 |
| `_scheduler_instance` 封装漏掉全局状态 | 中 | 中 | **中** | 封装后保持单例行为不变 |
| `sync_bridge.py` 逻辑分离后行为不一致 | 低 | 中 | **低** | 功能测试覆盖 |

### 4.2 回滚策略

**每个子任务独立可回滚:**

| 回滚级别 | 操作 | 适合场景 |
|---------|------|---------|
| 文件级 | `git checkout <file>` 恢复被修改的单个文件 | 单个导入语句修改出错 |
| 目录级 | `git checkout -- mobile_api_ai/` 恢复整个目录 | 批量迁移出错 |
| 分支级 | `git reset --hard HEAD~1` 回退一个提交 | 整个子任务方案错误 |

### 4.3 执行顺序依赖图

```
配置迁移 (issue 1)
  ├── 依赖: 无
  └── 阻塞: 后续需要配置的改动

重复模块清理 (issue 2)
  ├── 依赖: 可能依赖配置统一后的导入方式
  └── 阻塞: 无

API 接口抽象 (issue 3)
  ├── 依赖: 无
  └── 阻塞: 无
```

> **推荐执行顺序**: Issue 1 (配置) → Issue 3 (接口) → Issue 2 (清理)  
> 理由是配置迁移影响面最广，优先做；接口抽象改动量中等可并行；模块清理是删除操作，最后做最安全。
