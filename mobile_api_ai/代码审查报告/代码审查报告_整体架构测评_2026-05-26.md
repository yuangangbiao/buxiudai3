## 二、模块耦合与依赖分析

### 2.1 耦合度综合评估

| 子项 | 等级 | 详细说明 |
|------|:---:|---------|
| 整体耦合度 | **高** | 多处直接依赖具体实现而非接口，配置体系双轨并行 |
| API 层耦合 | **高** | 3个 API 文件顶层导入 services 细节，缺乏抽象层 |
| 配置耦合 | **高** | 两套配置体系并行（config.py + core.config），部分模块交叉引用 |
| 模块重复 | **中** | 5对功能重叠模块，container_center_client 问题最突出 |
| 循环依赖风险 | **低→高** | 当前无显式循环导入，但 dispatch_center 成中心枢纽增长风险 |

### 2.2 关键依赖关系

#### 正常的单向依赖
```
app.py → api/* → services/* → storage_layer (正常分层，单向调用)
```

#### 异常依赖（跨层直接调用）
```
api/stats.py ──顶层导入──→ services/stats_engine (L5)          ← API直接依赖服务实现
api/cost.py  ──顶层导入──→ services/cost_service (L5)           ← API直接依赖服务实现
api/reports.py──顶层导入──→ services/scheduler._scheduler_instance (L237) ← API依赖私有对象
api/process.py──顶层导入──→ core.database.get_db_cursor (L8)    ← API直接依赖存储层
```

#### 中心辐射式耦合（最严重）
```
                          ┌──→ container_center/storage
                          ├──→ container_center_v5
                          ├──→ sync_bridge
  wechat_server.py ──→ dispatch_center.py ──→ sync/event_bus
  wechat_work_bot_v2.py ──→         ──→ storage_layer
  api/quality.py ──→               ──→ core.config
  standalone_dispatch_server.py ──→ ──→ services/scheduler
```

### 2.3 模块重复清单

| 重复对 | 位置 | 风险等级 |
|--------|------|:-------:|
| container_center_client.py + container_center/client/ | 根目录 + 子包 | **高** |
| container_center_v5.py + container_center/v5_compatible_client.py | 根目录 + 子包 | **中** |
| container_center_api.py + container_center/api/ | 根目录 + 子包 | **中** |
| fault_tolerance.py + modules/circuit_breaker.py | 根目录 + modules | **中** |
| sync_bridge.py + sync/ | 根目录 + 子包 | **低** |

> **最严重**: `wechat_server.py`（云端专用）在同一文件中同时导入 `container_center_client` 的两种不同实现（L138 + L140），这是重复模块共存的典型耦合坏味道。

### 2.4 配置体系双轨并行

| 配置体系 | 被引用文件数 | 占比 | 状态 |
|---------|:----------:|:---:|:----:|
| 旧 config.py | ~22 | ~44% | 标记 DEPRECATED |
| 新 core.config | ~26 | ~52% | 新功能首选 |
| settings.py | 2 | ~4% | 几乎未被采纳 |

**核心问题**: 两套配置体系并存且交叉引用。例如 `container_center_client.py` 同时依赖 `core.config`（L20）和 `config.Config`（L66）。部分模块通过 `mobile_api_ai.config` 导入，部分通过 `core.config` 导入，维护开销大。

## 三、数据架构与存储分析

### 3.1 四套存储体系并存

| # | 存储 | 数据库文件 | 核心文件 | 主要用途 |
|:-:|------|-----------|---------|---------|
| 1 | SQLite (storage_layer) | `wechat_container.db` | storage_layer.py | 数据包、调度指令、流程记录、报工、成本 |
| 2 | DocumentStore (容器中心) | `data/system.db` | container_center/storage/ | 调度中心业务数据（JSON文档化） |
| 3 | MySQL | `steel_belt` (外部DB) | storage_mysql.py, core/database.py | CRM 订单/客户、生产工单状态 |
| 4 | 晨圣本地库 | `chengsheng.db` (外部) | sync/handlers/ | 第三方跟单系统数据 |

### 3.2 存储层抽象评估

| 检查项 | 评级 | 说明 |
|--------|:---:|------|
| 抽象设计 | **中** | BaseStorage + 15 Mixin 体系设计合理，关注点分离好 |
| MySQL 纳入工厂 | **差** | MySQLStorage 未继承 BaseStorage，StorageFactory 不管理 MySQL |
| 抽象层渗透率 | **低** | 大量模块（sync/handlers、face_checkin、temp_query 等）绕过抽象直接 sqlite3.connect |
| 接口统一性 | **中** | 15 个 Mixin 定义清晰，但 MySQLStorage 接口不同 |

### 3.3 数据流向混乱度

```
dispatch_center.py (单次操作)
  ├→ 1. DocumentStore (system.db) JSON 文档
  ├→ 2. SQLiteStorage (wechat_container.db) 关系表
  └→ 3. MySQL (production_orders, orders) 生产库
       └→ 4. EventBus → sync/handlers → chengsheng.db
```

**问题**: 一次业务操作同时写入三套存储，无底层事务保障。任何一次写入失败都会造成数据不一致。

### 3.4 Schema 管理风险

| 风险项 | 等级 | 说明 |
|--------|:---:|------|
| 缺乏 ORM | **高** | SQL 分散在 30+ 文件中，字段名变更需全局查找替换 |
| 无正式 migration | **高** | 运行时 ALTER TABLE 做迁移，生产环境容易出问题 |
| SQLite vs MySQL 模式不统一 | **高** | SQLite 用 TEXT 主键，MySQL 用 INT AUTO_INCREMENT |
| 四库数据冗余 | **中** | 数据在四个数据库间冗余，同步延迟导致不一致 |
| 无分布式事务 | **中** | dispatch_center 双写多库，无事务保障 |

## 五、架构优势（积极发现）

### 5.1 设计亮点
1. **存储层 Mixin 体系**：15 个 Mixin 实现了关注点分离，设计模式良好
2. **Blueprint 注册中心**：`blueprint_registry.py` 声明式配置，便于统一管理
3. **命令模式**：`commands/` 10 个文件实现命令模式，机器人消息处理可扩展
4. **熔断器实现**：`modules/circuit_breaker.py` 三态完整，配置灵活
5. **EventBus 事件驱动**：`core/__init__.py` 的 EventBus 用于状态变更通知
6. **延迟导入策略**：app.py 大量使用函数体内延迟导入，避免启动时循环依赖

### 5.2 已验证的架构质量
- 无显式循环导入（当前）
- 容器中心子服务与主服务有清晰的分层边界
- 同步层 handler 统一使用 `_cs_cursor()` 上下文管理器模式
- `.env` 文件正确使用 `load_dotenv()` 加载

## 八、建议的架构演进路线

### 短期（1-2 迭代）
1. **安全加固**（P0）：dispatch_center 添加认证、实现 RBAC 基础权限
